# 🏠 1. 웹 취약점 (OWASP TOP 10)_기초: 웹사이트의 약점 발견하기

## 🎯 이 문서에서 배울 내용

이 문서를 끝까지 읽으면 다음을 할 수 있게 됩니다:

- ✅ 웹사이트에서 발생할 수 있는 10가지 주요 보안 문제 이해하기
- ✅ 각 취약점이 실생활에서 어떤 문제를 일으키는지 예시로 설명하기
- ✅ 공격자가 웹사이트를 어떻게 공격하는지 기본 원리 이해하기
- ✅ 각 취약점을 어떻게 방어할 수 있는지 기초 방법 알기
- ✅ 실제 웹 보안 사고 사례를 통해 중요성 깨닫기

> **💡 미리보기:** 웹사이트는 우리가 매일 사용하는 서비스입니다. 쇼핑몰, 은행, 학교 홈페이지 등 모두 웹사이트죠. 하지만 이런 웹사이트에도 "약점"이 있고, 나쁜 사람들은 이 약점을 이용해서 우리 정보를 훔치거나 시스템을 망가뜨릴 수 있습니다. OWASP TOP 10은 웹사이트에서 가장 자주 발견되는 10가지 약점의 목록입니다.

---

## 🤔 웹 취약점을 왜 배워야 할까요?

### 🏪 실생활 비유: 쇼핑몰의 보안 약점

여러분이 쇼핑몰을 운영한다고 상상해봅시다:

```
🏬 쇼핑몰 보안 약점들:
┌─────────────────────────────────────────┐
│ 🚪 출입구 잠금장치 고장 (접근 통제 실패)  │
│ 🔑 비밀번호가 벽에 적혀있음 (암호화 실패) │
│ 🗝️ 모든 문이 같은 열쇠로 열림 (인증 실패) │
│ 📦 물건을 막 가져가도 알람 안 울림 (로깅 실패) │
└─────────────────────────────────────────┘
```

**웹사이트도 똑같습니다:**

- 🚪 **출입구 잠금장치 고장** = 로그인 없이 남의 정보 볼 수 있음
- 🔑 **비밀번호가 벽에 적혀있음** = 비밀번호가 암호화 안 되어 있음
- 🗝️ **모든 문이 같은 열쇠로 열림** = 한 번 로그인하면 모든 기능 사용 가능
- 📦 **물건을 가져가도 알람 안 울림** = 누가 언제 무엇을 했는지 기록이 없음

### 📊 웹 취약점으로 인한 실제 피해

| 사고 유형 | 실제 사례 | 피해 규모 |
|----------|---------|----------|
| 개인정보 유출 | 2017년 에퀴팩스 해킹 | 1억 4천만 명 정보 유출 |
| 금전 손실 | 2016년 방글라데시 중앙은행 | 8,100만 달러 도난 |
| 서비스 마비 | 2020년 가민 랜섬웨어 | 5일간 서비스 중단 |

> **⚠️ 중요:** 웹 취약점 하나로 수억 명의 정보가 유출되거나, 수백억 원의 손실이 발생할 수 있습니다.

## 🌐 웹사이트는 어떻게 작동할까요?

웹 취약점을 이해하려면, 먼저 웹사이트가 어떻게 작동하는지 알아야 합니다.

### 📬 웹사이트 = 우체국 시스템

```
[당신의 컴퓨터] ──── 편지 보냄 ────→ [웹 서버]
      ↑                                    ↓
      │                                 편지 읽음
      │                                    ↓
      └──── 답장 받음 ←──────────── 답장 작성
```

1. **당신이 편지를 씁니다** (브라우저에서 URL 입력)
   - "안녕하세요, 쇼핑몰님! 제 장바구니 보여주세요."

2. **우체부가 편지를 전달합니다** (인터넷을 통한 전송)

3. **웹 서버가 편지를 읽고 답장을 씁니다**
   - "네, 여기 당신의 장바구니입니다: 사과 3개, 우유 1개"

4. **답장이 당신에게 돌아옵니다** (웹 페이지가 화면에 표시됨)

### 🎭 공격자는 이 시스템의 약점을 노립니다

```
😈 공격자의 생각:
┌──────────────────────────────────────────┐
│ "편지 내용을 바꿀 수 있을까?"           │ → Injection 공격
│ "다른 사람의 편지를 가로챌 수 있을까?"  │ → 암호화 실패 악용
│ "남의 이름으로 편지를 보낼 수 있을까?"  │ → 접근 통제 우회
│ "서버에게 이상한 명령을 내릴 수 있을까?"│ → SSRF 공격
└──────────────────────────────────────────┘
```

## 💥 OWASP TOP 10: 웹사이트의 10대 약점

OWASP(Open Web Application Security Project)는 전 세계 보안 전문가들이 모여서 만든 비영리 단체입니다. 이들이 매년 조사하는 "가장 위험한 웹 취약점 10가지"가 바로 **OWASP TOP 10**입니다.

---

## 1️⃣ A01: 취약한 접근 통제 (Broken Access Control)

### 🏢 실생활 비유: 회사 출입증 시스템

```
정상적인 출입 통제:
┌─────────────────────────────────────┐
│ 👤 인턴: 1층 사무실만 출입 가능      │ ✅
│ 👨‍💼 팀장: 1층, 2층 출입 가능        │ ✅
│ 👔 임원: 모든 층 + 금고실 출입 가능  │ ✅
└─────────────────────────────────────┘

취약한 접근 통제:
┌─────────────────────────────────────┐
│ 👤 인턴이 URL 주소만 바꿔서          │
│    금고실 비밀번호를 볼 수 있음!     │ ❌
└─────────────────────────────────────┘
```

### 🔍 무엇이 문제일까요?

**간단한 예시:**

당신이 은행 웹사이트에 로그인했습니다. URL 주소창을 보니:
```
https://bank.com/account?id=12345
```

악의적인 사람이 "12345"를 "12346"으로 바꿔봤더니:
```
https://bank.com/account?id=12346
```

→ 다른 사람의 계좌 정보가 그대로 보입니다! 😱

### 💡 왜 발생하나요?

```python
# ❌ 나쁜 코드 예시 (개념)
def show_account(user_id):
    # 로그인한 사람이 정말 이 계좌의 주인인지 확인 안 함!
    return get_account_info(user_id)

# ✅ 좋은 코드 예시 (개념)
def show_account(user_id, logged_in_user):
    # 로그인한 사람이 이 계좌의 주인이 맞는지 확인!
    if user_id != logged_in_user:
        return "접근 권한이 없습니다!"
    return get_account_info(user_id)
```

### 🛡️ 어떻게 방어하나요?

1. **모든 요청마다 권한 확인하기**
   ```
   요청: "계좌 12345 보여줘"
   서버: "잠깐, 너 정말 12345의 주인이야? 확인해볼게."
   ```

2. **기본은 '차단', 허용은 예외로**
   ```
   모든 문: 기본적으로 잠금 🔒
   허가받은 사람만: 특정 문 열쇠 받음 🔑
   ```

3. **관리자 페이지는 숨기지 말고 막기**
   ```
   ❌ 나쁜 방법: /admin 페이지를 숨김 (찾으면 들어갈 수 있음)
   ✅ 좋은 방법: /admin 페이지에 강력한 권한 검증 추가
   ```

### 📚 실제 사례

**2019년 First American 금융회사 사건:**
- 문제: 웹사이트에서 URL의 숫자만 바꾸면 남의 주택 등기 서류를 볼 수 있었음
- 피해: 8억 8천만 건의 민감한 금융 문서 노출
- 원인: 서버에서 사용자 권한을 전혀 확인하지 않았음

## 2️⃣ A02: 암호화 실패 (Cryptographic Failures)

### 🔐 실생활 비유: 우편물 보내기

```
📮 일반 우편 (암호화 안 함):
[당신] ──"비밀번호: 1234"── 그대로 보임 ──→ [은행]
         ↑
    😈 누구나 볼 수 있음!

📮 등기우편 (암호화함):
[당신] ──"#@$%^&*"── 암호화됨 ──→ [은행]
         ↑
    😈 봐도 무슨 내용인지 모름!
```

### 🔍 무엇이 문제일까요?

웹사이트가 중요한 정보를 "그대로" 저장하거나 전송하는 경우:

**나쁜 예시들:**

1. **비밀번호를 그대로 저장**
   ```
   데이터베이스:
   사용자: khsqowp, 비밀번호: password123  ❌
   → 해커가 데이터베이스에 접근하면 비밀번호 그대로 훔쳐감
   ```

2. **HTTP로 비밀번호 전송**
   ```
   http://shopping.com/login?pw=mypassword  ❌
   → 중간에서 누군가 네트워크를 감청하면 비밀번호가 보임
   ```

3. **약한 암호화 사용**
   ```
   MD5로 암호화: 5f4dcc3b5aa765d61d8327deb882cf99
   → 인터넷에 검색하면 원래 비밀번호(password) 나옴  ❌
   ```

### 💡 왜 발생하나요?

```
개발자의 착각:
┌────────────────────────────────────┐
│ "우리 서버는 안전하니까            │
│  비밀번호를 그대로 저장해도 돼"    │  ❌
│                                    │
│ → 서버가 해킹당하면 끝!            │
└────────────────────────────────────┘

현실:
┌────────────────────────────────────┐
│ 서버는 언제든 해킹될 수 있음!      │
│ 데이터는 항상 암호화해야 함!       │  ✅
└────────────────────────────────────┘
```

### 🛡️ 어떻게 방어하나요?

1. **HTTPS 사용하기**
   ```
   ❌ http://bank.com  → 암호화 안 됨
   ✅ https://bank.com → 암호화됨 (자물쇠 표시 🔒)
   ```

2. **비밀번호는 해시로 저장**
   ```
   원본: password123
   저장: $2b$12$KIXqj4qE... (bcrypt로 해시)

   → 해커가 데이터베이스를 훔쳐도
      원래 비밀번호를 알 수 없음!
   ```

3. **민감한 데이터는 암호화해서 저장**
   ```
   신용카드 번호: 1234-5678-9012-3456  (원본)
   저장할 때: aE9jK2m... (AES-256으로 암호화)
   ```

### 📚 실제 사례

**2013년 Adobe 해킹 사건:**
- 문제: 1억 5천만 명의 비밀번호를 약한 암호화(ECB 모드)로 저장
- 피해: 암호화된 비밀번호가 쉽게 해독됨
- 교훈: 강력한 암호화 알고리즘 사용 필수

## 3️⃣ A03: 인젝션 (Injection)

### 💉 실생활 비유: 음식 주문서 조작

```
정상적인 주문:
┌────────────────────────────────┐
│ 주문서: "짜장면 1개 주세요"     │ → 짜장면 1개 나옴 ✅
└────────────────────────────────┘

악의적인 주문:
┌────────────────────────────────────────────┐
│ 주문서: "짜장면 1개 주세요.                │
│         그리고 주방에 있는 모든 음식 내놔" │
│                                            │
│ → 주방장이 주문서를 그대로 실행하면...    │
│    모든 음식이 나옴! 😱                    │ ❌
└────────────────────────────────────────────┘
```

### 🔍 무엇이 문제일까요?

웹사이트의 입력창에 "명령어"를 몰래 넣어서 데이터베이스를 조작하는 공격입니다.

**SQL Injection 예시:**

1. **정상적인 로그인:**
   ```
   사용자 입력:
   - 아이디: khsqowp
   - 비밀번호: mypassword

   서버가 실행하는 SQL:
   SELECT * FROM users
   WHERE id='khsqowp' AND password='mypassword'
   ```

2. **악의적인 로그인 시도:**
   ```
   사용자 입력:
   - 아이디: admin
   - 비밀번호: ' OR 1=1 --

   서버가 실행하는 SQL:
   SELECT * FROM users
   WHERE id='admin' AND password='' OR 1=1 --'
                                      ↑
                              항상 참이므로 로그인 성공! 😱
   ```

### 💡 왜 발생하나요?

```python
# ❌ 나쁜 코드 (사용자 입력을 그대로 SQL에 붙임)
def login(username, password):
    query = f"SELECT * FROM users WHERE id='{username}' AND pw='{password}'"
    # 사용자가 ' OR 1=1 -- 을 입력하면 SQL 구조가 바뀜!
    return database.execute(query)

# ✅ 좋은 코드 (Prepared Statement 사용)
def login(username, password):
    query = "SELECT * FROM users WHERE id=? AND pw=?"
    # ?, ?는 플레이스홀더. 사용자 입력이 SQL 구조를 바꿀 수 없음!
    return database.execute(query, username, password)
```

### 🛡️ 어떻게 방어하나요?

1. **절대 사용자 입력을 믿지 마세요**
   ```
   사용자 입력 = 독이 든 음식이라고 생각하기
   → 검증하고 정제해서 사용!
   ```

2. **Prepared Statement 사용**
   ```
   SQL 구조와 데이터를 분리하기
   구조: "SELECT * FROM users WHERE id=? AND pw=?"
   데이터: ["khsqowp", "mypassword"]
   → 데이터가 절대 SQL 구조를 바꿀 수 없음!
   ```

3. **입력값 검증**
   ```
   아이디 입력창:
   - 허용: 영문자, 숫자, _ 만 허용
   - 차단: ', ", --, ; 같은 SQL 특수문자
   ```

### 📚 실제 사례

**2017년 에퀴팩스 해킹:**
- 문제: Apache Struts 프레임워크의 인젝션 취약점
- 피해: 1억 4천만 명의 개인정보 유출
- 원인: 사용자 입력을 제대로 검증하지 않음

## 4️⃣ A04: 안전하지 않은 설계 (Insecure Design)

### 🏗️ 실생활 비유: 건물 설계 단계의 결함

```
❌ 나쁜 건물 설계:
┌────────────────────────────────┐
│ 🏢 은행 건물                   │
│ - 비상구가 금고실로 직접 연결  │
│ - 창문에 잠금장치 없음          │
│ - CCTV가 로비에만 있음         │
│                                │
│ → 나중에 고치려면 건물 다시 지어야 함!
└────────────────────────────────┘

✅ 좋은 건물 설계:
┌────────────────────────────────┐
│ 🏢 은행 건물                   │
│ - 금고실은 이중 잠금장치       │
│ - 모든 출입구에 경보 시스템    │
│ - CCTV가 곳곳에 설치됨         │
│                                │
│ → 처음부터 보안을 고려한 설계!
└────────────────────────────────┘
```

### 🔍 무엇이 문제일까요?

**기능만 생각하고, 보안은 나중에 생각하는 설계:**

**나쁜 예시 1: 비밀번호 재설정**
```
1. 사용자: "비밀번호 잊어버렸어요"
2. 시스템: "이메일로 재설정 링크 보냈어요"
   링크: reset.com/reset?user=khsqowp
                                ↑
   😈 공격자가 user=admin으로 바꾸면
      관리자 비밀번호를 재설정할 수 있음!
```

**나쁜 예시 2: 쿠폰 사용 제한 없음**
```
시스템 설계:
- 사용자당 쿠폰 1번만 사용 가능
- 하지만 검증 로직이 없음!

공격자:
- 같은 쿠폰을 100번 사용해서 무료로 물건 구매
```

### 💡 왜 발생하나요?

```
개발 프로세스:
1. 기획자: "비밀번호 재설정 기능 만들어주세요"  ✅
2. 개발자: "네, 이메일로 링크 보내면 되죠?"    ✅
3. 출시!                                        ✅

❌ 빠진 것:
- "링크를 가로챈 사람은?"
- "다른 사람 계정을 재설정하려 하면?"
- "링크를 여러 번 사용하면?"
→ 보안을 처음부터 고려하지 않음!
```

### 🛡️ 어떻게 방어하나요?

1. **위협 모델링 (Threat Modeling)**
   ```
   개발 전에 질문하기:
   - "누가 이 기능을 악용할 수 있을까?"
   - "가장 민감한 데이터는 무엇일까?"
   - "어떤 공격 시나리오가 가능할까?"
   ```

2. **보안 요구사항을 처음부터 정의**
   ```
   기능 요구사항: 비밀번호 재설정

   보안 요구사항:
   - 재설정 토큰은 1회용
   - 토큰은 30분 후 만료
   - 재설정 링크는 예측 불가능해야 함
   - 재설정 후 이메일 알림 발송
   ```

3. **보안 검토 단계 추가**
   ```
   개발 프로세스:
   1. 기획 → 보안 검토
   2. 설계 → 보안 검토
   3. 개발 → 보안 검토
   4. 테스트 → 침투 테스트
   5. 출시
   ```

### 📚 실제 사례

**2020년 Zoom 보안 문제:**
- 문제: 회의 ID가 순차적으로 생성되어 예측 가능
- 피해: "Zoom Bombing" - 무단 침입자가 회의에 난입
- 교훈: 처음부터 보안을 고려한 설계 필요

## 5️⃣ A05: 보안 설정 오류 (Security Misconfiguration)

### ⚙️ 실생활 비유: 새 스마트폰 설정

```
📱 새 스마트폰 샀을 때:

❌ 나쁜 습관:
- 잠금 비밀번호: 1234 (기본값 그대로)
- 블루투스: 항상 켜짐 (기본값 그대로)
- 위치 추적: 모든 앱에 허용 (기본값 그대로)
- 소프트웨어 업데이트: 안 함

✅ 좋은 습관:
- 잠금 비밀번호: 복잡한 비밀번호로 변경
- 블루투스: 사용할 때만 켬
- 위치 추적: 필요한 앱만 허용
- 소프트웨어 업데이트: 정기적으로 함
```

### 🔍 무엇이 문제일까요?

**웹 서버도 "기본 설정"이 위험합니다:**

**나쁜 예시들:**

1. **기본 계정 그대로 사용**
   ```
   관리자 계정:
   - ID: admin
   - PW: admin

   😈 공격자: "admin/admin으로 로그인 시도... 성공!"
   ```

2. **에러 메시지가 너무 자세함**
   ```
   로그인 실패 시:
   ❌ "비밀번호가 틀렸습니다"
      → 공격자: "아, ID는 맞구나. 비밀번호만 찾으면 되겠네!"

   ✅ "ID 또는 비밀번호가 틀렸습니다"
      → 공격자: "ID가 틀린 건지 PW가 틀린 건지 모르겠네"
   ```

3. **필요 없는 기능 켜져 있음**
   ```
   웹 서버 설정:
   - 디렉토리 리스팅: ON  ❌

   😈 공격자가 /backup/ 에 접근하면
      모든 백업 파일 목록이 보임!
   ```

### 💡 왜 발생하나요?

```
시스템 관리자의 실수:
┌────────────────────────────────┐
│ "빨리 설치하고 테스트해야 해!" │
│ → 기본 설정 그대로 사용         │
│                                │
│ "나중에 바꿔야지..."           │
│ → 까먹음                       │
│                                │
│ "업데이트는 다음에..."         │
│ → 취약점이 계속 남아있음        │
└────────────────────────────────┘
```

### 🛡️ 어떻게 방어하나요?

1. **보안 강화 체크리스트**
   ```
   ✅ 기본 계정/비밀번호 변경
   ✅ 불필요한 서비스 비활성화
   ✅ 에러 메시지 최소화
   ✅ 디렉토리 리스팅 차단
   ✅ 최신 보안 패치 적용
   ✅ 방화벽 규칙 설정
   ```

2. **자동화된 보안 스캐너 사용**
   ```
   도구: Nessus, OpenVAS
   - 자동으로 시스템 스캔
   - 잘못된 설정 찾아줌
   - 보안 권장사항 제시
   ```

3. **정기적인 보안 점검**
   ```
   매월:
   - 사용하지 않는 계정 삭제
   - 권한 검토
   - 로그 확인

   매주:
   - 보안 패치 확인
   - 취약점 스캔
   ```

### 📚 실제 사례

**2020년 MongoDB 데이터베이스 노출:**
- 문제: 수천 개의 MongoDB가 인증 없이 인터넷에 노출
- 피해: 수백만 건의 개인정보 유출
- 원인: 기본 설정으로 인증이 비활성화되어 있었고, 관리자가 변경하지 않음

## 6️⃣ A06: 취약하고 오래된 컴포넌트 (Vulnerable and Outdated Components)

### 🧩 실생활 비유: 자동차 부품

```
🚗 자동차 = 웹사이트
🔧 부품 = 라이브러리/프레임워크

상황:
┌────────────────────────────────────┐
│ 자동차 제조사에서 리콜 공지:       │
│ "브레이크 결함 발견! 무상 수리"    │
│                                    │
│ ❌ 무시: 계속 오래된 브레이크 사용 │
│    → 사고 위험 증가                │
│                                    │
│ ✅ 대응: 즉시 부품 교체            │
│    → 안전하게 운행                 │
└────────────────────────────────────┘
```

### 🔍 무엇이 문제일까요?

**웹사이트는 수많은 "부품"으로 만들어집니다:**

```
웹사이트 구성:
┌─────────────────────────────┐
│ 🏗️ 프레임워크:              │
│   - React 16.8.0 (2019년)   │ ⚠️ 오래됨
│   - Django 2.2 (2019년)     │ ⚠️ 오래됨
│                             │
│ 📦 라이브러리:              │
│   - jQuery 1.12 (2016년)    │ ⚠️ 아주 오래됨
│   - Log4j 2.14 (2021년)     │ 😱 심각한 취약점!
│                             │
│ → 이 중 하나라도 취약점이 있으면
│   전체 웹사이트가 위험!     │
└─────────────────────────────┘
```

### 💡 왜 발생하나요?

**개발자/관리자가 업데이트를 안 하는 이유:**

```
1. "지금 잘 작동하는데 뭐 하러 바꿔?" 🤔
   → 취약점은 눈에 안 보여도 존재함!

2. "업데이트하면 뭔가 깨질까봐 무서워" 😰
   → 테스트 환경에서 먼저 해보면 됨!

3. "우리가 뭘 쓰는지 기록을 안 해놨어" 📝
   → 의존성 관리 도구 사용!

4. "업데이트할 시간이 없어" ⏰
   → 자동화 도구 사용!
```

### 🛡️ 어떻게 방어하나요?

1. **사용하는 모든 라이브러리 목록 만들기**
   ```
   package.json (Node.js):
   {
     "dependencies": {
       "react": "18.2.0",
       "express": "4.18.2",
       "lodash": "4.17.21"
     }
   }

   → 이 파일만 봐도 무엇을 사용하는지 한눈에!
   ```

2. **취약점 스캔 도구 사용**
   ```
   명령어:
   npm audit  (Node.js)
   pip-audit  (Python)

   결과:
   ┌─────────────────────────────────────┐
   │ ⚠️ 발견된 취약점: 3개               │
   │ - lodash 4.17.20: CVE-2021-23337   │
   │   → 4.17.21로 업데이트 필요         │
   └─────────────────────────────────────┘
   ```

3. **자동 업데이트 설정**
   ```
   GitHub Dependabot:
   - 자동으로 취약한 라이브러리 탐지
   - Pull Request 자동 생성
   - "lodash를 4.17.21로 업데이트하세요"
   ```

### 📚 실제 사례

**2021년 Log4Shell (Log4j 취약점):**
- 문제: Java 로깅 라이브러리 Log4j에서 원격 코드 실행 취약점 발견
- 영향: 전 세계 수백만 개 웹사이트가 영향받음
- 심각도: 10점 만점에 10점 (CVSS Score)
- 교훈: 하나의 작은 라이브러리 취약점이 전 세계에 영향을 미칠 수 있음

## 7️⃣ A07: 식별 및 인증 실패 (Identification and Authentication Failures)

### 🔑 실생활 비유: 아파트 출입 시스템

```
❌ 약한 출입 시스템:
┌────────────────────────────────┐
│ 🏢 아파트                      │
│ - 출입 비밀번호: 1234          │
│ - 10번 틀려도 계속 입력 가능   │
│ - 비밀번호 한번 설정하면 영구  │
│                                │
│ 😈 도둑: "1234 입력... 성공!"  │
└────────────────────────────────┘

✅ 강한 출입 시스템:
┌────────────────────────────────┐
│ 🏢 아파트                      │
│ - 출입 비밀번호: 복잡한 8자리  │
│ - 3번 틀리면 5분간 차단        │
│ - 비밀번호 + 지문 인식 (MFA)  │
│ - 3개월마다 비밀번호 변경      │
│                                │
│ 😈 도둑: "포기..."             │
└────────────────────────────────┘
```

### 🔍 무엇이 문제일까요?

**약한 인증 시스템의 문제점:**

1. **약한 비밀번호 허용**
   ```
   허용되는 비밀번호:
   - 123456  ❌
   - password  ❌
   - qwerty  ❌

   → 이런 비밀번호는 1초 만에 뚫림!
   ```

2. **무차별 대입 공격 (Brute Force) 방어 없음**
   ```
   공격자의 시도:
   1. password
   2. password123
   3. password1234
   ... (계속 시도)
   1000. mypassword  ← 성공!

   → 시스템이 계속 허용하면 언젠가 성공
   ```

3. **세션 관리 문제**
   ```
   세션 ID: session123
            session124  ← 예측 가능!
            session125

   😈 공격자: "다음은 session126이겠네?"
   → 다른 사람의 세션 가로채기!
   ```

### 💡 왜 발생하나요?

```
편의성 vs 보안:
┌────────────────────────────────┐
│ 사용자: "비밀번호 외우기 힘들어!"│
│ → 간단한 비밀번호 설정          │
│                                │
│ 서비스: "사용자가 불편하다고 해" │
│ → 보안 규칙 완화                │
│                                │
│ 결과: 쉽게 뚫리는 시스템        │ ❌
└────────────────────────────────┘
```

### 🛡️ 어떻게 방어하나요?

1. **강력한 비밀번호 정책**
   ```
   비밀번호 요구사항:
   ✅ 최소 8자 이상
   ✅ 대문자, 소문자, 숫자, 특수문자 포함
   ✅ 사용자 이름, 이메일과 유사하면 안 됨
   ✅ 흔한 비밀번호 (예: password123) 차단
   ```

2. **다단계 인증 (MFA/2FA)**
   ```
   로그인 절차:
   1단계: 비밀번호 입력  ✅
   2단계: 휴대폰으로 받은 인증번호 입력  ✅

   → 비밀번호를 알아도 휴대폰이 없으면 로그인 불가!
   ```

3. **계정 잠금 정책**
   ```
   로그인 실패 시:
   1회 실패: 경고
   3회 실패: 5분간 잠금
   5회 실패: 30분간 잠금
   10회 실패: 관리자에게 알림 + 계정 잠금
   ```

4. **안전한 세션 관리**
   ```
   세션 ID:
   ❌ session123 (예측 가능)
   ✅ a9f3k2m8x... (랜덤 32자 이상)

   세션 타임아웃:
   - 30분 동안 활동 없으면 자동 로그아웃
   ```

### 📚 실제 사례

**2012년 LinkedIn 해킹:**
- 문제: 1억 6천만 개의 비밀번호가 약한 해시(SHA-1, 솔트 없음)로 저장됨
- 피해: 비밀번호가 대량으로 해독됨
- 교훈: 강력한 암호화 + 솔트 + MFA 필수

## 8️⃣ A08: 소프트웨어 및 데이터 무결성 실패 (Software and Data Integrity Failures)

### 📦 실생활 비유: 택배 배송

```
❌ 무결성 검증 없음:
┌────────────────────────────────┐
│ [판매자] → 택배 → [당신]       │
│                                │
│ 중간에 누군가 박스를 바꿔치기! │
│ 📦 → 💣                        │
│                                │
│ 당신: 박스를 열었는데... 폭탄! │ 😱
└────────────────────────────────┘

✅ 무결성 검증 있음:
┌────────────────────────────────┐
│ [판매자] → 봉인 스티커 → [당신] │
│              ✅                 │
│                                │
│ 봉인이 떼어진 흔적 발견!        │
│ → 받지 않고 반송               │ 🛡️
└────────────────────────────────┘
```

### 🔍 무엇이 문제일까요?

**소프트웨어 업데이트가 조작될 수 있습니다:**

1. **악성 업데이트 배포**
   ```
   정상 업데이트:
   [회사] → app_v2.0.zip → [사용자]

   조작된 업데이트:
   [회사] → app_v2.0.zip → [😈 해커가 바꿔치기] → [사용자]
                           ↓
                      악성코드 포함!

   → 사용자는 정상 업데이트인 줄 알고 설치
   ```

2. **CI/CD 파이프라인 공격**
   ```
   코드 배포 과정:
   [개발자] → GitHub → 자동 빌드 → 서버 배포
                        ↑
                   😈 이 단계에서 악성 코드 주입!

   → 정상 코드가 악성 코드로 변조됨
   ```

### 💡 왜 발생하나요?

```
신뢰의 문제:
┌────────────────────────────────┐
│ "공식 사이트에서 받았으니 안전해" │ ❌
│ → 중간에 조작될 수 있음!         │
│                                │
│ "자동 업데이트는 다 안전해"      │ ❌
│ → 업데이트 서버가 해킹될 수 있음! │
└────────────────────────────────┘
```

### 🛡️ 어떻게 방어하나요?

1. **디지털 서명 (Digital Signature)**
   ```
   소프트웨어 배포 시:
   1. 회사가 비밀 키로 "서명"
   2. 사용자가 공개 키로 "검증"

   검증 결과:
   ✅ 서명 유효: 회사가 배포한 게 맞음
   ❌ 서명 무효: 누군가 조작함!
   ```

2. **해시 값 비교**
   ```
   공식 사이트:
   app_v2.0.zip
   SHA-256: a3f9e2b8c...

   다운로드 후 확인:
   다운받은 파일의 SHA-256: a3f9e2b8c...

   ✅ 일치: 조작되지 않음
   ❌ 불일치: 조작됨!
   ```

3. **CI/CD 파이프라인 보안**
   ```
   보안 조치:
   - 모든 단계에서 로그 기록
   - 코드 변경은 승인 필요
   - 빌드 환경 격리
   - 최종 산출물에 서명
   ```

### 📚 실제 사례

**2020년 SolarWinds 공격:**
- 문제: 소프트웨어 업데이트 과정에서 악성코드가 주입됨
- 피해: 미국 정부 기관 및 수천 개 기업이 감염됨
- 방법: 빌드 시스템에 침투하여 정상 업데이트에 백도어 삽입
- 교훈: 소프트웨어 공급망 전체에 대한 무결성 검증 필수

## 9️⃣ A09: 보안 로깅 및 모니터링 실패 (Security Logging and Monitoring Failures)

❌ CCTV 없는 은행:
┌────────────────────────────────┐

│                                │
│ 😈 도둑: "완벽한 범죄!"         │
└────────────────────────────────┘

✅ CCTV 있는 은행:
┌────────────────────────────────┐
│ 🏦 은행                        │

│ - 출입 시간/인물 기록           │
│ - 실시간 모니터링               │
│                                │
│ 도둑이 오면:                   │
│ - 즉시 알람                    │ 🚨
│ - 경찰 출동                    │ 👮
│ - 증거 확보                    │ 📹
└────────────────────────────────┘
```

### 🔍 무엇이 문제일까요?

**웹사이트에서 무슨 일이 일어나는지 모르는 상황:**

1. **로그가 없음**
   ```
   공격자가 시스템에 침입했지만:
   - 언제 침입했는지? 모름
   - 어떻게 침입했는지? 모름
   - 무엇을 훔쳐갔는지? 모름
   - 아직도 숨어있는지? 모름

   → 완전히 속수무책! 😰
   ```

2. **로그는 있지만 안 봄**
   ```
   로그 파일:
   2025-01-15 03:45:23 로그인 실패 (admin)
   2025-01-15 03:45:25 로그인 실패 (admin)
   2025-01-15 03:45:27 로그인 실패 (admin)
   ... (100번 반복)
   2025-01-15 03:52:15 로그인 성공 (admin) 🚨

   → 이상한 패턴이지만 아무도 안 봄

### 💡 왜 발생하나요?

```
관리자의 착각:
┌────────────────────────────────┐
│ "로그는 나중에 문제 생기면 봐"   │ ❌
│ → 문제 생길 때는 이미 늦음!      │
│                                │
│ "우리는 안전하니까 괜찮아"       │ ❌
│ → 해킹당해도 모름!               │
│                                │
│ "로그가 너무 많아서 못 봐"       │ ❌
│ → 자동화 도구 사용 안 함!        │
└────────────────────────────────┘
```

### 🛡️ 어떻게 방어하나요?

1. **중요한 이벤트는 반드시 로그로 남기기**
   ```
   필수 로그 항목:
   ✅ 로그인/로그아웃
   ✅ 권한 변경
   ✅ 데이터 수정/삭제
   ✅ 설정 변경
   ✅ API 호출
   ✅ 보안 정책 위반 시도

   로그 포맷:
   [시간] [사용자] [행동] [결과] [IP주소]
   2025-01-15 10:30:25 | khsqowp | 로그인 시도 | 성공 | 192.168.1.10
   ```

2. **실시간 모니터링 및 알림**
   ```
   이상 징후 탐지:
   - 5분 안에 로그인 10번 실패 → 알림 🚨
   - 새벽 3시에 관리자 접속 → 알림 🚨
   - 평소보다 100배 많은 데이터 다운로드 → 알림 🚨

   알림 방법:
   - 이메일
   - SMS
   - Slack 메시지
   ```

3. **로그 보호**
   ```
   로그 저장소 보안:
   - 로그는 별도 서버에 저장
   - 로그 수정 불가 (읽기 전용)
   - 정기적으로 백업
   - 6개월~1년 이상 보관
   ```

## 🔟 A10: 서버 측 요청 위조 (SSRF - Server-Side Request Forgery)

### 📞 실생활 비유: 직원을 이용한 사기

```
❌ SSRF 공격:
┌────────────────────────────────────┐
│ [당신] 😈                          │
│   ↓                                │
│ "여기 사진 주소 불러와줘:           │
│  http://내부망/비밀문서.pdf"       │
│   ↓                                │
│ [회사 서버] 🤖                     │
│ "오케이!" (아무 생각 없이 실행)     │
│   ↓                                │
│ [내부 서버] 🔒                     │
│ "회사 서버가 요청했네? 줄게!"       │
│   ↓                                │
│ [당신] 😈                          │
│ "비밀문서 받았다!"                 │
└────────────────────────────────────┘

→ 회사 서버를 "심부름꾼"으로 이용!
```

### 🔍 무엇이 문제일까요?

**사용자가 서버에게 "가짜 심부름"을 시킵니다:**

**공격 시나리오 1: 내부망 정보 탈취**
```
정상적인 사용:
사용자: "이 이미지 불러와줘: https://cdn.com/photo.jpg"
서버: "오케이!" → 이미지 가져옴 ✅

악의적인 사용:
공격자: "이 '이미지' 불러와줘: http://192.168.1.50/admin"
서버: "오케이!" → 내부 관리 페이지 가져옴 😱
공격자: 내부 정보 획득!
```

**공격 시나리오 2: 클라우드 메타데이터 탈취**
```
AWS에서:
공격자: "이 URL 불러와줘: http://169.254.169.254/latest/meta-data/iam/"
서버: "오케이!" → AWS 임시 자격증명 가져옴 😱
공격자: 자격증명으로 AWS 리소스 제어!
```

### 💡 왜 발생하나요?

```
개발자의 착각:
┌────────────────────────────────┐
│ "사용자가 준 URL을 그냥 불러오면 │
│  편하게 기능 구현할 수 있어!"    │
│                                │
│ → 사용자 입력을 검증 안 함!      │ ❌
│                                │
│ "서버에서 보내는 요청은          │
│  다 안전할 거야"                │
│                                │
│ → 서버가 속을 수 있음!           │ ❌
└────────────────────────────────┘
```

### 🛡️ 어떻게 방어하나요?

1. **URL 입력값 검증**
   ```
   사용자 입력 URL 검사:

   ❌ 차단해야 할 패턴:
   - http://localhost/...
   - http://127.0.0.1/...
   - http://192.168.*.*/...  (내부 IP)
   - http://169.254.169.254/... (AWS 메타데이터)
   - http://10.*.*.*/...  (내부 IP)
   - file:///etc/passwd  (로컬 파일)

   ✅ 허용할 패턴:
   - 미리 정한 도메인 목록만 허용
   ```

2. **화이트리스트 방식**
   ```python
   # ✅ 좋은 코드
   ALLOWED_DOMAINS = ['cdn.example.com', 'images.example.com']

   def fetch_url(url):
       domain = extract_domain(url)
       if domain not in ALLOWED_DOMAINS:
           return "허용되지 않은 도메인입니다!"
       # 안전한 도메인만 요청
   ```

3. **네트워크 분리**
   ```
   서버 네트워크 구성:
   ┌──────────────────────────────┐
   │ [웹 서버]                    │
   │   ↓ (방화벽)                 │
   │ [내부 서버] ← 접근 차단!      │
   └──────────────────────────────┘

   → 웹 서버가 내부 서버에 접근 못하게 막기
   ```

### 📚 실제 사례

**2019년 Capital One 해킹:**
- 문제: SSRF 취약점을 통해 AWS 메타데이터 서비스에 접근
- 피해: 1억 명 이상의 고객 정보 유출
- 방법: 웹 애플리케이션의 SSRF 취약점 → AWS 자격증명 탈취 → S3 버킷 접근
- 교훈: 사용자 입력으로 URL을 만들 때 철저한 검증 필요

## ❓ 자주 묻는 질문 (FAQ)

### Q1: OWASP TOP 10은 얼마나 자주 바뀌나요?
**A:** 보통 3~4년마다 업데이트됩니다. 최신 버전은 2021년입니다. 다음 업데이트는 2024~2025년 예상됩니다.

### Q2: 이 10가지만 알면 모든 웹 취약점을 막을 수 있나요?
**A:** 아니요. 이것은 "가장 흔한" 10가지입니다. 다른 취약점도 많이 있습니다. 하지만 이 10가지만 잘 막아도 80% 이상의 공격을 방어할 수 있습니다.

### Q3: 개발자가 아닌데도 이걸 알아야 하나요?
**A:** 네! 웹사이트를 사용하는 일반 사용자도 기본적인 보안 개념을 알면:
- 피싱 사이트를 구별할 수 있습니다
- 안전한 비밀번호를 만들 수 있습니다
- 의심스러운 URL을 클릭하지 않게 됩니다

### Q4: 취약점을 발견하면 어떻게 해야 하나요?
**A:**
1. 절대 악용하지 마세요 (불법!)
2. 해당 웹사이트 운영자에게 연락하세요
3. 버그바운티 프로그램이 있으면 신고하세요 (보상 받을 수 있음!)

### Q5: 웹 개발할 때 보안 공부를 언제부터 해야 하나요?
**A:** **지금 바로!** 나중에 고치려면 10배 더 어렵습니다. 처음부터 보안을 고려하세요.

## 🧩 퀴즈: 배운 내용 확인하기

### 퀴즈 1
**상황:** 쇼핑몰 웹사이트에서 URL을 `shop.com/cart?user=123`에서 `shop.com/cart?user=124`로 바꿨더니 다른 사람의 장바구니가 보였습니다.

**질문:** 이것은 어떤 취약점일까요?

<details>
<summary>정답 보기</summary>

**A01: 취약한 접근 통제 (Broken Access Control)**

서버에서 요청한 사람이 정말 user=124의 주인인지 확인하지 않았기 때문입니다.
</details>

### 퀴즈 2
**상황:** 웹사이트 데이터베이스가 해킹당했습니다. 데이터베이스에 저장된 비밀번호:
```
사용자: admin, 비밀번호: password123
사용자: user1, 비밀번호: qwerty
```

**질문:** 이것은 어떤 취약점일까요?

<details>
<summary>정답 보기</summary>

**A02: 암호화 실패 (Cryptographic Failures)**

비밀번호가 암호화되지 않고 "평문(plain text)"으로 저장되어 있습니다. bcrypt 같은 강력한 해시로 저장했어야 합니다.
</details>

### 퀴즈 3
**상황:** 로그인 창에 아래와 같이 입력했더니 로그인이 됐습니다:
```
비밀번호: ' OR 1=1 --
```

**질문:** 이것은 어떤 취약점일까요?

<details>
<summary>정답 보기</summary>

**A03: 인젝션 (Injection) - 특히 SQL Injection**

사용자 입력을 검증 없이 SQL 쿼리에 직접 삽입했기 때문입니다. Prepared Statement를 사용했어야 합니다.
</details>

## 🎯 요약: 핵심만 기억하기

### 🔑 가장 중요한 3가지

1. **사용자 입력은 절대 믿지 마세요**
   - 모든 입력을 검증하고 정제하세요
   - SQL 인젝션, XSS, SSRF 모두 이것 때문에 발생합니다

2. **보안은 "나중에"가 아닙니다**
   - 개발 처음부터 보안을 고려하세요
   - 나중에 고치려면 10배 더 어렵습니다

3. **로그를 남기고 확인하세요**
   - 무슨 일이 일어나는지 알아야 막을 수 있습니다
   - CCTV 없는 은행은 위험합니다
