# 🛡️ 2. 기술적 관리적 보호조치_기초: 개인정보를 지키는 실제 방법들

## 🎯 이 문서에서 배울 내용

이 문서는 **개인정보보호를 실제로 구현하는 방법**에 대해 배우는 기초 가이드입니다. 완전 초보자를 위해 작성되었으며, 다음과 같은 내용을 다룹니다:

- **기술적 보호조치**: 시스템과 컴퓨터로 개인정보를 지키는 방법 (암호화, 접근 통제 등)
- **관리적 보호조치**: 사람과 규칙으로 개인정보를 지키는 방법 (교육, 정책 등)
- **실생활 비유**: 집의 보안 시스템에 비유하여 쉽게 이해
- **AI의 역할**: AI가 어떻게 보안을 자동으로 체크하고 개선하는지
- **실습 예제**: Python 코드로 직접 해볼 수 있는 간단한 예제들

## 🏠 일상생활 비유: 집을 지키는 방법으로 이해하기

개인정보보호의 기술적/관리적 보호조치를 이해하기 위해, 먼저 **여러분의 집을 지키는 방법**을 생각해봅시다.

### 집의 보안 시스템

```
┌─────────────────────────────────────────────────────────┐
│                     우리 집                              │
│                                                         │
│  [기술적 보호조치] - 물리적 장치들                         │
│  ━━━━━━━━━━━━━━━━━━━━━━━━━━━━                          │
│  🔐 현관 도어락     ← 접근 통제                          │
│  📹 CCTV          ← 접근 기록 관리                       │
│  🔒 금고           ← 암호화                             │
│  🚨 경보 시스템     ← 침입 탐지                          │
│  💾 중요 서류 복사본 ← 백업                              │
│                                                         │
│  [관리적 보호조치] - 사람과 규칙                           │
│  ━━━━━━━━━━━━━━━━━━━━━━━━━━━━                          │
│  📋 가족 규칙       ← 보안 정책                          │
│  👨‍👩‍👧‍👦 가족 교육       ← 보안 교육                          │
│  🔍 정기 점검       ← 보안 감사                          │
│  📝 비상 연락망     ← 사고 대응 계획                      │
│                                                         │
└─────────────────────────────────────────────────────────┘
```

### 실제 개인정보보호와의 비교

| 집 보안 | 개인정보보호 | 설명 |
|---------|-------------|------|
| 🔐 **현관 도어락** | **접근 통제** | 허가된 사람만 들어올 수 있게 함 |
| 📹 **CCTV** | **접근 기록 관리** | 누가 언제 들어왔는지 기록 |
| 🔒 **금고** | **암호화** | 중요한 것은 안전하게 보관 |
| 🚨 **경보 시스템** | **침입 탐지 시스템** | 이상한 일이 생기면 즉시 알림 |
| 💾 **서류 복사본** | **백업** | 원본이 손상되어도 복구 가능 |
| 📋 **가족 규칙** | **보안 정책** | 지켜야 할 규칙 명시 |
| 👨‍👩‍👧‍👦 **가족 교육** | **보안 교육** | 모두가 보안의 중요성 이해 |
| 🔍 **정기 점검** | **보안 감사** | 보안 시스템이 잘 작동하는지 확인 |

> **핵심 개념**:
>
> - **기술적 보호조치** = 물리적 보안 장치 (도어락, CCTV, 금고 등)
> - **관리적 보호조치** = 사람과 규칙 (가족 교육, 집안 규칙 등)
> - 두 가지가 **함께 작동**해야 집이 안전하듯, 개인정보도 기술과 관리가 함께 필요!

## 🤔 왜 기술적/관리적 보호조치가 필요할까요?

### 법을 아는 것만으로는 부족합니다

지난 시간에 배운 **개인정보보호법**은 "무엇을 해야 하는가"를 알려줍니다.
하지만 **실제로 어떻게 하는가**는 별개의 문제입니다!

**비유로 이해하기:**

```
법규:          "차는 안전하게 운행해야 한다"
  ↓
기술적 조치:    에어백, ABS, 안전벨트 설치
관리적 조치:    운전자 교육, 정기 점검, 안전 수칙
```

### 실제 사례: 보호조치 없이 일어난 사고

**❌ 나쁜 사례: 2014년 카드사 개인정보 유출**

```
문제 상황:

│   - 대량 다운로드 탐지 없음         │
│                                     │
│ ❌ 관리적 문제:                      │
│   - 보안 교육 부족                  │
│   - 접근 권한 관리 미흡             │
│   - 정기 감사 없음                  │
│                                     │
│ 결과: 1억 명 정보 유출! 💥          │
└─────────────────────────────────────┘
```

**✅ 좋은 사례: 제대로 보호조치를 한 A은행**

```
보호 시스템:
┌─────────────────────────────────────┐
│ 직원이 고객정보에 접근 시도         │
│           ↓                         │

│   - 2단계 인증 요구 (MFA)           │

│                                     │
│ ✅ 관리적 보호:                      │
│   - 분기별 보안 교육 실시           │
│   - 최소 권한 원칙 적용             │
│   - 매월 보안 감사 수행             │
│                                     │
│ 결과: 10년간 개인정보 유출 0건! ✨   │
└─────────────────────────────────────┘
```

### 보호조치가 중요한 이유

1. **법규 위반 방지**: 벌금 최대 5천만원, 매출의 3% (더 큰 금액 적용)
2. **고객 신뢰 유지**: 개인정보 유출 시 고객 이탈
3. **비즈니스 연속성**: 정보 손실 시 사업 중단 가능
4. **사회적 책임**: 개인의 프라이버시 보호는 기업의 의무

## 1. 🛡️ 기술적 보호조치: 컴퓨터로 개인정보 지키기

기술적 보호조치는 **컴퓨터, 서버, 네트워크 등의 기술을 이용해** 개인정보를 보호하는 방법입니다.

### 1.1. 🔑 접근 통제 (Access Control): 허가된 사람만 들어오게!

#### 기본 개념

**접근 통제**란 개인정보가 저장된 시스템에 **권한이 있는 사람만 접근**할 수 있도록 제한하는 것입니다.

**일상생활 비유:**

```
🏢 회사 건물 출입

1층 로비:        누구나 들어올 수 있음
  ↓ (사원증)
2층 일반 사무실:  직원만 들어올 수 있음
  ↓ (관리자 카드)
3층 서버실:      IT 관리자만 들어올 수 있음
  ↓ (생체 인증)
4층 임원실:      임원과 CEO만 들어올 수 있음
```

#### 접근 통제의 핵심 원칙

**1) 최소 권한 원칙 (Least Privilege)**

사람들에게 **업무에 필요한 최소한의 권한만** 부여합니다.

```
예시: 쇼핑몰 직원의 권한

❌ 나쁜 예:
모든 직원 → 고객의 모든 정보 접근 가능
(주소, 전화번호, 구매 이력, 카드 번호 등)

✅ 좋은 예:
├─ 고객센터 직원 → 이름, 전화번호만 볼 수 있음
├─ 배송 담당자   → 이름, 주소만 볼 수 있음
└─ 결제 담당자   → 주문 금액만 볼 수 있음 (카드 번호는 암호화)
```

**2) 역할 기반 접근 통제 (RBAC: Role-Based Access Control)**

**역할**에 따라 권한을 부여하는 방식입니다.

```
역할별 권한 예시:

┌──────────────┬────────┬────────┬────────┬──────────┐
│   역할       │ 읽기   │ 쓰기   │ 수정   │  삭제    │
├──────────────┼────────┼────────┼────────┼──────────┤
│ 👨‍💼 CEO       │   ✅   │   ✅   │   ✅   │    ✅    │
│ 👨‍💻 관리자     │   ✅   │   ✅   │   ✅   │    ❌    │
│ 👨‍🏫 매니저     │   ✅   │   ✅   │   ❌   │    ❌    │
│ 👤 일반 직원  │   ✅   │   ❌   │   ❌   │    ❌    │
└──────────────┴────────┴────────┴────────┴──────────┘
```

**3) 다중 인증 (MFA: Multi-Factor Authentication)**

**여러 단계**를 거쳐 본인임을 확인합니다.

```
🔐 MFA의 3가지 요소:

1️⃣ 알고 있는 것 (Something you know)
   → 비밀번호, PIN 번호

2️⃣ 가지고 있는 것 (Something you have)
   → 스마트폰 앱, 보안 카드, USB 키

3️⃣ 자신인 것 (Something you are)
   → 지문, 얼굴 인식, 홍채

예시: 은행 앱 로그인
  ① 비밀번호 입력 (알고 있는 것)
  ② 문자로 받은 인증 번호 입력 (가지고 있는 것)
  ③ 지문 인식 (자신인 것)
```

#### 접근 통제 실습: Python으로 간단한 RBAC 만들기

**초보자를 위한 단계별 설명:**

```python
# === 1단계: 사용자 클래스 만들기 ===
class User:
    """사용자 정보를 저장하는 클래스"""
    def __init__(self, username, roles):
        self.username = username  # 사용자 이름
        self.roles = roles        # 사용자의 역할 목록 (예: ["manager", "staff"])

# === 2단계: 접근 통제 클래스 만들기 ===
class AccessControl:
    """역할에 따라 권한을 관리하는 클래스"""

    def __init__(self):
        # 각 역할별로 허용된 작업들을 정의
        self.permissions = {
            "admin": ["read_pii", "write_pii", "delete_pii", "manage_users"],
            "manager": ["read_pii", "write_pii"],
            "staff": ["read_pii"]
        }

    def has_permission(self, user, action):
        """
        사용자가 특정 작업을 할 수 있는지 확인

        매개변수:
            user: User 객체
            action: 확인할 작업 (예: "read_pii")

        반환값:
            True: 권한 있음
            False: 권한 없음
        """
        # 사용자의 모든 역할을 확인
        for role in user.roles:
            # 이 역할이 요청한 작업을 할 수 있는지 확인
            if role in self.permissions and action in self.permissions[role]:
                return True
        return False

# === 3단계: 실제 사용 예시 ===
# 접근 통제 시스템 생성
ac = AccessControl()

# 다양한 역할을 가진 사용자 생성
admin_user = User("김철수", ["admin"])
manager_user = User("이영희", ["manager"])
staff_user = User("박민수", ["staff"])

print("=" * 50)
print("🔑 접근 통제 시스템 테스트")
print("=" * 50)

# 각 사용자가 다양한 작업을 할 수 있는지 테스트
print("\n1️⃣ 관리자(김철수)의 권한:")
print(f"   개인정보 읽기: {ac.has_permission(admin_user, 'read_pii')}")
print(f"   개인정보 삭제: {ac.has_permission(admin_user, 'delete_pii')}")

print("\n2️⃣ 매니저(이영희)의 권한:")
print(f"   개인정보 읽기: {ac.has_permission(manager_user, 'read_pii')}")
print(f"   개인정보 삭제: {ac.has_permission(manager_user, 'delete_pii')}")

print("\n3️⃣ 일반 직원(박민수)의 권한:")
print(f"   개인정보 읽기: {ac.has_permission(staff_user, 'read_pii')}")
print(f"   사용자 관리: {ac.has_permission(staff_user, 'manage_users')}")

print("\n" + "=" * 50)
```

**출력 결과:**

```
==================================================
🔑 접근 통제 시스템 테스트
==================================================

1️⃣ 관리자(김철수)의 권한:
   개인정보 읽기: True
   개인정보 삭제: True

2️⃣ 매니저(이영희)의 권한:
   개인정보 읽기: True
   개인정보 삭제: False

3️⃣ 일반 직원(박민수)의 권한:
   개인정보 읽기: True
   사용자 관리: False

==================================================
```

#### 접근 기록 관리: 누가 언제 접근했는지 기록

모든 접근은 **로그(기록)**로 남겨야 합니다.

**로그에 포함되어야 할 정보:**

```
접근 로그 예시:
┌─────────────────────────────────────────────────────────────┐
│ 시간: 2024-11-22 14:30:25                                   │
│ 사용자: 이영희 (manager)                                     │
│ IP 주소: 192.168.1.100                                      │
│ 작업: 고객 정보 조회                                         │
│ 대상: customer_id=12345                                     │
│ 결과: 성공                                                   │
└─────────────────────────────────────────────────────────────┘
```

**왜 중요할까요?**

- **사고 추적**: 문제가 생겼을 때 누가 무엇을 했는지 추적
- **내부자 위협 탐지**: 직원의 비정상적인 접근 패턴 발견
- **법적 증거**: 개인정보 유출 시 조사 자료로 활용

### 1.2. 🔐 암호화 (Encryption): 내용을 알아볼 수 없게!

#### 기본 개념

**암호화**란 정보를 **특수한 방법으로 변환**하여, 열쇠(키)가 없으면 읽을 수 없도록 만드는 것입니다.

**일상생활 비유:**

```
📧 편지 보내기

일반 편지:
"안녕하세요. 제 주민등록번호는 123456-1234567입니다."
  → 누구나 읽을 수 있음 ❌

암호화된 편지:
"Xk#9mP@qR5$nW7&tL2!dF4*gH8^jK1%sA6"
  → 암호를 아는 사람만 읽을 수 있음 ✅
```

#### 두 가지 암호화: 저장 vs 전송

**1) 저장 시 암호화 (Encryption at Rest)**

데이터베이스나 파일에 **저장할 때** 암호화합니다.

```
데이터베이스에 저장:

❌ 암호화 안 함:
┌─────────────────────────────────────┐
│ 이름    │ 전화번호      │ 주민번호    │
├─────────────────────────────────────┤
│ 김철수  │ 010-1234-5678 │ 800101-... │
│ 이영희  │ 010-9876-5432 │ 900202-... │
└─────────────────────────────────────┘
→ DB를 해킹당하면 즉시 정보 유출! 💥

✅ 암호화 함:
┌─────────────────────────────────────┐
│ 이름    │ 전화번호          │ 주민번호        │
├─────────────────────────────────────┤
│ 김철수  │ 010-1234-5678     │ Xk#9mP@qR5$... │
│ 이영희  │ 010-9876-5432     │ Yw$7nQ!tL3^... │
└─────────────────────────────────────┘
→ DB를 해킹당해도 주민번호는 읽을 수 없음! ✅
```

**2) 전송 시 암호화 (Encryption in Transit)**

네트워크로 **전송할 때** 암호화합니다.

```
웹사이트 접속:

❌ HTTP (암호화 안 함):
   사용자 → [평문 데이터] → 서버
              ↑
         누구나 엿볼 수 있음!

✅ HTTPS (암호화 함):
   사용자 → [암호화된 데이터] → 서버
              ↑
         엿봐도 읽을 수 없음!
```

**웹사이트 주소 확인하기:**

```
❌ http://example.com
   → 자물쇠 표시 없음, 암호화 안 됨

✅ https://example.com
   → 🔒 자물쇠 표시, 암호화 됨
```

#### 암호화 알고리즘: 무엇을 사용해야 할까?

**강력한 암호화 (사용 권장 ✅)**

```
대칭키 암호화:
  AES-256: 가장 널리 사용되는 강력한 암호화

비대칭키 암호화:
  RSA-2048 이상: 공개키 암호화에 사용

해시 함수:
  SHA-256 이상: 데이터 무결성 검증
  bcrypt, scrypt, Argon2: 비밀번호 저장용
```

**약한 암호화 (사용 금지 ❌)**

```
MD5, SHA-1: 이미 깨진 알고리즘
DES, 3DES: 너무 약한 암호화
```

#### 암호화 실습: Python으로 데이터 암호화하기

**초보자를 위한 단계별 설명:**

```python
# 암호화 라이브러리 가져오기
from cryptography.fernet import Fernet

# === 1단계: 암호화 키 생성 ===
def generate_key():
    """
    암호화에 사용할 키를 생성합니다.
    이 키는 안전한 곳에 보관해야 합니다!
    """
    key = Fernet.generate_key()
    print(f"🔑 생성된 암호화 키: {key}")
    return key

# === 2단계: 데이터 암호화 ===
def encrypt_data(data, key):
    """
    평문 데이터를 암호화합니다.

    매개변수:
        data: 암호화할 텍스트 (예: "주민번호: 123456-1234567")
        key: 암호화 키

    반환값:
        암호화된 데이터 (읽을 수 없는 형태)
    """
    f = Fernet(key)
    encrypted_data = f.encrypt(data.encode())
    return encrypted_data

# === 3단계: 데이터 복호화 ===
def decrypt_data(encrypted_data, key):
    """
    암호화된 데이터를 원래 형태로 복원합니다.
    올바른 키가 필요합니다!

    매개변수:
        encrypted_data: 암호화된 데이터
        key: 복호화 키 (암호화할 때 사용한 키와 동일해야 함)

    반환값:
        원본 텍스트
    """
    f = Fernet(key)
    decrypted_data = f.decrypt(encrypted_data).decode()
    return decrypted_data

# === 4단계: 실제 사용 예시 ===
print("=" * 60)
print("🔐 개인정보 암호화 실습")
print("=" * 60)

# 보호해야 할 개인정보
original_data = "주민등록번호: 800101-1234567, 카드번호: 1234-5678-9012-3456"
print(f"\n📄 원본 데이터:\n   {original_data}")

# 암호화 키 생성
key = generate_key()

# 데이터 암호화
encrypted = encrypt_data(original_data, key)
print(f"\n🔒 암호화된 데이터:\n   {encrypted}")
print("   → 이 상태로는 누구도 읽을 수 없습니다!")

# 데이터 복호화 (올바른 키 사용)
decrypted = decrypt_data(encrypted, key)
print(f"\n🔓 복호화된 데이터:\n   {decrypted}")
print(f"\n✅ 원본과 복호화된 데이터 일치 여부: {original_data == decrypted}")

# 잘못된 키로 복호화 시도
print("\n" + "=" * 60)
print("❌ 잘못된 키로 복호화 시도")
print("=" * 60)
wrong_key = Fernet.generate_key()
try:
    decrypt_data(encrypted, wrong_key)
except Exception as e:
    print(f"❌ 복호화 실패! 올바른 키가 필요합니다.")
    print(f"   에러 메시지: {type(e).__name__}")

print("\n" + "=" * 60)
```

**출력 결과:**

```
============================================================

============================================================

📄 원본 데이터:
   주민등록번호: 800101-1234567, 카드번호: 1234-5678-9012-3456

🔑 생성된 암호화 키: b'a9K7m3P...'

🔒 암호화된 데이터:
   b'gAAAAABmXk9...'
   → 이 상태로는 누구도 읽을 수 없습니다!

🔓 복호화된 데이터:

#### 비밀번호는 특별하게: 해싱

**일반 암호화와 해싱의 차이:**

```
암호화 (Encryption):
  평문 → [암호화] → 암호문 → [복호화] → 평문
  양방향 (되돌릴 수 있음)

해싱 (Hashing):
  평문 → [해싱] → 해시값
  단방향 (되돌릴 수 없음)
```

**비밀번호 저장 방법:**

```
❌ 나쁜 예: 평문 저장
┌──────────┬──────────┐
│ 사용자   │ 비밀번호 │
├──────────┼──────────┤
│ 김철수   │ pass1234 │
│ 이영희   │ qwerty   │
└──────────┴──────────┘
→ DB 해킹 시 모든 비밀번호 유출!

❌ 나쁜 예: 약한 해시 (MD5)
┌──────────┬──────────────────────────────────┐
│ 사용자   │ 비밀번호 (MD5)                   │
├──────────┼──────────────────────────────────┤
│ 김철수   │ 5f4dcc3b5aa765d61d8327deb882cf99 │
└──────────┴──────────────────────────────────┘
→ 레인보우 테이블로 쉽게 역추적!

✅ 좋은 예: 강력한 해시 (bcrypt)
┌──────────┬──────────────────────────────────────────────────────────┐
│ 사용자   │ 비밀번호 (bcrypt)                                        │
├──────────┼──────────────────────────────────────────────────────────┤
│ 김철수   │ $2b$12$KIXxKJ4GEM9JDsGHvK7qDOeQbVqC9Y6QfJlTY1cX9ZvEqm... │
└──────────┴──────────────────────────────────────────────────────────┘
→ Salt 포함, 역추적 거의 불가능!
```

### 1.3. 🛡️ 보안 프로그램: 바이러스와 해커 막기

#### 기본 개념

**보안 프로그램**은 컴퓨터를 위협하는 **악성 소프트웨어와 해킹 공격**을 막아주는 도구들입니다.

**일상생활 비유:**

```
신체의 면역 시스템:

백혈구    → 백신 프로그램 (바이러스 퇴치)
피부      → 방화벽 (외부 침입 차단)
경보 시스템 → 침입 탐지 시스템 (이상 징후 감지)
```

#### 주요 보안 프로그램

**1) 백신 프로그램 (Anti-Virus)**

```
백신의 역할:
┌─────────────────────────────────────┐
│ 파일 다운로드                        │
│         ↓                           │
│ [백신 검사] ← 바이러스 데이터베이스  │
│         ↓                           │
│ ✅ 안전 → 실행 허용                  │
│ ❌ 위험 → 격리 또는 삭제             │
└─────────────────────────────────────┘

주요 기능:
  - 실시간 감시
  - 예약 검사
  - 바이러스 정의 자동 업데이트
```

**2) 방화벽 (Firewall)**

```
방화벽의 역할:

인터넷 ← → [방화벽] ← → 내부 네트워크
           ↓
      규칙 확인:
      ✅ 허용된 트래픽: 통과
      ❌ 차단된 트래픽: 차단

예시 규칙:
  - 웹 서버(80, 443 포트): 허용
  - SSH(22 포트): 특정 IP만 허용
  - 나머지 모든 포트: 차단
```

**3) 침입 탐지/방지 시스템 (IDS/IPS)**

```
IDS (Intrusion Detection System):
  이상한 활동을 '탐지'하고 '알림'

IPS (Intrusion Prevention System):
  이상한 활동을 '탐지'하고 '차단'

작동 방식:
┌─────────────────────────────────────┐
│ 네트워크 트래픽 모니터링             │
│         ↓                           │
│ 비정상 패턴 탐지                     │
│   - 포트 스캔                        │
│   - SQL 인젝션 시도                  │
│   - 대량 데이터 전송                 │
│         ↓                           │
│ IDS: 관리자에게 알림 📧              │
│ IPS: 즉시 차단 🚫                   │
└─────────────────────────────────────┘
```

**4) 웹 방화벽 (WAF: Web Application Firewall)**

```
웹 공격 전문 방어:

사용자 → [WAF] → 웹 서버
         ↓
    공격 차단:
    - SQL 인젝션
    - XSS (크로스 사이트 스크립팅)
    - CSRF
    - 파일 업로드 공격
```

#### 보안 업데이트: 왜 중요할까?

**보안 패치가 중요한 이유:**

```
소프트웨어 취약점 발견:
┌─────────────────────────────────────┐
│ 1. 보안 연구자가 취약점 발견         │
│         ↓                           │
│ 2. 제조사에 신고                     │
│         ↓                           │
│ 3. 제조사가 패치 개발                │
│         ↓                           │
│ 4. 보안 업데이트 배포               │
│         ↓                           │
│ ✅ 사용자가 업데이트 적용            │
│ ❌ 사용자가 업데이트 무시            │
│         ↓                           │
│    해커가 취약점 악용! 💥            │
└─────────────────────────────────────┘
```

**실제 사례: WannaCry 랜섬웨어 (2017년)**

```
상황:
1. Microsoft가 Windows 보안 패치 발표 (2017년 3월)

결과:

### 1.4. 💾 백업과 복구: 만약을 대비하기

#### 기본 개념

**백업**은 중요한 데이터의 **복사본을 안전한 곳에 보관**하는 것입니다.

**일상생활 비유:**

```
중요한 서류 보관:

원본:     집 서재에 보관
복사본 1: 은행 금고에 보관
복사본 2: 부모님 집에 보관
복사본 3: 클라우드에 스캔 파일 저장

→ 집에 화재가 나도 다른 곳에 복사본이 있어서 안전!
```

#### 3-2-1 백업 규칙

가장 널리 알려진 백업 전략입니다:

```
3-2-1 규칙:
┌─────────────────────────────────────────────────────┐
│ 3: 데이터의 복사본을 3개 만들기                       │
│    (원본 1개 + 백업 2개)                             │
│                                                      │
│ 2: 2가지 다른 저장 매체에 보관                        │
│    (예: 하드디스크 + 클라우드)                        │
│                                                      │
│ 1: 1개는 외부(오프사이트)에 보관                      │
│    (예: 다른 건물, 다른 지역)                         │
└─────────────────────────────────────────────────────┘

예시:
├─ 원본: 회사 서버 (서울 본사)
├─ 백업 1: 외장 하드디스크 (서울 본사)
└─ 백업 2: 클라우드 스토리지 (해외 데이터센터)
```

#### 백업 전략

**1) 전체 백업 (Full Backup)**

```
모든 데이터를 완전히 복사

장점: 복구 시 한 번에 모두 복원 가능
단점: 시간이 오래 걸리고, 저장 공간 많이 필요

예시:
월요일: [전체 데이터 복사] → 100GB
```

**2) 증분 백업 (Incremental Backup)**

```
마지막 백업 이후 변경된 부분만 백업

장점: 빠르고, 저장 공간 적게 필요
단점: 복구 시 여러 백업 파일 필요

예시:
월요일: [전체 백업] → 100GB
화요일: [변경분만] → 5GB
수요일: [변경분만] → 3GB
```

**3) 차등 백업 (Differential Backup)**

```
마지막 전체 백업 이후 변경된 모든 부분 백업

장점: 복구가 비교적 간단 (전체 + 최신 차등)
단점: 시간이 갈수록 백업 크기 증가

예시:
월요일: [전체 백업] → 100GB
화요일: [월요일 이후 변경분] → 5GB
수요일: [월요일 이후 변경분] → 8GB
```

#### RTO와 RPO: 얼마나 빨리, 얼마나 많이?

**복구 목표 시간 (RTO: Recovery Time Objective)**

```
RTO = 시스템을 얼마나 빨리 복구해야 하는가?

예시:
┌────────────────┬──────────┬──────────────┐
│ 시스템 중요도  │ RTO      │ 의미         │
├────────────────┼──────────┼──────────────┤
│ 매우 중요      │ 1시간    │ 1시간 내 복구│
│ 중요           │ 4시간    │ 4시간 내 복구│
│ 보통           │ 24시간   │ 하루 내 복구 │
└────────────────┴──────────┴──────────────┘
```

**복구 목표 시점 (RPO: Recovery Point Objective)**

```
RPO = 얼마만큼의 데이터 손실을 감수할 수 있는가?

예시:
┌────────────────┬──────────┬──────────────────┐
│ 데이터 중요도  │ RPO      │ 의미             │
├────────────────┼──────────┼──────────────────┤
│ 매우 중요      │ 0분      │ 손실 불가        │
│ 중요           │ 1시간    │ 1시간 데이터 손실│
│ 보통           │ 24시간   │ 하루 데이터 손실 │
└────────────────┴──────────┴──────────────────┘
```

**시각적 이해:**

```
타임라인:
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━→ 시간
    ↑              ↑                      ↑
  마지막 백업     사고 발생             복구 완료
    |←───── RPO ────→|
                      |←──── RTO ─────→|

RPO: 마지막 백업 시점까지만 복구 가능 (이 사이 데이터는 손실)
RTO: 사고 발생부터 복구 완료까지 걸리는 시간
```

#### 백업도 암호화!

백업 데이터에도 개인정보가 포함되어 있으므로 **반드시 암호화**해야 합니다.

```
❌ 나쁜 예:
원본 DB:     [암호화됨] ✅
백업 파일:   [평문] ❌
  → 백업을 탈취당하면 정보 유출!

✅ 좋은 예:
원본 DB:     [암호화됨] ✅
백업 파일:   [암호화됨] ✅
  → 백업을 탈취당해도 안전!
```

## 2. 📝 관리적 보호조치: 사람과 규칙으로 지키기

관리적 보호조치는 **사람, 정책, 절차**를 통해 개인정보를 보호하는 방법입니다.

**왜 중요할까요?**

```
보안 사고의 원인:

📊 통계:
├─ 기술적 문제: 30%
└─ 인적 오류: 70% ⚠️

즉, 대부분의 보안 사고는 "사람의 실수"로 발생!
→ 관리적 보호조치가 매우 중요!
```

### 2.1. 📜 개인정보보호 정책: 우리의 규칙

#### 기본 개념

**개인정보보호 정책**은 조직이 개인정보를 어떻게 다룰 것인지에 대한 **공식적인 약속과 규칙**입니다.

**일상생활 비유:**

```
학교/회사의 규칙:

학교:
  - 복장 규정
  - 출석 규칙
  - 시험 규정

회사:
  - 근무 시간 규정
  - 비밀 유지 규정
  - 개인정보보호 정책 ← 우리가 배우는 것!
```

#### 개인정보 처리 방침

**모든 회사가 공개해야 하는 문서**입니다.

**포함되어야 할 내용:**

```
개인정보 처리 방침에 포함될 내용:

1. 📋 수집하는 개인정보 항목
   예: 이름, 이메일, 전화번호, 배송 주소

2. 🎯 수집 목적
   예: 회원 가입, 상품 배송, 고객 상담

3. ⏰ 보유 기간
   예: 회원 탈퇴 시까지, 또는 3년

4. 🗑️ 파기 방법
   예: 암호화된 데이터는 복구 불가능한 방법으로 삭제

5. 👥 제3자 제공 여부
   예: 배송 업체에 주소 제공 (동의 필요)

6. 🔒 안전성 확보 조치
   예: 암호화, 접근 통제, 백업

7. 📞 개인정보 보호 책임자
   예: 홍길동 (privacy@company.com)
```

```

│ 2. 수집 목적                                            │
│    - 회원 가입 및 관리                                   │

│                                                         │

│                                                         │

│    - 이름: 홍길동                                        │
│    - 연락처: privacy@abcshop.com                        │
│    - 전화: 02-1234-5678                                 │
│                                                         │

#### 내부 관리 계획

**직원들이 지켜야 할 내부 규칙**입니다.

```
내부 관리 계획 예시:

1. 📋 개인정보 보호 책임자 지정
   - CPO (Chief Privacy Officer) 임명
   - 각 부서별 개인정보 관리자 지정

2. 🔐 접근 권한 관리
   - 역할별 최소 권한 부여
   - 퇴사자 권한 즉시 회수
   - 6개월마다 권한 재검토

   - 연 2회 필수 교육

   - 개인정보 취급 부서 심화 교육

4. 🔍 접근 기록 보관
   - 모든 개인정보 접근 로그 기록
   - 최소 3년 보관
   - 월 1회 이상 로그 검토

5. 📝 개인정보 영향평가
   - 신규 시스템 도입 전 필수
   - 위험 요소 사전 분석

6. 🚨 사고 대응 절차
   - 유출 사고 발생 시 24시간 내 보고
   - 사고 조사 및 재발 방지 대책 수립
```

### 2.3. 📝 개인정보 영향평가 (PIA)

#### 기본 개념

**PIA (Privacy Impact Assessment)**는 새로운 시스템을 만들거나 변경할 때, **미리 개인정보 침해 위험을 평가**하는 것입니다.

**일상생활 비유:**

```
새 건물 짓기:

설계 단계:
┌─────────────────────────────────────┐
│ 건축하기 전 안전 검사                │
│   - 구조 안전성 평가                 │
│   - 화재 대피로 확인                 │
│   - 지진 대비 설계 검토              │
│                                     │
│ → 문제 발견 시 설계 단계에서 수정!  │
│ → 건물 지은 후에 고치면 비용 100배! │
└─────────────────────────────────────┘

PIA도 마찬가지:
시스템 개발 전 → 개인정보 위험 평가
→ 미리 문제 발견하고 수정!
```

#### PIA를 해야 하는 경우

```
PIA 의무 대상:

1. 📊 대규모 개인정보 처리
   - 100만 명 이상의 개인정보 보유
   - 또는 민감정보/고유식별정보 10만 명 이상

2. 🆕 새로운 시스템 도입
   - 개인정보 처리 시스템 신규 구축
   - 기존 시스템의 중요한 변경

3. 🔗 개인정보 연계/결합
   - 여러 데이터베이스를 연결하는 경우

4. 📹 영상정보처리기기 (CCTV)
   - 공개된 장소에 30대 이상 설치
```

#### PIA 평가 절차

```
PIA 7단계:

1단계: 계획 수립
  └─ 평가 범위, 일정, 담당자 결정

2단계: 개인정보 흐름 분석
  └─ 어떤 개인정보가 어디서 어디로 이동하나?

3단계: 법적 요구사항 검토
  └─ 개인정보보호법 준수 여부 확인

4단계: 위험 식별
  └─ 어떤 침해 위험이 있나?

5단계: 위험 평가
  └─ 위험의 심각도와 발생 가능성 평가

6단계: 보호대책 수립
  └─ 위험을 줄이기 위한 대책 마련

7단계: 평가 보고서 작성
  └─ 결과를 문서화하고 보고
```

#### 위험 평가 매트릭스

```
위험도 = 발생 가능성 × 영향도

┌────────────────────────────────────────────┐
│         발생 가능성 →                       │
│  영향도  │ 낮음  │ 보통  │ 높음  │         │
│    ↓     │  (1)  │  (2)  │  (3)  │         │
│ ─────────┼───────┼───────┼───────┤         │
│ 높음(3)  │   3   │   6   │   9   │ ← 매우 위험│
│ 보통(2)  │   2   │   4   │   6   │ ← 위험   │
│ 낮음(1)  │   1   │   2   │   3   │ ← 주의   │
└────────────────────────────────────────────┘

위험도별 조치:
  7-9: 즉시 조치 필요 (프로젝트 중단 고려)
  4-6: 대책 마련 필수
  1-3: 모니터링
```

**예시:**

```
예시 1: 암호화 안 된 개인정보 저장
  - 영향도: 높음 (3)
  - 발생 가능성: 높음 (3)
  - 위험도: 9 (매우 위험!)
  - 조치: 즉시 암호화 적용

예시 2: 직원의 권한 과부여
  - 영향도: 보통 (2)
  - 발생 가능성: 보통 (2)
  - 위험도: 4 (위험)
  - 조치: 최소 권한 원칙 적용
```

### 2.4. 🔍 보안 감사와 점검: 제대로 지켜지는지 확인

#### 기본 개념

**보안 감사**는 우리가 세운 보안 정책과 기술적 조치가 **실제로 잘 지켜지고 있는지 확인**하는 것입니다.

**일상생활 비유:**

```
자동차 정기 점검:

6개월마다:
┌─────────────────────────────────────┐
│ 정비소 방문                          │
│   - 엔진 오일 상태 확인              │
│   - 브레이크 점검                    │
│   - 타이어 마모 상태 확인            │
│   - 각종 센서 작동 확인              │
│                                     │
│ → 문제 발견 시 즉시 수리            │
│ → 큰 사고 예방!                     │
└─────────────────────────────────────┘

보안 감사도 마찬가지:
정기적으로 보안 상태 점검
→ 문제 발견 시 즉시 개선
→ 큰 사고 예방!
```

#### 감사 종류

```
1. 🏢 내부 감사
   ────────────────────────
   - 회사 내부 감사팀이 수행
   - 정기 감사: 연 1-2회
   - 수시 감사: 사고 발생 시

   장점: 내부 사정을 잘 알고 있음
   단점: 객관성 부족 가능

2. 🏛️ 외부 감사
   ────────────────────────
   - 전문 감사 기관에 의뢰
   - 공인된 인증 취득 (ISO 27001 등)

   장점: 객관적, 전문적
   단점: 비용 발생

3. 🤖 자동화 감사
   ────────────────────────
   - AI/도구를 이용한 실시간 감사
   - 24시간 모니터링

   장점: 빠르고 지속적
   단점: 초기 구축 비용
```

#### 감사 체크리스트

**감사 시 확인하는 항목들:**

```
✅ 기술적 보호조치 점검
──────────────────────────
□ 접근 통제
  ├─ MFA 적용 여부
  ├─ 비밀번호 정책 준수
  ├─ 접근 기록 보관 상태
  └─ 불필요한 계정 존재 여부

□ 암호화
  ├─ 민감 정보 암호화 여부
  ├─ 강력한 알고리즘 사용 여부
  ├─ 키 관리 적절성
  └─ HTTPS 사용 여부

□ 보안 프로그램
  ├─ 백신 최신 업데이트 여부
  ├─ 방화벽 설정 적절성
  ├─ 보안 패치 적용 여부
  └─ IDS/IPS 작동 상태

□ 백업
  ├─ 정기 백업 수행 여부
  ├─ 백업 데이터 암호화 여부
  ├─ 복구 테스트 수행 여부
  └─ 백업 데이터 보관 장소

✅ 관리적 보호조치 점검
──────────────────────────
□ 정책 및 절차
  ├─ 개인정보 처리 방침 최신화
  ├─ 내부 관리 계획 수립
  ├─ 책임자 지정 및 역할 명확
  └─ 사고 대응 절차 수립

□ 교육
  ├─ 정기 교육 실시 여부
  ├─ 교육 이수율
  ├─ 교육 효과 측정
  └─ 교육 자료 최신화

```python
import pandas as pd

access_logs = pd.DataFrame({
    'timestamp': [
        '2024-11-22 09:00',
        '2024-11-22 09:30',
        '2024-11-22 02:00',  # 새벽 접속 (비정상)
        '2024-11-22 10:00',
        '2024-11-22 10:05'
    ],

})

print("=" * 70)
print("🤖 AI 기반 보안 이상 탐지 시스템")

# === 2. 비정상 시간대 접속 탐지 ===
print("\n1️⃣ 비정상 시간대 접속 탐지")
print("-" * 70)

for idx, row in access_logs.iterrows():
    hour = int(row['timestamp'].split()[1].split(':')[0])

    # 새벽 시간 (00시~06시) 접속 체크
    if 0 <= hour < 6:
        print(f"🚨 경고: {row['user']}님이 새벽 {hour}시에 접속했습니다!")
        print(f"   시간: {row['timestamp']}")
        print(f"   IP: {row['ip']}")
        print(f"   → 본인 접속이 맞는지 확인 필요!")

# === 3. 비정상 IP 접속 탐지 ===
print("\n2️⃣ 비정상 IP 접속 탐지")
print("-" * 70)

# 내부 IP 범위: 192.168.x.x
for idx, row in access_logs.iterrows():
    if not row['ip'].startswith('192.168'):
        print(f"🚨 경고: {row['user']}님이 외부 IP에서 접속했습니다!")
        print(f"   IP: {row['ip']}")
        print(f"   시간: {row['timestamp']}")
        print(f"   → VPN 사용 여부 확인 필요!")

# === 4. 대량 데이터 접근 탐지 ===
print("\n3️⃣ 대량 데이터 접근 탐지")
print("-" * 70)

for idx, row in access_logs.iterrows():
    if row['rows_accessed'] > 10000:
        print(f"🚨 경고: {row['user']}님이 대량의 데이터에 접근했습니다!")
        print(f"   접근 행 수: {row['rows_accessed']:,}행")
        print(f"   시간: {row['timestamp']}")
        print(f"   → 업무상 필요한 접근인지 확인 필요!")

# === 5. 종합 보안 점수 계산 ===
print("\n4️⃣ 종합 보안 점수")
print("-" * 70)

total_alerts = 0
if any(0 <= int(row['timestamp'].split()[1].split(':')[0]) < 6
       for idx, row in access_logs.iterrows()):
    total_alerts += 1

if any(not row['ip'].startswith('192.168')
       for idx, row in access_logs.iterrows()):
    total_alerts += 1

if any(row['rows_accessed'] > 10000
       for idx, row in access_logs.iterrows()):
    total_alerts += 1

security_score = max(0, 100 - (total_alerts * 20))

print(f"전체 보안 점수: {security_score}/100")
if security_score >= 80:
    print("✅ 양호: 보안 상태가 양호합니다.")
elif security_score >= 60:
    print("⚠️ 주의: 일부 보안 이슈가 발견되었습니다.")
else:
    print("🚨 위험: 즉시 보안 점검이 필요합니다!")

print("\n" + "=" * 70)
```

```
비유:

  기술적 = 에어백, ABS, 안전벨트

   - 국내에 데이터 보관
   - 또는 국내 리전 제공 업체 이용
     (AWS 서울 리전, Azure 한국 리전 등)
```

### Q7: 비밀번호를 자주 바꾸라고 해야 하나요?

**A:** 최신 보안 가이드라인에서는 **정기 변경보다 강력한 비밀번호와 MFA**를 권장합니다!

```
❌ 옛날 방식:
   "비밀번호를 3개월마다 변경하세요"

   문제점:
   - 사람들이 쉬운 비밀번호 사용
     (password1, password2, password3)
   - 포스트잇에 적어둠
   - 패턴이 뻔함

✅ 최신 방식:
   "강력한 비밀번호 + MFA 사용"

   조건:
   - 12자 이상
   - 대소문자, 숫자, 특수문자
   - 사전 단어 사용 금지
   - 다른 사이트와 다른 비밀번호
   - 비밀번호 관리자 사용 (1Password, Bitwarden)
   - MFA 필수

   변경 시점:
   - 유출 의심될 때
   - 퇴사자가 알고 있을 때
   - 보안 사고 발생 시
```

### 퀴즈 1: 다음 중 기술적 보호조치가 아닌 것은?

```
A) 데이터베이스 암호화
B) 직원 보안 교육
C) 방화벽 설치
D) 백업 시스템 구축

정답: B
해설: 직원 보안 교육은 관리적 보호조치입니다.
```

### 퀴즈 2: 최소 권한 원칙의 의미는?

```
A) 모든 직원에게 최소한의 급여만 지급
B) 업무에 필요한 최소한의 권한만 부여
C) 최소한의 개인정보만 수집
D) 최소한의 보안 조치만 적용

정답: B
해설: 최소 권한 원칙은 각 사용자에게 업무 수행에
      필요한 최소한의 권한만 부여하는 것입니다.
```

### 퀴즈 3: 다음 중 가장 강력한 암호화 알고리즘은?

```
A) MD5
B) SHA-1
C) AES-256
D) DES

정답: C
해설: AES-256이 현재 가장 안전한 암호화 알고리즘입니다.
      MD5와 SHA-1은 이미 깨진 알고리즘이고, DES는 너무 약합니다.
```

### 퀴즈 4: 3-2-1 백업 규칙이란?

```
A) 3시간마다, 2개 서버에, 1개 백업
B) 3개 복사본, 2가지 매체, 1개는 외부 보관
C) 3일마다, 2번 백업, 1주일 보관
D) 3명이, 2시간 동안, 1번 백업

정답: B
해설: 3개의 복사본을 2가지 다른 저장 매체에 보관하되,
      1개는 외부(오프사이트)에 보관하는 규칙입니다.
```

### 퀴즈 5: PIA를 반드시 실시해야 하는 경우는?

```
A) 직원 10명 이상 회사
B) 개인정보 100만 명 이상 보유
C) 매출 100억 이상 회사
D) 설립 1년 이상 회사

정답: B
해설: 개인정보 100만 명 이상을 보유하거나,
      민감정보/고유식별정보 10만 명 이상을
      보유하는 경우 PIA가 의무입니다.
```
