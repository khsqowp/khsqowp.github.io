# 🔑 2. 계정 및 접근 관리 (IAM)_기초: 클라우드의 출입문 관리하기

## 🎯 학습 목표

이 문서를 끝까지 읽고 나면, 여러분은 다음을 할 수 있습니다:
- IAM이 무엇이고 왜 중요한지 이해하기
- 사용자, 그룹, 역할, 정책의 차이점 알기
- 클라우드 계정을 안전하게 관리하는 기본 방법 배우기
- 최소 권한 원칙이 무엇인지 이해하기
- 실제로 IAM 사용자 만들고 권한 부여해보기

## 🤔 IAM이 뭔가요? 왜 필요할까요?

### IAM을 이해하기 위한 실생활 비유

**🏢 회사 출입증 시스템을 생각해보세요:**

```
회사 건물에 들어가려면:
1. 📛 출입증이 필요합니다 (신분 확인)
2. 🚪 특정 층만 갈 수 있습니다 (권한 제한)
3. 📝 출입 기록이 남습니다 (감사 추적)
4. 🚫 퇴사하면 출입증이 회수됩니다 (권한 회수)

예시:
- CEO: 모든 층 출입 가능
- 개발팀: 개발실만 출입 가능
- 인턴: 사무실만 출입 가능
- 손님: 1층 로비만 출입 가능

IAM도 똑같습니다!
클라우드 자원(서버, 데이터 등)에 누가 접근할 수 있는지
정확하게 관리하는 시스템입니다!
```

**🏦 은행 계좌 권한으로 비유하기:**

```
은행 계좌에는 여러 권한이 있습니다:

👨 주계좌 소유자:
✅ 입금 가능
✅ 출금 가능
✅ 계좌 삭제 가능
✅ 다른 사람에게 권한 부여 가능

👩 공동 소유자:
✅ 입금 가능
✅ 출금 가능 (일정 한도까지)
❌ 계좌 삭제 불가
❌ 권한 부여 불가

👶 조회만 가능한 사람:
✅ 잔액 조회 가능
❌ 입금 불가
❌ 출금 불가

이것이 바로 IAM의 "최소 권한 원칙"입니다!
필요한 권한만 딱 부여하는 거예요!
```

## 📚 IAM의 핵심 구성 요소 4가지

IAM은 4가지 주요 구성 요소로 이루어져 있습니다. 학교로 비유해서 이해해봅시다!

### 1. 👤 사용자 (Users) - "학생"

**실생활 비유: 학교의 학생**

```
🎓 학교 학생증

김철수 (학생)
- 학생증 번호: 20240001
- 비밀번호: *****
- 소속: 3학년 1반

학생증으로 할 수 있는 것:
✅ 교실 출입
✅ 도서관 출입
✅ 급식실 출입
❌ 교무실 출입 (선생님만 가능)
❌ 성적 입력 (선생님만 가능)
```

**클라우드로 비유하면:**

```
IAM 사용자: developer-john
- 사용자 ID: developer-john
- 비밀번호: MyP@ssw0rd!2024
- 액세스 키: AKIAIOSFODNN7EXAMPLE

할 수 있는 것:
✅ EC2 서버 조회
✅ S3 파일 읽기
❌ EC2 서버 삭제 (관리자만 가능)
❌ 다른 사용자 추가 (관리자만 가능)
```

**실제 사용 사례:**

```
회사에 신입 개발자 김코딩이 입사했습니다.

1. IAM 사용자 생성:
   - 이름: kim-coding
   - 비밀번호: 첫 로그인 시 변경 필요
   - MFA(2단계 인증): 활성화

2. 권한 부여:
   - 개발 서버 접속 권한
   - 개발용 S3 버킷 읽기/쓰기 권한
   - 운영 서버 접속 불가 ❌

3. 퇴사 시:
   - IAM 사용자 삭제
   - 모든 권한 자동 회수
   - 더 이상 클라우드 접속 불가
```

### 2. 👥 그룹 (Groups) - "반"

**실생활 비유: 학교의 반**

```
🎓 3학년 1반 (그룹)

3학년 1반에 속한 학생들:
- 김철수
- 이영희
- 박민수

3학년 1반 학생들의 공통 권한:
✅ 3학년 교실 출입
✅ 3학년 사물함 사용
✅ 3학년 전용 도서 대출

새로운 학생이 전학 오면:
→ 3학년 1반에 배정하기만 하면
→ 모든 권한이 자동으로 부여됨!
```

**클라우드로 비유하면:**

```
IAM 그룹: Developers (개발자 그룹)

그룹에 속한 사용자들:
- developer-john
- developer-sarah
- developer-mike

Developers 그룹의 공통 권한:
✅ 개발 서버 접속
✅ 개발용 데이터베이스 접근
✅ 개발용 S3 버킷 사용

새로운 개발자가 입사하면:
→ Developers 그룹에 추가하기만 하면
→ 모든 권한이 자동으로 부여됨!
```

**왜 그룹을 사용할까요?**

```
❌ 그룹을 사용하지 않으면:

신입 개발자 10명이 입사
→ 각 사람에게 일일이 권한 부여
→ 권한 1: EC2 접근
→ 권한 2: S3 접근
→ 권한 3: RDS 접근
→ ... (반복 10번!)
→ 시간 낭비! 실수 가능성 높음!

✅ 그룹을 사용하면:

신입 개발자 10명이 입사
→ "Developers" 그룹에 10명 추가
→ 끝! (모든 권한 자동 부여)
→ 시간 절약! 실수 없음!
```

### 3. 👮 역할 (Roles) - "일시적 출입증"

**실생활 비유: 공사 업체 직원**

```
🏗️ 건물 공사 중

공사 업체 직원:
- 우리 회사 직원이 아님
- 하지만 공사를 위해 건물 출입 필요

해결 방법:
1. 일시적 출입증 발급
2. 공사 기간 동안만 유효
3. 공사 끝나면 자동 회수
4. 필요한 구역만 출입 가능

예시:
- 전기 공사: 전기실만 출입 가능
- 청소 용역: 모든 층 출입 가능
- 배달원: 1층 로비만 출입 가능
```

**클라우드로 비유하면:**

```
IAM 역할: EC2-S3-Access

상황:
- EC2 서버에서 실행되는 프로그램이
- S3 저장소의 파일을 읽어야 함

❌ 나쁜 방법:
- 프로그램 코드에 비밀번호 입력
- 보안 위험! 코드 유출되면 큰일!

✅ 좋은 방법 (역할 사용):
- EC2 서버에 "EC2-S3-Access" 역할 부여
- 프로그램이 자동으로 임시 비밀번호 받음
- 임시 비밀번호는 1시간마다 자동 갱신
- 코드에 비밀번호 없어도 됨!
```

**역할의 장점:**

```
1. 보안:
   - 코드에 비밀번호를 넣지 않아도 됨
   - 비밀번호가 자동으로 주기적으로 변경됨

2. 편리함:
   - 비밀번호 관리 불필요
   - 자동으로 권한 부여/회수

3. 유연함:
   - 필요할 때만 권한 부여
   - 작업 끝나면 자동 회수

실제 사례:
📱 모바일 앱 → AWS 서버에 사진 업로드
- 역할 사용: 앱이 임시 권한 받아서 업로드
- 1시간 후: 권한 자동 만료
- 만약 앱이 해킹되어도 피해 최소화!
```

### 4. 📜 정책 (Policies) - "규칙서"

**실생활 비유: 건물 출입 규칙서**

```
📖 건물 출입 규칙서

규칙 1:
- 누구: 전 직원
- 무엇: 1층 로비
- 허용: ✅ 출입 가능

규칙 2:
- 누구: 개발팀
- 무엇: 3층 개발실
- 허용: ✅ 출입 가능

규칙 3:
- 누구: 모든 사람
- 무엇: CEO 사무실
- 거부: ❌ 출입 불가
- 예외: CEO만 출입 가능

규칙 4:
- 누구: 인턴
- 무엇: 모든 층
- 조건: 평일 오전 9시~오후 6시만 출입 가능
```

**클라우드로 비유하면:**

```json
{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Effect": "Allow",
      "Action": "s3:GetObject",
      "Resource": "arn:aws:s3:::my-photos/*"
    }
  ]
}
```

**쉽게 이해하기:**

```
위의 JSON 정책을 한국어로 번역하면:

"허용합니다 (Effect: Allow)
누가: 이 정책을 받은 사람이
무엇을: S3에서 파일 읽기를 (Action: s3:GetObject)
어디서: my-photos 버킷에서 (Resource: s3:::my-photos/*)"

실생활로 비유:
"허용합니다
누가: 김철수가
무엇을: 책 읽기를
어디서: 도서관에서"
```

**정책 예시 - 초보자용:**

```json
{
  "Effect": "Allow",
  "Action": "s3:ListBucket",
  "Resource": "arn:aws:s3:::company-files"
}
```

```
한국어 번역:
"회사 파일 목록 보기를 허용합니다"

비유:
"도서관에 어떤 책이 있는지 목록 보는 것을 허용"
```

```json
{
  "Effect": "Allow",
  "Action": "s3:GetObject",
  "Resource": "arn:aws:s3:::company-files/*"
}
```

```
한국어 번역:
"회사 파일을 다운로드하는 것을 허용합니다"

비유:
"도서관에서 책을 빌리는 것을 허용"
```

```json
{
  "Effect": "Deny",
  "Action": "s3:DeleteObject",
  "Resource": "arn:aws:s3:::company-files/*"
}
```

```
한국어 번역:
"회사 파일을 삭제하는 것을 거부합니다"

비유:
"도서관에서 책을 찢는 것을 금지"
```

## 🔐 IAM의 핵심 원칙: 최소 권한 원칙

### 최소 권한 원칙이란?

**"필요한 것만 딱 주기"**

```
🏠 집 열쇠 비유:

❌ 나쁜 예:
청소 도우미에게 집의 모든 방 열쇠를 줌
- 안방 열쇠
- 서재 열쇠
- 금고 열쇠
→ 필요 없는 곳까지 접근 가능
→ 위험!

✅ 좋은 예:
청소 도우미에게 거실과 화장실 열쇠만 줌
- 거실 열쇠
- 화장실 열쇠
→ 청소에 필요한 곳만 접근
→ 안전!
```

**클라우드에서의 예시:**

```
상황: 신입 개발자가 입사했습니다

❌ 과도한 권한 (위험!):
{
  "Effect": "Allow",
  "Action": "*",              // 모든 작업 허용!
  "Resource": "*"             // 모든 자원에!
}

이렇게 하면:
- 서버 삭제 가능 ❌
- 데이터베이스 삭제 가능 ❌
- 다른 사용자 계정 삭제 가능 ❌
- 회사 망하기 딱 좋음! 💣

✅ 최소 권한 (안전!):
{
  "Effect": "Allow",
  "Action": "s3:GetObject",   // 파일 읽기만
  "Resource": "arn:aws:s3:::dev-bucket/*"  // 개발용 버킷만
}

이렇게 하면:
- 개발에 필요한 파일만 읽을 수 있음 ✅
- 서버 삭제 불가 ✅
- 실수로 인한 피해 최소화 ✅
```

### 실제 회사 시나리오

```
📊 마케팅팀 이영희 씨

업무:
- 고객 데이터 분석
- 보고서 작성

필요한 권한:
✅ 고객 데이터베이스 읽기
✅ 보고서 저장용 S3 버킷 쓰기

불필요한 권한:
❌ 서버 생성/삭제
❌ 데이터베이스 수정
❌ 다른 직원 계정 관리

부여할 정책:
- 고객 DB 읽기 전용
- 보고서 버킷 읽기/쓰기
→ 딱 필요한 것만!
```

## 🛡️ 계정을 안전하게 지키는 방법

### 1. 🔐 2단계 인증 (MFA) 필수!

**2단계 인증을 자물쇠로 비유하기:**

```
🔒 일반 자물쇠 (비밀번호만):
- 비밀번호 하나만 알면 열림
- 비밀번호 유출되면 끝!

🔒🔒 이중 자물쇠 (MFA):
- 비밀번호 + 스마트폰 인증 코드
- 비밀번호 유출되어도
- 스마트폰 없으면 못 열림!
```

**실제 사용 예시:**

```
AWS 로그인 과정:

일반 로그인:
1. 아이디 입력: admin
2. 비밀번호 입력: MyPassword123
3. 끝! 로그인 완료

→ 비밀번호 유출되면 누구나 로그인 가능 ❌

MFA 로그인:
1. 아이디 입력: admin
2. 비밀번호 입력: MyPassword123
3. 스마트폰 앱 실행
4. 6자리 인증 코드 확인: 123456
5. 인증 코드 입력: 123456
6. 로그인 완료!

→ 비밀번호 유출되어도
→ 스마트폰 없으면 로그인 불가 ✅
```

**MFA 설정 방법 (초보자용):**

```
1. 스마트폰에 "Google Authenticator" 앱 설치
   - Android: Play 스토어
   - iPhone: App Store

2. AWS 콘솔 로그인

3. 우측 상단 계정 이름 클릭

4. "보안 자격 증명" 클릭

5. "MFA 디바이스 할당" 클릭

6. "Authenticator 앱" 선택

7. QR 코드가 화면에 나타남

8. 스마트폰 앱으로 QR 코드 스캔

9. 앱에 표시된 6자리 숫자 2번 입력

10. 완료! 이제부터 로그인할 때마다
    스마트폰 인증 코드 필요

💡 팁: 인증 코드는 30초마다 변경됩니다!
```

### 2. 🚫 루트 계정 사용하지 않기

**루트 계정 = 마스터 키**

```
🏢 회사 비유:

루트 계정 (CEO):
- 모든 권한 보유
- 회사 해체 가능
- 직원 전원 해고 가능
- 은행 계좌 삭제 가능

일반 직원:
- 자기 업무에 필요한 권한만
- 회사 해체 불가
- 다른 직원 해고 불가

원칙:
CEO도 평소에는 일반 직원처럼 행동!
꼭 필요할 때만 CEO 권한 사용!
```

**실제 권장 사항:**

```
✅ 올바른 방법:

1. 회사 AWS 계정 생성 (루트 계정 생성됨)

2. 루트 계정에 강력한 비밀번호 설정
   예: MyC0mp@ny#R00t!2024

3. 루트 계정에 MFA 설정

4. 루트 계정 비밀번호를 금고에 보관

5. 관리자용 IAM 사용자 생성
   이름: admin-john

7. 루트 계정은 1년에 1~2번만 사용
   (예: 과금 설정 변경, 계정 폐쇄 등)

❌ 나쁜 방법:
- 루트 계정으로 매일 로그인
- 루트 계정으로 서버 생성/삭제
- 루트 계정 비밀번호를 메모장에 저장
```

## ❓ FAQ: 자주 묻는 질문

**Q1: IAM 사용자와 AWS 계정의 차이는?**
```
A:

AWS 계정 (루트 계정):
- 회사 전체를 대표하는 계정
- 모든 권한 보유
- 결제 정보 연결됨
- 1개만 존재

IAM 사용자:
- 개인 직원을 위한 계정
- 제한된 권한만 보유
- 결제 정보 없음
- 여러 개 생성 가능

비유:
- AWS 계정 = 회사
- IAM 사용자 = 직원
```

**Q2: 그룹과 역할의 차이는?**
```
A:

그룹 (Groups):
- 여러 사람을 묶어놓은 것
- 고정된 멤버십
- 예: 개발팀, 마케팅팀

역할 (Roles):
- 임시로 권한을 빌려주는 것
- 사람, 프로그램 모두 사용 가능
- 예: "S3 업로드 권한" 역할

비유:
- 그룹 = 학교의 반 (3학년 1반)
- 역할 = 당번 (오늘의 청소 당번)
```

**Q3: 정책을 어떻게 만들어야 하나요?**
```
A:

초보자 권장 방법:

1. AWS 제공 정책 사용:
   - "AmazonS3ReadOnlyAccess"
   - "AmazonEC2ReadOnlyAccess"
   → 안전하고 검증됨!

2. 정책 생성기 사용:
   → IAM 콘솔에서 시각적으로 생성
   → JSON 직접 작성 안 해도 됨!

3. 기존 정책 복사해서 수정:
   → AWS 정책을 템플릿으로 사용
   → 필요한 부분만 변경

직접 JSON 작성은 숙련자만!
```

**Q4: 액세스 키가 유출됐어요! 어떻게 하나요?**
```
A:

긴급 대응 절차:

1. 즉시 해당 액세스 키 비활성화
   IAM 콘솔 → 사용자 → 보안 자격 증명
   → 액세스 키 "비활성화" 클릭

2. 새로운 액세스 키 생성

3. 애플리케이션 코드에 새 키 적용

4. 비활성화된 키가 사용 안 되는지 확인
   (며칠 후)

5. 비활성화된 키 삭제

6. CloudTrail 로그 확인
   → 유출된 키로 무엇을 했는지 확인

7. 비정상 활동 발견 시
   → AWS Support에 문의

💡 예방:
- 코드에 키를 직접 입력하지 마세요!
- 역할을 사용하세요!
- GitHub에 업로드하지 마세요!
```

**Q5: MFA를 잃어버렸어요!**
```
A:

1. 백업 MFA 디바이스가 있다면:
   → 백업 디바이스로 로그인

2. 백업이 없다면:
   → AWS Support에 문의
   → 신원 확인 절차 필요
   → 시간이 오래 걸릴 수 있음

💡 예방:
- MFA 설정 시 백업 코드 저장
- 여러 MFA 디바이스 등록
- 복구 연락처 설정
```

```
Q3. 최소 권한 원칙이란?
a) 모든 권한 부여

Q4. MFA는 무엇의 약자?

Q5. 루트 계정은 언제 사용해야 하나요?
```
