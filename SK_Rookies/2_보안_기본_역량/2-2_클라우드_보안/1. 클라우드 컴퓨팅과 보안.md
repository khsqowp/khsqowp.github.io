# ☁️ 1. 클라우드 컴퓨팅과 보안: 새로운 인프라, 새로운 위협

## 🔄 이전 단원 복습: 네트워크 보안 솔루션과 AI 연동

이전 단원 `4. 보안 솔루션.md`에서 우리는 방화벽, IDS/IPS, VPN 등 주요 네트워크 보안 솔루션의 작동 원리와 AI 기반 보안 시스템과의 연동 방안을 학습했습니다. AI가 탐지한 위협에 대해 이러한 솔루션들이 어떻게 실제적인 방어 조치를 취하는지 이해했습니다.

> **핵심 복습:**
>
> *   **보안 솔루션의 역할:** AI의 '지능'과 결합하여 '실행력'을 제공함으로써 지능형 자동 방어 시스템을 구축하는 핵심 요소.
> *   **방화벽:** AI가 탐지한 악성 IP/포트 패턴을 기반으로 규칙을 자동 업데이트하여 네트워크 최전선 방어.
> *   **IDS/IPS:** AI가 미탐/오탐을 줄이고 새로운 공격 패턴에 대한 시그니처를 자동 생성하여 침입 탐지 및 방지 강화.
> *   **VPN:** AI가 비정상 접속 및 사용자 행위 패턴을 탐지하여 원격 접속 보안 강화.
> *   **통합 관제:** AI가 다양한 보안 솔루션의 데이터를 통합 분석하여 위협을 탐지하고 각 솔루션에 연동하여 자동화된 대응을 수행.
>
> 이제 우리는 네트워크 환경에서의 공격과 방어, 그리고 AI를 활용한 보안 솔루션 연동에 대한 깊이 있는 이해를 갖추었습니다. 하지만 현대 IT 인프라의 패러다임은 온프레미스(On-premise)에서 **클라우드(Cloud)**로 빠르게 전환되고 있습니다. 클라우드 환경은 기존 온프레미스 환경과는 근본적으로 다른 특성을 가지며, 이는 새로운 보안 위협과 도전 과제를 야기합니다. 오늘부터 시작하는 `2-2. 클라우드 보안` 파트에서는 이러한 클라우드 환경에서의 보안을 집중적으로 학습합니다.

---

## 🤔 클라우드 컴퓨팅과 보안을 왜 배워야 할까요? (The "Why")

클라우드 컴퓨팅은 IT 자원을 서비스 형태로 제공하여 유연성, 확장성, 비용 효율성을 극대화합니다. 기업들은 더 이상 값비싼 서버를 직접 구매하고 관리할 필요 없이, 필요한 만큼의 컴퓨팅 자원을 빌려 쓰고 사용한 만큼만 비용을 지불합니다. 하지만 이러한 편리함 뒤에는 새로운 보안 위험이 도사리고 있습니다.

**[클라우드 보안 지식의 중요성]**
*   **새로운 공격 표면:** 클라우드 환경은 API, 가상화 기술, 공유 인프라 등 온프레미스와는 다른 공격 표면(Attack Surface)을 가집니다.
*   **책임 공유 모델:** 클라우드 서비스 제공자(CSP, Cloud Service Provider)와 사용자 간의 보안 책임이 명확히 구분되므로, 사용자가 자신의 책임을 정확히 이해하고 이행해야 합니다.
*   **설정 오류의 위험:** 클라우드 서비스의 복잡한 설정은 작은 실수 하나로도 심각한 보안 취약점을 야기할 수 있습니다. (예: S3 버킷 공개 설정)
*   **AI 기반 클라우드 보안:** 클라우드 환경에서 발생하는 방대한 로그와 이벤트 데이터를 AI로 분석하여 위협을 탐지하고, 클라우드 API를 통해 자동화된 대응을 수행하는 것이 필수적입니다.

> **보안 전문가에게 클라우드 컴퓨팅과 보안이란?**
>
> 현대 IT 인프라의 **'새로운 지형'**을 이해하고, 그 위에서 발생하는 **'새로운 위협'**에 대응하는 방법을 배우는 것입니다. 클라우드 환경의 특성을 모른다면, 기존의 보안 지식만으로는 효과적인 방어가 불가능합니다. AI 모델이 클라우드 환경에서 발생하는 위협을 어떻게 탐지하고, 클라우드 서비스의 API를 통해 어떻게 자동화된 방어 조치를 취할 수 있는지 이해하는 **미래 지향적인 보안 역량**입니다.

## 1. ☁️ 클라우드 컴퓨팅의 이해: 서비스 모델과 배포 모델

클라우드 컴퓨팅은 IT 자원을 서비스 형태로 제공하는 방식에 따라 크게 세 가지 서비스 모델로 나뉩니다.

### 1.1. 💡 서비스 모델 (Service Models)

| 모델 | 설명 | 관리 책임 (사용자) | 보안 관점에서의 의미 |
| :--- | :--- | :--- | :--- |
| **IaaS (Infrastructure as a Service)** | 가상 서버, 스토리지, 네트워크 등 인프라 자원 제공 | 운영체제, 미들웨어, 애플리케이션, 데이터 | 가장 많은 보안 책임. OS, 앱, 데이터 보안 직접 관리. |
| **PaaS (Platform as a Service)** | 애플리케이션 개발 및 실행을 위한 플랫폼 제공 | 애플리케이션, 데이터 | IaaS보다 적은 책임. 플랫폼 자체의 보안은 CSP 책임. |
| **SaaS (Software as a Service)** | 완성된 소프트웨어 애플리케이션 제공 | 데이터 (일부 설정) | 가장 적은 보안 책임. 데이터 보안 및 접근 제어에 집중. |

### 1.2. 🌐 배포 모델 (Deployment Models)

*   **퍼블릭 클라우드 (Public Cloud):** AWS, Azure, GCP 등 CSP가 소유하고 운영하는 인프라를 여러 고객이 공유하여 사용. (가장 일반적)
*   **프라이빗 클라우드 (Private Cloud):** 특정 조직만을 위해 구축된 클라우드 인프라. 조직 내부에서 직접 관리하거나 외부 CSP가 관리.
*   **하이브리드 클라우드 (Hybrid Cloud):** 퍼블릭 클라우드와 프라이빗 클라우드를 결합하여 사용하는 형태.
*   **멀티 클라우드 (Multi-Cloud):** 여러 CSP의 퍼블릭 클라우드를 동시에 사용하는 형태.

## 2. 🛡️ 클라우드 보안의 핵심: 책임 공유 모델 (Shared Responsibility Model)

클라우드 환경에서 보안 책임은 CSP와 사용자 간에 공유됩니다. 이 모델을 정확히 이해하는 것이 클라우드 보안의 시작입니다.

```mermaid
graph TD
    subgraph Cloud Service Provider (CSP)
        CSP_Physical[물리적 보안]
        CSP_Infra[인프라 (하드웨어, 네트워크)]
        CSP_Virtual[가상화 계층]
    end

    subgraph Customer (사용자)
        Customer_Data[데이터]
        Customer_App[애플리케이션]
        Customer_OS[운영체제]
        Customer_Network[네트워크 구성 (VPC, Security Group)]
        Customer_IAM[접근 및 계정 관리 (IAM)]
    end

    CSP_Physical --> CSP_Infra;
    CSP_Infra --> CSP_Virtual;

    CSP_Virtual -- "보안 'of' the Cloud" --> Customer_OS;
    CSP_Virtual -- "보안 'of' the Cloud" --> Customer_Network;

    Customer_OS -- "보안 'in' the Cloud" --> Customer_App;
    Customer_App -- "보안 'in' the Cloud" --> Customer_Data;
    Customer_Network -- "보안 'in' the Cloud" --> Customer_IAM;
```

| 책임 주체 | 책임 범위 | 보안 관점에서의 의미 |
| :--- | :--- | :--- |
| **CSP** | **클라우드 '자체'의 보안 (Security *of* the Cloud)** <br> 물리적 인프라, 호스트 OS, 가상화 계층, 네트워크 인프라 | CSP는 클라우드 서비스의 기반이 되는 인프라의 보안을 책임집니다. (예: 데이터센터 물리적 보안, 하이퍼바이저 보안) |
| **사용자** | **클라우드 '내'의 보안 (Security *in* the Cloud)** <br> 데이터, 애플리케이션, 운영체제, 네트워크 구성, 접근 및 계정 관리 | 사용자는 클라우드 위에 배포하는 자신의 자원(데이터, 앱, OS)과 설정(네트워크, IAM)의 보안을 책임집니다. **대부분의 클라우드 보안 사고는 사용자 책임 영역에서 발생합니다.** |

> **핵심 경고 ⚠️:**
>
> **"클라우드에 올리면 자동으로 안전해진다"는 오해는 금물입니다.** CSP는 클라우드 인프라의 보안을 책임지지만, 그 위에 무엇을 어떻게 올리고 설정하는지는 전적으로 사용자의 책임입니다. 잘못된 설정(Misconfiguration)은 클라우드 보안 사고의 가장 큰 원인입니다.

## 3. 🛡️ 클라우드 환경의 주요 보안 위협

클라우드 환경은 기존 온프레미스와는 다른 새로운 보안 위협에 노출됩니다.

### 3.1. 💥 설정 오류 (Misconfiguration)

클라우드 보안 사고의 90% 이상이 설정 오류에서 발생합니다. S3 버킷 공개, 보안 그룹(Security Group)의 과도한 포트 개방, IAM 권한 과부여 등이 대표적입니다.

#### **AI 기반 탐지 및 방어 전략**

| 구분 | AI 연동 방안 |
| :--- | :--- |
| **탐지** | AI가 클라우드 설정(IAM 정책, 보안 그룹 규칙, S3 버킷 정책 등)을 주기적으로 스캔하고, 알려진 취약한 설정 패턴이나 비정상적인 변경 사항을 탐지. <br> (예: S3 버킷이 Public으로 변경되거나, 모든 IP에 대해 SSH 포트가 열리는 경우) |
| **자동 수정/경고** | AI가 탐지한 설정 오류에 대해 자동으로 안전한 설정으로 되돌리거나, 관리자에게 즉시 경고 알림. |
| **컴플라이언스 검사** | AI가 클라우드 리소스가 GDPR, HIPAA 등 규제 준수 여부를 자동으로 검사. |

### 3.2. 💥 계정 탈취 (Account Compromise)

클라우드 계정은 모든 클라우드 자원에 대한 접근 권한을 가지므로, 계정 탈취는 가장 치명적인 위협입니다. 약한 비밀번호, 피싱, API 키 유출 등이 원인입니다.

#### **AI 기반 탐지 및 방어 전략**

| 구분 | AI 연동 방안 |
| :--- | :--- |
| **비정상 로그인 탐지** | AI가 클라우드 로그인 로그(시간, 출발지 IP, 지역, 로그인 성공/실패 횟수)를 분석하여 평소와 다른 비정상적인 로그인 시도(예: 다른 지역에서의 동시 로그인, 짧은 시간 내 대량 로그인 실패)를 탐지. |
| **권한 상승 탐지** | AI가 IAM 역할 변경, 새로운 사용자 생성 등 비정상적인 권한 상승 시도를 탐지. |
| **API 키 유출 탐지** | AI가 GitHub 등 공개 저장소를 모니터링하여 유출된 클라우드 API 키를 탐지. |

### 3.3. 💥 안전하지 않은 API (Insecure APIs)

클라우드 서비스는 대부분 API를 통해 관리됩니다. API의 취약점은 클라우드 자원에 대한 무단 접근을 허용할 수 있습니다.

#### **AI 기반 탐지 및 방어 전략**

| 구분 | AI 연동 방안 |
| :--- | :--- |
| **API 호출 패턴 분석** | AI가 클라우드 API 호출 로그를 분석하여 비정상적인 호출 패턴(예: 특정 API의 비정상적인 호출 빈도, 평소 사용하지 않던 API 호출)을 탐지. |
| **API 취약점 스캐닝** | AI 기반 자동화된 도구가 클라우드 API의 취약점을 스캔하고 보고. |

## 👨‍💻 현직자 통합 시나리오: AI 기반 클라우드 보안 형상 관리 (CSPM) 시스템

**[상황]**
AI 보안 엔지니어 '제미니'는 회사의 AWS 클라우드 환경에서 발생하는 설정 오류와 계정 탈취 위협을 자동으로 탐지하고 대응하는 클라우드 보안 형상 관리(CSPM, Cloud Security Posture Management) 시스템을 구축하고자 합니다.

**[워크플로우]**
1.  **클라우드 설정 데이터 수집:** AWS Config, CloudTrail 로그, IAM 정책, Security Group 규칙 등 클라우드 설정 및 활동 로그를 수집합니다. (AWS API)
2.  **AI 기반 설정 오류 탐지:**
    *   AI가 S3 버킷 정책을 분석하여 Public으로 설정된 버킷을 탐지합니다.
    *   AI가 Security Group 규칙을 분석하여 모든 IP(`0.0.0.0/0`)에 대해 SSH(22) 또는 RDP(3389) 포트가 열려있는 경우를 탐지합니다.
3.  **AI 기반 계정 탈취 탐지:**
    *   AI가 CloudTrail 로그인 로그를 분석하여 평소와 다른 지역에서 로그인 시도(비정상 로그인)를 탐지합니다.
    *   AI가 IAM 사용자 활동 로그를 분석하여 비정상적인 권한 변경(예: AdministratorAccess 정책 추가)을 탐지합니다.
4.  **자동 대응 및 알림:**
    *   탐지된 설정 오류에 대해 자동으로 안전한 설정으로 되돌리거나, 관리자에게 즉시 경고 알림을 보냅니다.

    return [
        {'group_id': 'sg-001', 'port': 80, 'source': '0.0.0.0/0'},
        {'group_id': 'sg-002', 'port': 443, 'source': '0.0.0.0/0'},
        {'group_id': 'sg-003', 'port': 22, 'source': '0.0.0.0/0'}, # ⚠️ 설정 오류
        {'group_id': 'sg-004', 'port': 3389, 'source': '10.0.0.0/8'}
    ]

def get_cloudtrail_login_logs():
    """가상 CloudTrail 로그인 로그 데이터 반환"""
    return pd.DataFrame({
        'timestamp': pd.to_datetime(['2023-11-12 10:00', '2023-11-12 10:05', '2023-11-12 10:10', '2023-11-12 10:11', '2023-11-12 10:12']),
        'user_id': ['user1', 'user1', 'user2', 'user1', 'user1'],
        'source_ip': ['1.1.1.1', '1.1.1.1', '2.2.2.2', '5.5.5.5', '1.1.1.1'], # 5.5.5.5는 비정상 IP 가정
        'region': ['ap-northeast-2', 'ap-northeast-2', 'us-east-1', 'eu-west-1', 'ap-northeast-2'], # eu-west-1은 비정상 지역 가정
        'login_success': [True, True, True, True, True]
    })

# --- 2. AI 기반 설정 오류 탐지 ---
print("--- ☁️ AI 기반 클라우드 설정 오류 탐지 ---")
s3_policies = get_s3_bucket_policies()
for bucket in s3_policies:
    if bucket['policy'] == 'public':
        print(f"  - 🚨 경고: S3 버킷 '{bucket['bucket_name']}'이 Public으로 설정되어 있습니다! (설정 오류)")
        # 💡 자동 수정: boto3를 사용하여 버킷 정책을 private으로 변경하는 코드
        # s3_client.put_bucket_policy(Bucket=bucket['bucket_name'], Policy=json.dumps(private_policy))

sg_rules = get_security_group_rules()
for rule in sg_rules:
    if rule['source'] == '0.0.0.0/0' and rule['port'] in [22, 3389]:
        print(f"  - 🚨 경고: 보안 그룹 '{rule['group_id']}'에 모든 IP에 대한 {rule['port']}번 포트가 열려 있습니다! (설정 오류)")
        # 💡 자동 수정: boto3를 사용하여 해당 규칙을 삭제하거나 특정 IP로 제한하는 코드

# --- 3. AI 기반 계정 탈취 탐지 ---
print("\n--- ☁️ AI 기반 클라우드 계정 탈취 탐지 ---")
login_logs_df = get_cloudtrail_login_logs()

# 비정상 로그인 탐지 (IsolationForest 활용)
# IP와 지역을 숫자로 인코딩
login_logs_df['source_ip_encoded'] = pd.factorize(login_logs_df['source_ip'])[0]
login_logs_df['region_encoded'] = pd.factorize(login_logs_df['region'])[0]

# 특징 선택
features = login_logs_df[['source_ip_encoded', 'region_encoded']]

# IsolationForest 모델 훈련 및 예측
# contamination은 이상치 비율을 나타내며, 실제 환경에서는 도메인 지식으로 설정
iso_forest = IsolationForest(random_state=42, contamination=0.1) 
login_logs_df['anomaly'] = iso_forest.fit_predict(features)

# 이상치(-1)로 분류된 로그 확인
anomalous_logins = login_logs_df[login_logs_df['anomaly'] == -1]

if not anomalous_logins.empty:
    for index, row in anomalous_logins.iterrows():
        print(f"  - 🚨 경고: 비정상 로그인 탐지! 사용자 '{row['user_id']}' (IP: {row['source_ip']}, 지역: {row['region']})")
        # 💡 자동 대응: boto3를 사용하여 해당 IAM 계정을 일시 잠그거나 MFA 강제
        # iam_client.update_user(UserName=row['user_id'], Status='Inactive')

    print("  - 비정상 로그인 활동이 탐지되지 않았습니다.")

## 📌 요약 정리 (Executive Summary)

1.  **클라우드 보안의 중요성**: 클라우드 환경은 새로운 공격 표면과 책임 공유 모델을 가지므로, 이에 특화된 보안 지식이 필수적이다. 잘못된 설정(Misconfiguration)은 클라우드 보안 사고의 가장 큰 원인이다.
2.  **클라우드 서비스 모델**:
    *   **IaaS**: 인프라 제공, 가장 많은 사용자 책임.
    *   **PaaS**: 플랫폼 제공, 중간 수준의 사용자 책임.
    *   **SaaS**: 소프트웨어 제공, 가장 적은 사용자 책임.
3.  **책임 공유 모델**: CSP는 '클라우드 자체'의 보안을, 사용자는 '클라우드 내'의 보안을 책임진다. 사용자의 책임 영역에서 대부분의 보안 사고가 발생한다.
4.  **클라우드 환경의 주요 보안 위협**:
    *   **설정 오류 (Misconfiguration)**: S3 버킷 공개, 보안 그룹 과도한 개방 등. AI는 설정 스캔 및 비정상 변경 탐지로 대응.
    *   **계정 탈취 (Account Compromise)**: 약한 비밀번호, 피싱, API 키 유출 등. AI는 비정상 로그인, 권한 상승, API 키 유출 탐지로 대응.
    *   **안전하지 않은 API (Insecure APIs)**: 클라우드 관리 API의 취약점. AI는 API 호출 패턴 분석으로 대응.
5.  **AI 기반 클라우드 보안 형상 관리 (CSPM)**: AI가 클라우드 설정 및 활동 로그를 분석하여 설정 오류와 계정 탈취 위협을 탐지하고, 자동으로 안전한 설정으로 되돌리거나 계정을 잠그는 등의 대응 조치를 수행한다.
6.  **보안 관점**: 클라우드 환경에서는 AI가 방대한 로그와 이벤트를 분석하여 위협을 탐지하고, 클라우드 API를 통해 자동화된 대응을 수행하는 것이 필수적이다. 클라우드 서비스의 복잡한 설정을 정확히 이해하고, AI를 활용하여 지속적으로 보안 상태를 관리하는 것이 핵심이다.
