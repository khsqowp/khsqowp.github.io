## 2. 우리 조 아이디어 브레인스토밍

공지사항을 숙지한 후, 저희 조는 본격적으로 프로젝트 아이디어에 대한 논의를 시작했습니다.

### 2.1. 1차 아이디어: "챗봇(Chatbot) 만들어보기"

가장 먼저 나온 아이디어는 '챗봇'이었습니다. 최근 기술 트렌드와 가깝고, 사용자와 직접 상호작용하는 결과물을 만들 수 있다는 점에서 매력적인 주제였습니다.

-   **적용 분야 논의**
    -   "보통 챗봇을 많이 쓰는 곳이 쇼핑몰이나 여행/숙박업소 같은 곳이니까, 그런 도메인을 타겟으로 해보면 어떨까요?"
    -   "평소에 앱에서 사용하는 챗봇들은 대부분 미리 정해진 질문을 클릭하면 정해진 답변이 나오는 매크로 방식이 많은 것 같아요."
    -   "결국 우리가 만들 챗봇이라는 게 ChatGPT와 어떻게 다른 걸까요? 구조를 잘 모르겠네요."

> **📌 노트:** 챗봇이라는 아이디어는 좋았지만, 대부분의 팀원들이 챗봇의 구체적인 작동 방식이나 기존 서비스와의 차별점을 만드는 방법에 대해 막연하게 느끼고 있었습니다. 특히, 단순한 매크로 방식이 아닌, 의미 있는 답변을 생성하는 챗봇을 어떻게 구현할지에 대한 기술적 이해가 부족한 상태였습니다.

### 2.2. 핵심 기술 논의: `LangChain`과 데이터셋의 중요성

챗봇 아이디어는 자연스럽게 다음 주에 배울 `LangChain`에 대한 논의로 이어졌습니다.

-   "랭체인을 써보신 분 계신가요? 저도 써본 적이 없어서 뭘 준비해야 할지 잘 모르겠어요."
-   "수업을 들어봐야 확실해지겠지만, 제가 찾아본 바로는 우리가 직접 LLM을 만드는 게 아니라, 기존 LLM API를 가져와서 파이썬으로 응답을 제어하는 느낌인 것 같아요."
-   "결국 우리가 할 일은 **(1)데이터를 뽑아서** **(2)분석하고** **(3)시각화**하는 것까지가 핵심 아닐까요? 랭체인은 이 결과를 보여주는 하나의 방식일 것 같아요."

> **💡 중요!:** 논의를 통해, `LangChain` 자체에 매몰되기보다는 **'어떤 데이터를 가지고, 어떤 목적의 분석을 할 것인가'**를 정하는 것이 프로젝트의 성패를 가르는 가장 중요한 첫걸음이라는 점에 모두가 공감했습니다.

-   **데이터셋 선정 기준 논의**
    -   "데이터가 제일 많은 쪽으로 가는 게 구현하기 편할 것 같아요."
    -   "맞아요. 그리고 `Object` 타입(문자열) 데이터보다는 **수치화하기 편한 숫자 데이터**가 분석이나 시각화에 더 유리할 것 같습니다."

#### 🧠 '랭체인(LangChain)' 개념 상세 분석

회의 중 '랭체인'에 대한 불확실성이 계속 언급되었습니다. 강사님의 이전 설명과 팀원들의 논의를 바탕으로 랭체인의 개념을 단계별로 정리하면 다음과 같습니다.

1.  **1단계 (기존 LLM의 한계):** ChatGPT와 같은 거대 언어 모델(LLM)은 학습된 특정 시점까지의 데이터만 알고 있습니다. 따라서 최신 정보에 대한 질문에 답할 수 없으며, 우리 팀의 내부 문서나 특정 도메인의 전문 지식(예: 특정 게임 공략법)에 대해서는 전혀 알지 못합니다.
2.  **2단계 (RAG - 검색 증강 생성):** 이 한계를 극복하기 위해, LLM이 답변을 생성하기 전에 **외부 데이터 소스(우리가 가진 DB, 문서, 웹사이트 등)에서 질문과 관련된 정보를 실시간으로 '검색(Retrieval)'**해오는 과정이 추가됩니다. 이 기술을 **RAG(Retrieval-Augmented Generation)**라고 합니다.
3.  **3단계 (LangChain의 역할 - 파이프라인 구축):** '랭체인'은 바로 이 과정, 즉 **[①사용자 질문 접수 -> ②외부 데이터 소스 검색 -> ③검색된 정보와 원래 질문을 조합하여 LLM에 전달 -> ④LLM이 정보를 참고하여 최종 답변 생성]**으로 이어지는 전체적인 '사슬(Chain)' 또는 파이프라인을 개발자가 쉽게 구축하고 관리할 수 있도록 도와주는 **개발 프레임워크**입니다.
4.  **최종 결과:** 따라서 우리 조는 LLM을 직접 만들거나 파인튜닝하는 것이 아니라, `LangChain`을 이용해 우리가 선정한 외부 데이터(예: 수집한 게임 데이터, 분석한 로그 데이터 등)를 OpenAI의 GPT 모델과 같은 기존 LLM과 '연결'하여, **우리의 데이터를 기반으로 질문에 답변하는** 특화된 애플리케이션을 만드는 것을 목표로 할 수 있습니다.

### 2.3. 프로젝트 주제 구체화: 보안 vs 비보안

FT님의 공지를 통해 주제가 보안에 국한될 필요가 없다는 점이 명확해지면서, 다양한 아이디어가 논의되었습니다.

> **FT님 공지 요약:** "보안 자동화가 주제가 아닙니다. 이번에 배운 파이썬과 데이터 분석, 그리고 LLM 기술을 최대한 활용하여 **데이터 분석 및 시각화를 통한 인사이트 도출**이 주제입니다. 보안 관련 주제를 찾으셔도 좋고, 화장품 트렌드 분석처럼 완전히 다른 주제를 하셔도 좋습니다."

#### 🏛️ FT님이 제시한 프로젝트 예시 구조 분석

FT님께서 몇 가지 구체적인 예시를 들어주셨고, 이를 통해 프로젝트의 방향성을 더 명확히 할 수 있었습니다.

1.  **예시 1: 파일 확장자를 이용한 악성 파일 탐지 분석**
    *   **데이터:** 악성 파일 샘플들의 메타데이터 (파일 크기, 확장자, 생성일 등).
    *   **분석:** 정상 파일과 악성 파일의 확장자 분포, 특정 확장자(예: `.exe`를 `.txt`로 위장) 변경 패턴 등을 통계적으로 분석.
    *   **시각화:** 확장자별 악성/정상 파일 개수를 비교하는 막대 차트, 시간에 따른 특정 악성 확장자 유행 추이를 보여주는 선 그래프 등.
    *   **인사이트:** '최근에는 `.pdf`로 위장한 악성코드가 급증하고 있으므로, 이메일 필터링 시 `.pdf` 파일에 대한 검사를 강화해야 한다'와 같은 결론 도출.

2.  **예시 2: 웹 로그 분석을 통한 공격자 패턴 분석**
    *   **데이터:** 웹 서버의 접근 로그(Access Log). (누가, 언제, 무엇을 요청했는지 기록된 텍스트 파일)
    *   **분석:** 특정 IP가 짧은 시간 동안 여러 페이지에 대해 SQL Injection 공격 패턴(`' OR 1=1--`)을 시도하는 로그를 필터링. 공격 시도 횟수가 많은 IP Top 10을 집계.
    *   **시각화:** 시간대별 공격 시도 횟수를 보여주는 히스토그램, 공격 유형별 비율을 보여주는 파이 차트, 공격 IP의 국가 분포를 보여주는 지도 시각화(`Folium` 활용).
    *   **인사이트:** '특정 국가(예: 중국) 대역의 IP에서 새벽 3-4시 사이에 SQL Injection 공격 시도가 집중적으로 발생하고 있다'는 패턴 발견.

> **📌 노트:** 위 예시들을 통해, 우리 프로젝트가 **[데이터 수집/선정 -> 분석을 통한 패턴/이상 탐지 -> 시각화를 통한 근거 제시 -> 실행 가능한 조치/전략(인사이트) 도출]**의 흐름을 가져야 한다는 점을 명확히 이해할 수 있었습니다.

## 3. 다른 조 염탐 및 아이디어 공유

우리 조의 논의가 어느 정도 진행된 후, 다른 조는 어떤 아이디어를 가지고 있는지 알아보기 위해 잠시 다른 소회의실에 방문하여 의견을 나누었습니다.

-   **8조의 아이디어:** 우리 조와 비슷하게, 지도 위에 데이터를 시각화하는 아이디어를 생각하고 있었습니다. 구체적으로는 **'히트맵(Heatmap)'**을 만들어 특정 지역의 데이터 밀집도를 표현하는 방식을 논의하고 있었습니다.
-   **팀장들의 만남:** 다른 조의 팀장님들도 잠시 우리 회의실에 방문하여 서로의 진행 상황을 공유했습니다. 대부분의 조가 아직 구체적인 주제를 확정하지는 못했지만, `LangChain`의 활용 가능성과 적절한 데이터셋을 찾는 데 어려움을 겪고 있다는 공통적인 고민을 가지고 있었습니다.

> **📌 노트:** 다른 조와의 교류를 통해, 우리 조가 겪는 어려움이 보편적인 것임을 확인하며 안도감을 느낌과 동시에, 먼저 구체적인 데이터셋을 확보하고 분석 방향을 빠르게 정하는 팀이 프로젝트를 주도해나갈 수 있겠다는 위기감도 함께 느꼈습니다.
