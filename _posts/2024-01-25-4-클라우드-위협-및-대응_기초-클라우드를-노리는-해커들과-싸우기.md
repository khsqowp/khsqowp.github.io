## 🤔 클라우드 위협이 뭔가요? 왜 위험할까요?

### 클라우드 위협을 이해하기 위한 실생활 비유

**🏦 은행을 생각해보세요:**

```
일반 은행 (온프레미스):
🏢 우리 동네 은행
- 직접 방문해서 돈 입출금
- 은행 직원이 직접 관리
- 강도가 침입하려면 물리적으로 와야 함
- 지역 한정

인터넷 뱅킹 (클라우드):
💻 온라인 은행
- 인터넷으로 어디서든 접속
- 자동화 시스템으로 관리
- 전 세계 어디서든 해킹 시도 가능 ⚠️
- 24시간 공격 노출

새로운 위협:
❌ 피싱: 가짜 은행 웹사이트
❌ 비밀번호 탈취: 자동 대입 공격
❌ 앱 취약점: 뱅킹 앱 버그
❌ API 해킹: 시스템 간 통신 공격
```

**클라우드의 위험이 큰 이유:**

```
1. 전 세계에 노출
   🌍 인터넷만 있으면 누구나 접근 시도 가능
   → 러시아, 중국, 북한 등 전 세계 해커들

2. 24시간 공격 대상
   ⏰ 해커는 잠을 자지 않음
   → 자동화 도구로 지속적 공격

3. 가치 있는 목표
   💎 클라우드에는 중요한 데이터 가득
   → 고객 정보, 금융 데이터, 회사 기밀

4. 복잡한 시스템
   🔧 설정이 복잡함
   → 작은 실수 하나로 큰 사고

5. 빠른 변화
   🚀 서비스가 빠르게 변화
   → 보안 업데이트 놓치기 쉬움
```

## 💥 클라우드에서 발생하는 주요 위협 5가지

### 1. 🔓 데이터 유출 - "금고 문이 열려있어요!"

**실생활 비유: 은행 금고 문을 열어둔 상태**

```
상황:
🏦 은행 금고 문이 활짝 열림
💰 누구나 들어가서 돈 가져갈 수 있음
📸 CCTV도 꺼져 있음

결과:
→ 전 고객의 돈이 사라짐
→ 은행 파산
→ 은행장 구속

클라우드에서도 똑같습니다!
```

**실제 사고 사례:**

**사례 1: S3 버킷 공개 사고 (2019년)**
```
회사: Capital One (미국 대형 은행)

문제:
- AWS S3 버킷을 Public으로 설정
- 1억 명의 고객 정보 저장
- 누구나 다운로드 가능한 상태!

피해:
→ 1억 명의 신용카드 정보 유출
→ 주민등록번호, 계좌번호 유출
→ 수천억 원 벌금
→ 주가 폭락

원인:
❌ S3 버킷 설정 실수 (딱 한 번의 클릭 실수!)

교훈:
✅ S3 버킷은 기본적으로 Private으로!
✅ 정기적으로 설정 점검!
```

**사례 2: 데이터베이스 노출 (2020년)**
```
회사: 여러 중소기업

문제:
- MongoDB 데이터베이스를 인터넷에 직접 노출
- 비밀번호 없이 접속 가능
- 검색 엔진에 색인됨

피해:
→ 2억 건 이상의 개인정보 유출
→ 회사 폐업
→ 집단 소송

원인:
❌ Public 서브넷에 DB 배치
❌ 방화벽 미설정
❌ 기본 비밀번호 사용

교훈:
✅ 데이터베이스는 Private 서브넷에!
✅ 강력한 비밀번호 필수!
✅ 방화벽으로 IP 제한!
```

**데이터 유출을 막는 방법:**

```
1. 올바른 서브넷 배치
   ✅ 웹 서버 → Public 서브넷
   ✅ 데이터베이스 → Private 서브넷
   ✅ 백업 서버 → Private 서브넷

2. 접근 권한 최소화
   ✅ 필요한 사람에게만 권한 부여
   ✅ "모든 권한" 절대 금지
   ✅ 정기적으로 권한 검토

3. 데이터 암호화
   ✅ 저장된 데이터 암호화
   ✅ 전송 중인 데이터 암호화 (HTTPS)
   ✅ 백업도 암호화

4. 정기 점검
   ✅ 매주 S3 버킷 설정 확인
   ✅ 공개된 자원 없는지 검색
   ✅ 로그 분석

5. 경보 시스템
   ✅ 비정상 접근 시 알림
   ✅ 대량 다운로드 감지
   ✅ 설정 변경 시 알림
```

### 2. ⚙️ 설정 실수 - "문단속을 깜빡했어요!"

**실생활 비유: 집을 나서면서 문 안 잠그기**

```
아침에 출근하면서:
❌ 대문 안 잠금
❌ 창문 열어둠
❌ 보안 시스템 안 켬

저녁에 귀가:
→ 집이 털림!
→ "문단속만 잘했어도..."

클라우드도 똑같습니다!
작은 설정 실수 하나가 큰 사고로!
```

**흔한 설정 실수 TOP 5:**

**1위: 모든 IP에 SSH 포트 개방**
```
❌ 잘못된 설정:

보안 그룹:
- 포트: 22 (SSH)
- 출발지: 0.0.0.0/0 (전 세계 모든 IP!)

결과:
→ 중국, 러시아 등에서 24시간 로그인 시도
→ 하루에 수천 번의 공격
→ 언젠가는 뚫림!

실제 로그:
2024-11-22 01:23:45 Failed login from 45.123.45.67 (중국)
2024-11-22 01:23:47 Failed login from 45.123.45.67 (중국)
2024-11-22 01:23:49 Failed login from 91.234.56.78 (러시아)
... (수천 번 반복)

✅ 올바른 설정:

보안 그룹:
- 포트: 22 (SSH)
- 출발지: 203.0.113.5/32 (내 회사 IP만!)

결과:
→ 회사에서만 접속 가능
→ 해커 차단!
```

**2위: S3 버킷을 Public으로 설정**
```
❌ 위험한 설정:

버킷: company-secrets
권한: Public (공개)
내용: 고객 정보, 회사 기밀

→ 구글에서 검색 가능!
→ 누구나 다운로드 가능!

✅ 안전한 설정:

버킷: company-secrets
권한: Private (비공개)
암호화: 활성화
접근: IAM 역할만 허용
```

**3위: 루트 계정 사용**
```
❌ 나쁜 습관:

매일 루트 계정으로 로그인
→ 모든 권한 보유
→ 실수로 뭐든지 삭제 가능
→ 탈취당하면 회사 망함

✅ 좋은 습관:

루트 계정:
- 금고에 비밀번호 보관
- 1년에 1~2번만 사용
- MFA 필수

일상 업무:
- IAM 사용자 계정 사용
- 필요한 권한만 부여
- 안전!
```

**4위: MFA 미설정**
```
❌ 위험:

계정: admin
비밀번호: 12345678
MFA: 없음

→ 비밀번호 유출되면 끝!

✅ 안전:

계정: admin
비밀번호: MyS3cur3P@ssw0rd!2024
MFA: 구글 인증 앱 사용

→ 비밀번호 유출되어도
→ 스마트폰 없으면 로그인 불가!
```

**5위: 로그 미활성화**
```
❌ 위험:

CloudTrail: 꺼짐
VPC Flow Logs: 꺼짐

→ 해커가 들어와도 모름
→ 언제 들어왔는지도 모름
→ 무엇을 했는지도 모름

✅ 안전:

CloudTrail: 켜짐
VPC Flow Logs: 켜짐
로그 보관: 90일

→ 모든 활동 기록
→ 비정상 활동 탐지 가능
→ 사고 발생 시 조사 가능
```

### 3. 🔑 계정 탈취 - "도둑이 내 열쇠를 훔쳤어요!"

**실생활 비유: 집 열쇠를 도둑이 훔침**

```
상황:
🔑 집 열쇠를 길에서 분실
😈 도둑이 주움
🏠 도둑이 내 집에 마음대로 출입

결과:
→ 물건 도난
→ 더 심하면 열쇠 복제해서 계속 들어옴
→ 내가 모르는 사이에!
```

**계정 탈취의 흔한 경로:**

**1. 피싱 이메일**
```
📧 가짜 이메일 받음:

발신자: aws-security@gmail.com (가짜!)
제목: [긴급] AWS 계정이 곧 정지됩니다!

내용:
"귀하의 계정에 의심스러운 활동이 감지되었습니다.
24시간 내에 확인하지 않으면 계정이 정지됩니다.
👉 여기를 클릭하세요: aws-login.com (가짜 사이트!)

❌ 클릭하면:
→ 가짜 로그인 페이지로 이동
→ 비밀번호 입력
→ 해커가 비밀번호 획득!
→ 계정 탈취 완료!

✅ 올바른 대응:
1. 링크 클릭하지 않기
2. AWS 공식 사이트에 직접 접속
3. 실제로 문제가 있는지 확인
4. 의심스러운 메일 삭제
```

**2. 비밀번호 유출**
```
해커의 공격 방법:

1단계: 다른 사이트 해킹
→ 작은 쇼핑몰 사이트 해킹
→ 회원 정보 (이메일, 비밀번호) 획득

2단계: 같은 비밀번호 시도
→ 많은 사람이 여러 사이트에 같은 비밀번호 사용
→ 획득한 이메일과 비밀번호로 AWS 로그인 시도

3단계: 로그인 성공!
→ AWS 계정 탈취
→ 서버 삭제하거나 암호화폐 채굴

방어 방법:
✅ 사이트마다 다른 비밀번호 사용
✅ 비밀번호 관리자 앱 사용
✅ MFA 필수!
```

**3. 코드에 비밀번호 입력**
```
❌ 절대 금지:

// config.py
AWS_KEY = "AKIAIOSFODNN7EXAMPLE"
AWS_SECRET = "wJalrXUtnFEMI/K7MDENG"

→ GitHub에 업로드
→ 전 세계에 공개!
→ 해커의 자동 스캔 도구가 발견
→ 15분 만에 탈취!

실제 타임라인:
10:00 AM - 코드를 GitHub에 업로드
10:15 AM - 해커의 봇이 API 키 발견
10:20 AM - 해커가 암호화폐 채굴 시작
1 달 후 - 수천만 원의 AWS 청구서 도착 💣

✅ 올바른 방법:

# .env 파일 (Git에 올리지 말 것!)
AWS_KEY=your_key_here
AWS_SECRET=your_secret_here

# .gitignore에 추가
.env

# 또는 IAM 역할 사용 (더 안전!)
```

**계정 탈취를 막는 방법:**

```
1. 강력한 비밀번호
   ✅ 12자 이상
   ✅ 대소문자 + 숫자 + 특수문자
   ✅ 사이트마다 다른 비밀번호

2. MFA (2단계 인증)
   ✅ 스마트폰 인증 앱
   ✅ 비밀번호 + 인증 코드 둘 다 필요
   ✅ 비밀번호 유출되어도 안전

3. 피싱 주의
   ✅ 이메일 링크 클릭하지 않기
   ✅ 공식 사이트에 직접 접속
   ✅ URL 주소 꼼꼼히 확인

4. 정기적인 비밀번호 변경
   ✅ 3개월마다 변경
   ✅ 이전 비밀번호와 완전히 다르게

5. 로그 모니터링
   ✅ 비정상 로그인 감지
   ✅ 다른 국가에서 접속 시 알림
   ✅ 여러 번 로그인 실패 시 계정 잠금
```

### 4. 🌐 안전하지 않은 API - "뒷문이 열려있어요!"

**실생활 비유: 건물의 비상구 문단속**

```
건물 정문:
🚪 경비원 상주
🔐 출입증 확인
📹 CCTV 감시

건물 뒷문 (API):
🚪 경비원 없음 ❌
🔓 누구나 출입 가능 ❌
📹 CCTV 없음 ❌

→ 도둑이 뒷문으로 침입!
```

**API 관련 위협:**

**1. API 키 유출**
```
개발자의 실수:

# 코드에 API 키 입력
api_key = "sk-1234567890abcdef"

# GitHub에 업로드
# Public 저장소로 설정

결과:
→ 해커가 자동 스캔으로 발견
→ API 키 도용
→ 여러분의 클라우드 자원으로:
  - 암호화폐 채굴
  - 스팸 발송
  - DDoS 공격
→ 엄청난 요금 폭탄!

실제 피해 사례:
- 한 학생이 API 키를 GitHub에 올림
- 24시간 만에 $5,000 (약 700만 원) 청구
- 암호화폐 채굴에 악용됨

방어:
✅ API 키를 코드에 넣지 말 것
✅ 환경 변수 사용
✅ .gitignore에 .env 추가
✅ GitHub 스캔 도구 사용
```

**2. API 남용**
```
공격 시나리오:

해커가 API를 1초에 1,000번 호출
→ 서버 과부하
→ 정상 사용자 접속 불가
→ 서비스 다운

방어:
✅ Rate Limiting 설정
  - 1분에 최대 60번 호출
  - 초과 시 차단

✅ API 키별 사용량 제한
  - 하루 10,000번 제한

✅ 비정상 패턴 감지
  - 갑자기 호출량 급증 시 알림
```

### 5. 🦠 공급망 공격 - "배달 음식에 독이!"

**실생활 비유: 배달 음식 중독**

```
상황:
🍕 피자 주문
📦 배달 도중 누군가 독 투입
🏠 집에서 받아서 먹음
💀 중독!

→ 내가 잘못한 게 없는데
→ 중간 과정에서 문제 발생
```

**클라우드 공급망 공격:**

**1. 취약한 라이브러리**
```
개발 과정:

1. npm install express
   → Express 라이브러리 설치

2. Express가 다른 라이브러리에 의존
   → express → body-parser → qs → ...

3. 깊은 의존성 중 하나가 취약
   → qs 라이브러리에 보안 취약점!

4. 해커가 취약점 악용
   → 내 서버 해킹!

실제 사례: Log4Shell (2021년)
- Log4j 라이브러리에 심각한 취약점
- 전 세계 수백만 서버 영향
- 단 한 줄의 코드로 서버 장악 가능
- "역사상 최악의 취약점"

방어:
✅ 의존성 스캔 도구 사용
✅ 정기적인 업데이트
✅ 취약점 알림 모니터링
```

**2. 변조된 Docker 이미지**
```
위험한 시나리오:

1. Docker Hub에서 이미지 다운로드
   docker pull nginx

2. 하지만 가짜 이미지였음!
   → 악성 코드 포함
   → 암호화폐 채굴 프로그램
   → 백도어

3. 컨테이너 실행
   → 악성 코드 실행
   → 서버 장악당함

방어:
✅ 공식 이미지만 사용
✅ 이미지 서명 확인
```
