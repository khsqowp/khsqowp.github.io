# 🌐 3. 네트워크 격리 (VPC): 클라우드 네트워크의 안전한 설계

## 🔄 이전 단원 복습: 계정 및 접근 관리 (IAM)

이전 단원 `2. 계정 및 접근 관리 (IAM).md`에서 우리는 AWS IAM의 핵심 구성 요소(사용자, 그룹, 역할, 정책)를 이해하고, IAM 보안 모범 사례(MFA, 최소 권한, 임시 자격 증명)와 AI 기반 IAM 정책 분석 및 비정상 접근 탐지 전략을 학습했습니다.

> **핵심 복습:**
>
> *   **IAM의 중요성:** 클라우드 자원에 대한 '누가', '무엇에', '어떻게' 접근할 수 있는지를 통제하는 클라우드 보안의 첫 번째 방어선.
> *   **최소 권한 원칙:** 필요한 최소한의 권한만 부여하여 공격 시 피해 범위를 최소화.
> *   **AI 기반 IAM 보안:** AI가 IAM 정책의 취약점(권한 과부여, 와일드카드 사용)을 분석하고, CloudTrail 로그를 통해 비정상 로그인 및 권한 변경 시도를 탐지하여 자동 대응.
>
> IAM이 '누가' 접근할 수 있는지를 통제한다면, 오늘 배울 **VPC(Virtual Private Cloud)**는 '어디서', '어떻게' 네트워크 트래픽이 흐를 수 있는지를 통제하여 클라우드 환경의 네트워크를 안전하게 격리하고 제어하는 핵심 서비스입니다. VPC는 클라우드 인프라의 네트워크 보안을 위한 가장 기본적인 구성 요소이며, AI 기반으로 VPC 설정을 분석하고 비정상적인 네트워크 흐름을 탐지하는 방법을 학습합니다.

## 🤔 네트워크 격리 (VPC)를 왜 배워야 할까요? (The "Why")

클라우드 환경은 기본적으로 공유 인프라 위에 구축됩니다. 즉, 나의 가상 서버가 다른 고객의 가상 서버와 물리적으로 동일한 하드웨어에 존재할 수 있다는 의미입니다. 이러한 공유 환경에서 나의 자원을 안전하게 보호하고, 외부의 위협으로부터 격리하며, 내부 자원 간의 통신을 제어하는 것은 클라우드 보안의 필수 요소입니다.

**[VPC 지식의 중요성]**
*   **논리적 격리:** 퍼블릭 클라우드 환경에서 나만의 독립적인 가상 네트워크 공간을 생성하여, 다른 고객의 자원으로부터 나의 자원을 논리적으로 완벽하게 분리합니다. 이는 클라우드 환경에서 가장 기본적인 보안 경계선입니다.
*   **네트워크 제어:** IP 주소 범위, 서브넷, 라우팅 테이블, 네트워크 게이트웨이 등을 직접 정의하여 네트워크 트래픽의 흐름을 세밀하게 제어할 수 있습니다. 어떤 트래픽을 허용하고 차단할지, 어디로 보낼지 등을 결정합니다.
*   **다층 방어 구축:** VPC 내부에 여러 서브넷을 생성하고, 각 서브넷에 보안 그룹(Security Group)과 네트워크 ACL(Access Control List)을 적용하여 다층적인 네트워크 보안을 구축할 수 있습니다. 이는 공격자가 하나의 방어선을 뚫더라도 다음 방어선에서 저지될 수 있도록 합니다.
*   **AI 기반 VPC 보안:** AI는 VPC의 네트워크 설정(보안 그룹, NACL, 라우팅 테이블)을 분석하여 취약점을 탐지하고, VPC Flow Logs를 분석하여 비정상적인 네트워크 트래픽 패턴을 탐지하여 자동 대응할 수 있습니다. 방대한 설정과 동적인 트래픽을 인간이 모두 모니터링하기는 불가능합니다.

> **보안 전문가에게 VPC란?**
>
> 클라우드 환경에서 **'나만의 안전한 성(城)을 쌓고, 성벽과 해자를 구축하며, 성문과 통로를 통제하는 설계도'**입니다. VPC를 통해 우리는 클라우드 자원을 외부 위협으로부터 보호하고, 내부 자원 간의 통신을 안전하게 제어하며, AI 기반 보안 시스템이 네트워크 트래픽을 분석하고 대응할 수 있는 기반 환경을 마련합니다.

## 1. 🌐 AWS VPC의 핵심 구성 요소 심층 분석

AWS VPC(Virtual Private Cloud)는 AWS 클라우드에서 정의하는 논리적으로 격리된 네트워크입니다.

### 1.1. 🗺️ VPC (Virtual Private Cloud)

*   **설명:** AWS 클라우드 내에서 사용자가 정의하는 가상 네트워크입니다. IP 주소 범위(CIDR 블록)를 지정하여 생성합니다. 이 CIDR 블록은 VPC 내의 모든 서브넷과 인스턴스가 사용할 수 있는 IP 주소의 전체 범위를 정의합니다.
    *   **CIDR (Classless Inter-Domain Routing):** `10.0.0.0/16`과 같이 IP 주소와 네트워크 마스크를 함께 표기하는 방식입니다. `/16`은 네트워크 주소에 16비트를 사용하고, 나머지 16비트는 호스트 주소에 사용됨을 의미합니다.
*   **보안 관점:** VPC는 클라우드 자원을 위한 격리된 공간을 제공합니다. VPC 외부의 트래픽은 명시적으로 허용되지 않는 한 접근할 수 없습니다. 잘못된 CIDR 블록 선택은 IP 충돌이나 네트워크 확장성 문제를 야기할 수 있습니다.

### 1.2. 🗺️ 서브넷 (Subnet)

*   **설명:** VPC를 더 작은 네트워크 세그먼트로 나눈 것입니다. 각 서브넷은 VPC의 CIDR 블록 내에서 고유한 CIDR 블록을 가집니다. 서브넷은 가용 영역(Availability Zone) 내에 존재하며, 고가용성을 위해 여러 AZ에 서브넷을 분산 배치합니다.
*   **보안 관점:**
    *   **Public Subnet:** 인터넷 게이트웨이를 통해 인터넷과 직접 통신할 수 있는 서브넷. 웹 서버, 로드 밸런서 등 외부에 노출되어야 하는 자원 배치.
    *   **Private Subnet:** 인터넷 게이트웨이를 통해 인터넷과 직접 통신할 수 없는 서브넷. 데이터베이스, 애플리케이션 서버 등 민감한 자원 배치.
    *   **네트워크 분리:** 서브넷을 통해 네트워크를 분리하여, 한 서브넷에서 발생한 침해가 다른 서브넷으로 확산되는 것을 방지합니다. (네트워크 세그멘테이션)

### 1.3. 🛣️ 라우팅 테이블 (Route Table)

*   **설명:** 네트워크 트래픽이 어디로 가야 하는지(라우팅)를 결정하는 규칙들의 집합입니다. 각 서브넷은 하나의 라우팅 테이블과 연결됩니다. 라우팅 테이블의 각 규칙은 목적지(Destination)와 대상(Target)으로 구성됩니다.
    *   **목적지 (Destination):** 트래픽이 향하는 IP 주소 범위 (CIDR 블록).
    *   **대상 (Target):** 트래픽을 전달할 다음 홉(Next Hop) (예: 인터넷 게이트웨이, NAT 게이트웨이, VPC 피어링 연결).
*   **보안 관점:** 잘못된 라우팅 설정은 트래픽이 의도치 않은 경로로 흐르거나, 민감한 자원이 외부에 노출될 수 있습니다. (예: Private Subnet의 트래픽이 인터넷 게이트웨이로 라우팅되는 경우)

### 1.4. 🚪 인터넷 게이트웨이 (Internet Gateway, IGW)

*   **설명:** VPC와 인터넷 간의 통신을 가능하게 하는 게이트웨이입니다. Public Subnet이 인터넷과 통신하려면 IGW가 필요합니다. IGW는 단일 장애 지점(Single Point of Failure)이 아니며, 고가용성을 제공합니다.
*   **보안 관점:** IGW는 VPC 외부로 나가는 트래픽의 유일한 통로이므로, IGW를 통해 들어오고 나가는 트래픽에 대한 보안 정책(NACL, 보안 그룹)을 철저히 적용해야 합니다.

### 1.5.  NAT 게이트웨이 (NAT Gateway)

*   **설명:** Private Subnet에 있는 인스턴스가 인터넷으로 아웃바운드 트래픽을 보낼 수 있도록 하면서도, 외부에서 Private Subnet으로 인바운드 트래픽을 시작할 수 없도록 합니다. Public Subnet에 배치됩니다.
*   **보안 관점:** Private Subnet의 인스턴스가 보안 업데이트를 받거나 외부 API와 통신해야 할 때, 외부 노출 없이 안전하게 인터넷에 접근할 수 있도록 합니다.

### 1.6. 🛡️ 네트워크 ACL (Network Access Control List, NACL)

*   **설명:** 서브넷 수준에서 트래픽을 제어하는 방화벽입니다. 인바운드(Inbound) 및 아웃바운드(Outbound) 규칙을 모두 정의할 수 있습니다. 규칙은 번호 순서대로 평가되며, 가장 낮은 번호의 규칙이 먼저 적용됩니다.
*   **보안 관점:**
    *   **Stateless:** 연결 상태를 추적하지 않으므로, 인바운드 규칙을 허용하면 해당 응답 트래픽에 대한 아웃바운드 규칙도 명시적으로 허용해야 합니다.
    *   **Deny 규칙:** 명시적인 Deny 규칙을 설정할 수 있어, 특정 IP 대역이나 포트의 트래픽을 서브넷 전체에서 차단하는 데 유용합니다. (예: 알려진 악성 IP 대역 차단)

### 1.7. 🛡️ 보안 그룹 (Security Group)

*   **설명:** EC2 인스턴스 등 인스턴스 수준에서 트래픽을 제어하는 가상 방화벽입니다. 인스턴스에 연결되며, 인바운드 및 아웃바운드 규칙을 정의합니다.
*   **보안 관점:**
    *   **Stateful:** 연결 상태를 추적하므로, 인바운드 규칙을 허용하면 해당 응답 트래픽에 대한 아웃바운드 규칙은 자동으로 허용됩니다.
    *   **최소 권한 원칙:** 필요한 포트와 IP 대역만 허용하여 공격 표면을 최소화해야 합니다. `0.0.0.0/0` (모든 IP)에 대한 SSH(22) 또는 RDP(3389) 포트 개방은 매우 위험합니다.
    *   **참조 규칙:** 다른 보안 그룹을 소스로 지정하여 해당 보안 그룹에 속한 인스턴스 간의 통신을 허용할 수 있습니다.

## 2. 🛡️ VPC 보안 모범 사례와 AI 기반 분석

VPC는 클라우드 네트워크의 근간이므로, 안전하게 설계하고 관리하는 것이 중요합니다.

### 2.1. 🌐 VPC 설계: Public/Private Subnet 분리 및 다계층 아키텍처

웹 서버는 Public Subnet에, 데이터베이스는 Private Subnet에 배치하여 민감한 자원을 외부로부터 격리합니다. 또한, 웹 계층, 애플리케이션 계층, 데이터베이스 계층을 각각 다른 서브넷에 배치하는 다계층 아키텍처를 구성합니다.

#### **AI 연동 전략**

| 구분 | AI 연동 방안 |
| :--- | :--- |
| **설계 검토** | AI가 VPC 네트워크 구성을 분석하여 Public Subnet에 민감한 자원(예: RDS 인스턴스, 민감 데이터가 포함된 EC2)이 배치되어 있는지 탐지하고 경고. <br> **AI 모델:** 그래프 신경망 (GNN)을 사용하여 VPC 구성 요소 간의 연결 관계를 분석. |
| **트래픽 흐름 분석** | AI가 VPC Flow Logs를 분석하여 Private Subnet에서 외부 인터넷으로의 비정상적인 직접 통신 시도를 탐지. <br> **AI 모델:** 비지도 학습 (IsolationForest)으로 정상 트래픽 패턴에서 벗어나는 이상치 탐지. |

### 2.2. 🛡️ 보안 그룹 및 NACL 규칙 최적화

최소 권한 원칙에 따라 필요한 포트와 IP 대역만 허용하고, 불필요한 규칙은 제거합니다.

#### **AI 연동 전략**

| 구분 | AI 연동 방안 |
| :--- | :--- |
| **규칙 취약점 탐지** | AI가 보안 그룹 및 NACL 규칙을 분석하여 `0.0.0.0/0`에 대한 SSH/RDP 포트 개방, 불필요한 포트 개방 등 취약한 규칙을 탐지. <br> **AI 모델:** NLP (규칙 텍스트 분석), 지도 학습 (취약한 규칙 패턴 분류). |
| **규칙 최적화 제안** | AI가 실제 트래픽 흐름을 분석하여 사용되지 않는 규칙을 식별하고 제거를 제안하거나, 너무 광범위한 규칙을 더 세분화하도록 제안. <br> **AI 모델:** 강화 학습 (최적의 규칙 조합 탐색), 통계 분석. |
| **비정상 규칙 변경 탐지** | AI가 CloudTrail 로그를 분석하여 보안 그룹/NACL 규칙의 비정상적인 변경 시도를 탐지. <br> **AI 모델:** 비지도 학습 (정상적인 설정 변경 패턴 학습 후 이상치 탐지). |

### 2.3. 🔬 VPC Flow Logs 분석

VPC Flow Logs는 VPC 내의 네트워크 트래픽 흐름에 대한 정보를 기록합니다. 이를 분석하여 비정상적인 통신 패턴을 탐지할 수 있습니다.

*   **기록되는 정보:** 출발지/목적지 IP 주소, 포트, 프로토콜, 전송된 바이트/패킷 수, 트래픽 허용/거부 여부 등.
*   **활성화 방법:** VPC 콘솔 또는 AWS CLI/API를 통해 특정 네트워크 인터페이스, 서브넷 또는 VPC 전체에 대해 활성화할 수 있습니다.

#### **AI 연동 전략**

| 구분 | AI 연동 방안 |
| :--- | :--- |
| **이상 트래픽 탐지** | AI가 VPC Flow Logs를 분석하여 평소와 다른 포트/프로토콜 사용, 비정상적인 데이터 전송량, 특정 IP로부터의 대량 연결 시도 등 이상 트래픽을 탐지. <br> (예: 비지도 학습 모델(IsolationForest)을 사용하여 정상 트래픽 패턴에서 벗어나는 이상치 탐지) |
| **위협 인텔리전스 연동** | AI가 Flow Logs의 출발지/목적지 IP를 외부 위협 인텔리전스(VirusTotal 등)와 연동하여 악성 IP와의 통신을 탐지. |
| **공격 단계 예측** | AI가 Flow Logs의 시계열 데이터를 분석하여 공격자의 네트워크 내부 이동(Lateral Movement) 패턴을 탐지하고 공격 단계를 예측. <br> **AI 모델:** RNN/LSTM (시계열 분석). |

### 2.4. 💡 VPC Endpoints

*   **설명:** VPC 내에서 AWS 서비스(예: S3, DynamoDB)에 프라이빗하게 연결할 수 있도록 하는 기능입니다. 인터넷 게이트웨이, NAT 게이트웨이, VPN 연결 또는 AWS Direct Connect 연결을 통하지 않고도 AWS 서비스에 접근할 수 있습니다.
*   **보안 관점:** AWS 서비스로의 트래픽이 인터넷을 통과하지 않으므로 데이터 유출 위험을 줄이고, 네트워크 성능을 향상시킵니다.

### 2.5. 💡 Transit Gateway

*   **설명:** 수많은 VPC와 온프레미스 네트워크를 중앙에서 연결하고 관리할 수 있는 네트워크 허브입니다. 복잡한 네트워크 토폴로지를 단순화합니다.
*   **보안 관점:** 중앙 집중식 라우팅 및 보안 정책 적용을 통해 대규모 네트워크 환경의 보안 관리를 용이하게 합니다.

## 3. 💥 일반적인 VPC 취약점 및 공격 시나리오

VPC는 강력한 격리 기능을 제공하지만, 잘못 설정될 경우 다양한 공격에 노출될 수 있습니다.

### 3.1. 💥 과도하게 열린 보안 그룹/NACL 규칙

*   **설명:** `0.0.0.0/0` (모든 IP)에 대해 SSH(22), RDP(3389), 데이터베이스 포트(3306, 5432) 등이 열려 있는 경우.
*   **공격 시나리오:** 공격자가 무차별 대입 공격(Brute-force)을 통해 SSH/RDP 계정을 탈취하거나, 데이터베이스에 직접 접근하여 데이터를 유출할 수 있습니다.
*   **AI 탐지:** AI가 보안 그룹/NACL 규칙을 주기적으로 스캔하여 이러한 취약한 패턴을 탐지하고 경고.

### 3.2. 💥 Private Subnet 노출

*   **설명:** Private Subnet에 있는 인스턴스가 인터넷 게이트웨이로 직접 라우팅되거나, Public Subnet의 인스턴스에 불필요하게 연결되어 외부에서 접근 가능한 경우.
*   **공격 시나리오:** 민감한 데이터베이스나 애플리케이션 서버가 외부 공격에 직접 노출될 수 있습니다.
*   **AI 탐지:** AI가 라우팅 테이블과 보안 그룹 규칙을 분석하여 Private Subnet의 비정상적인 외부 연결 경로를 탐지.

### 3.3. 💥 VPC 피어링 연결의 과도한 권한

*   **설명:** 두 VPC 간에 직접 네트워크 연결을 설정하는 VPC 피어링 연결 시, 불필요하게 광범위한 IP 대역에 대한 접근을 허용하는 경우.
*   **공격 시나리오:** 한 VPC가 침해되면, 피어링 연결을 통해 다른 VPC로 공격이 확산될 수 있습니다.
*   **AI 탐지:** AI가 VPC 피어링 연결의 라우팅 테이블과 보안 그룹 규칙을 분석하여 과도한 권한 부여를 탐지.

### 3.4. 💥 VPC Flow Logs 미활성화 또는 미분석

*   **설명:** VPC Flow Logs가 활성화되지 않거나, 활성화되어 있어도 로그를 분석하지 않는 경우.
*   **공격 시나리오:** 네트워크 내부에서 발생하는 비정상적인 트래픽(예: C2 서버 통신, 내부망 스캔)을 탐지하지 못하여 공격자가 장기간 잠복하거나 데이터를 유출할 수 있습니다.
*   **AI 탐지:** AI가 AWS Config를 통해 VPC Flow Logs 활성화 여부를 모니터링하고, Flow Logs 데이터를 지속적으로 분석하여 이상 징후를 탐지.

## 📌 요약 정리 (Executive Summary)

1.  **VPC의 중요성**: 클라우드 환경에서 나만의 독립적인 가상 네트워크 공간을 생성하고, 네트워크 트래픽의 흐름을 세밀하게 제어하여 클라우드 자원을 외부 위협으로부터 격리하는 핵심 서비스이다.
2.  **AWS VPC 핵심 구성 요소**:
    *   **VPC**: 사용자가 정의하는 논리적으로 격리된 네트워크. CIDR 블록으로 IP 주소 범위 정의.
    *   **서브넷**: VPC를 더 작은 네트워크 세그먼트로 분리 (Public/Private). 가용 영역(AZ) 내에 존재.
    *   **라우팅 테이블**: 네트워크 트래픽의 경로를 결정하는 규칙 집합. 목적지-대상으로 구성.
    *   **인터넷 게이트웨이 (IGW)**: VPC와 인터넷 간의 통신을 가능하게 하는 통로.
    *   **NAT 게이트웨이 (NAT Gateway)**: Private Subnet 인스턴스의 아웃바운드 인터넷 접근 허용 (인바운드 차단).
    *   **네트워크 ACL (NACL)**: 서브넷 수준의 Stateless 방화벽. 명시적 Deny 규칙 설정 가능.
    *   **보안 그룹 (Security Group)**: 인스턴스 수준의 Stateful 방화벽. 최소 권한 원칙 적용.
3.  **VPC 보안 모범 사례**:
    *   **Public/Private Subnet 분리**: 민감한 자원을 Private Subnet에 배치하여 외부로부터 격리. 다계층 아키텍처 구성.
    *   **보안 그룹 및 NACL 규칙 최적화**: 최소 권한 원칙에 따라 필요한 포트와 IP 대역만 허용. `0.0.0.0/0`에 대한 SSH/RDP 개방은 매우 위험.
    *   **VPC Flow Logs 분석**: 네트워크 트래픽 흐름을 기록하여 비정상적인 통신 패턴 탐지.
    *   **VPC Endpoints**: AWS 서비스에 프라이빗하게 연결하여 인터넷 통과 없이 데이터 유출 위험 감소.
    *   **Transit Gateway**: 대규모 네트워크 환경에서 VPC 간 연결 및 중앙 관리.
4.  **AI 기반 VPC 보안**:
    *   **설정 취약점 분석**: AI가 보안 그룹, NACL, 라우팅 테이블 규칙을 분석하여 취약한 설정(예: 과도한 포트 개방)을 탐지하고 개선 제안.
    *   **비정상 트래픽 탐지**: AI가 VPC Flow Logs를 분석하여 평소와 다른 포트/프로토콜 사용, 비정상적인 데이터 전송량 등 이상 트래픽을 탐지 (비지도 학습 활용).
    *   **자동 대응**: 탐지된 위협에 대해 보안 그룹 규칙 변경, IP 차단 등의 자동화된 조치 수행.
5.  **보안 관점**: VPC는 클라우드 네트워크 보안의 근간이며, AI를 활용하여 VPC 설정을 지속적으로 모니터링하고, 비정상적인 트래픽 흐름을 탐지하며, 자동화된 대응을 통해 클라우드 인프라를 더욱 안전하게 구축하고 운영해야 한다. AWS CLI와 같은 도구를 통해 VPC 리소스를 관리하고 감사하는 능력은 모의 해킹 및 보안 분석에 필수적이다.
