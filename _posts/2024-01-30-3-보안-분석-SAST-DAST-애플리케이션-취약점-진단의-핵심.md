# 🔬 3. 보안 분석 (SAST, DAST): 애플리케이션 취약점 진단의 핵심

## 🔄 이전 단원 복습: 시큐어 코딩으로 개발 단계부터 보안 내재화

이전 단원 `2. 시큐어 코딩.md`에서 우리는 개발 단계부터 보안을 고려하여 안전한 코드를 작성하는 시큐어 코딩의 7대 원칙(입력값 검증, 보안 기능, 에러 처리 등)을 학습했습니다. 또한 AI 기반 시큐어 코딩 코드 리뷰 시스템 시나리오를 통해 AI가 코드의 보안 약점을 탐지하고 개선 방안을 제시하는 방법을 살펴보았습니다.

> **핵심 복습:**
> 
> *   **시큐어 코딩의 중요성:** 개발 초기 단계부터 취약점을 선제적으로 방지하여 비용을 절감하고 소프트웨어의 신뢰도를 높이는 필수적인 개발 방법론.
> *   **7대 원칙:** 입력값 검증, 보안 기능 구현, 에러 처리 등 각 원칙별로 발생할 수 있는 취약점과 방어 기법을 이해.
> *   **AI 기반 코드 리뷰:** AI가 정규표현식 기반 패턴 매칭과 머신러닝 모델을 활용하여 코드의 보안 약점을 탐지하고, 시큐어 코딩 가이드라인을 제시.
> 
> 시큐어 코딩은 취약점을 예방하는 가장 효과적인 방법이지만, 인간의 실수나 복잡한 로직으로 인해 모든 취약점을 완벽하게 제거하기는 어렵습니다. 따라서 개발된 애플리케이션에 대해 **보안 분석(Security Analysis)**을 수행하여 잠재적인 취약점을 찾아내고 개선하는 과정이 필수적입니다. 오늘 배울 '보안 분석 (SAST, DAST)' 단원에서는 애플리케이션 취약점 진단의 핵심 방법론인 정적 분석(SAST)과 동적 분석(DAST)의 원리, 장단점, 그리고 AI 기반으로 이러한 분석을 고도화하는 방법을 학습합니다.

---

## 🤔 보안 분석 (SAST, DAST)을 왜 배워야 할까요? (The "Why")

아무리 시큐어 코딩을 잘해도, 개발자의 실수나 새로운 공격 기법의 등장으로 인해 애플리케이션에 취약점이 존재할 수 있습니다. 이러한 취약점은 서비스 배포 전에 반드시 찾아내어 제거해야 합니다. SAST(Static Application Security Testing)와 DAST(Dynamic Application Security Testing)는 애플리케이션의 보안 상태를 점검하고 취약점을 진단하는 대표적인 방법론입니다.

**[보안 분석 지식의 중요성]**
*   **선제적 취약점 발견:** 서비스 배포 전에 잠재적인 취약점을 발견하여 심각한 보안 사고를 예방합니다.
*   **개발 프로세스 통합:** 개발 초기부터 운영 단계까지 소프트웨어 개발 생명 주기(SDLC) 전반에 걸쳐 보안을 내재화하는 데 기여합니다. 'Shift Left' 보안 전략의 핵심입니다.
*   **규제 준수:** 국내외 보안 관련 법규 및 가이드라인(예: 전자금융거래법, 정보통신망법, GDPR)은 애플리케이션 보안 취약점 진단을 의무화하고 있습니다.
*   **AI 기반 분석 고도화:** AI는 SAST/DAST 도구의 한계를 보완하고, 분석의 정확도와 효율성을 높이며, 새로운 형태의 취약점까지 탐지할 수 있도록 돕습니다.

> **보안 전문가에게 SAST와 DAST란?**
> 
> 애플리케이션이라는 **'건물의 안전 진단 도구'**입니다. SAST는 **'설계도(소스 코드)'**를 분석하여 구조적 결함을 찾아내고, DAST는 **'실제 건물(실행 중인 애플리케이션)'**을 흔들어보며 숨겨진 약점을 찾아냅니다. AI는 이 안전 진단 과정에서 **'정밀 진단 전문가'** 역할을 수행하여, 인간이나 기존 도구가 놓칠 수 있는 미세한 균열까지 찾아내고, 진단 결과를 바탕으로 보강 공사(패치)를 제안합니다.

## 1. 🔬 정적 애플리케이션 보안 테스팅 (SAST, Static Application Security Testing)

SAST는 애플리케이션을 **실행하지 않고** 소스 코드, 바이트 코드 또는 바이너리 코드를 분석하여 보안 취약점을 탐지하는 방법입니다. 개발 초기 단계부터 적용하여 취약점을 빠르게 발견하고 수정할 수 있습니다.

### 1.1. ⚙️ SAST의 작동 원리 및 주요 도구

*   **코드 파싱:** 소스 코드를 구문 분석하여 추상 구문 트리(AST, Abstract Syntax Tree)와 같은 내부 표현으로 변환합니다.
*   **데이터 흐름 분석 (Data Flow Analysis):** 변수가 어떻게 정의되고 사용되는지, 데이터가 함수 간에 어떻게 전달되는지 등을 추적하여 민감 데이터의 흐름을 파악합니다.
*   **제어 흐름 분석 (Control Flow Analysis):** 프로그램의 실행 경로를 분석하여 조건문, 반복문 등의 로직을 파악하고, 모든 가능한 실행 경로를 탐색합니다.
*   **패턴 매칭:** 알려진 취약점 패턴(예: SQL Injection에 취약한 문자열 연결)과 일치하는 코드를 식별합니다.
*   **규칙 기반 분석:** 미리 정의된 보안 규칙(예: OWASP Top 10, CWE)에 위배되는 코드를 탐지합니다.
*   **주요 SAST 도구:**
    *   **상용:** Checkmarx, Fortify, SonarQube (일부 유료 기능)
    *   **오픈소스:** SonarQube (커뮤니티 에디션), Bandit (Python), FindBugs (Java), ESLint (JavaScript)

### 1.2. 🛡️ SAST의 장단점

| 장점 | 단점 |
| :--- | :--- |
| **개발 초기 단계 적용 가능:** 코드가 완성되지 않아도 분석 가능하며, 개발자가 취약점을 조기에 수정할 수 있습니다. | **오탐(False Positive) 발생 가능성:** 실제 취약점이 아닌데도 경고를 발생시키는 경우가 많아 개발자의 피로도를 높일 수 있습니다. |
| **모든 코드 라인 분석:** 실행 경로에 관계없이 모든 코드 분석이 가능하여 잠재적인 취약점을 광범위하게 탐지합니다. | **미탐(False Negative) 발생 가능성:** 런타임 환경이나 외부 라이브러리 상호작용으로 발생하는 취약점 탐지 어려움. 특히 비즈니스 로직 취약점은 탐지하기 어렵습니다. |
| **취약점 위치 정확:** 소스 코드 레벨에서 취약점 위치를 정확히 지적하여 개발자가 수정하기 용이합니다. | **복잡한 비즈니스 로직 취약점 탐지 어려움:** 실제 사용자 행위나 데이터 흐름을 시뮬레이션하기 어려워 논리적 결함은 탐지하기 어렵습니다. |

#### **AI 연동 전략**

| 구분 | AI 연동 방안 |
| :--- | :--- |
| **오탐/미탐 감소** | AI가 SAST 도구의 분석 결과를 학습하여 오탐을 줄이고, 기존 규칙 기반 SAST가 놓치는 새로운 형태의 취약점 패턴을 탐지.<br> **AI 모델:** 지도 학습 (오탐/정탐 분류), 비지도 학습 (새로운 취약점 패턴 탐지). |
| **코드 문맥 이해** | AI가 코드의 문맥과 개발자의 의도를 이해하여, 단순 패턴 매칭을 넘어 실제 취약점으로 이어질 가능성이 높은 코드를 식별.<br> **AI 모델:** NLP (코드 임베딩, 의미론적 분석). |
| **자동 수정 제안** | AI가 탐지된 취약점에 대해 시큐어 코딩 가이드라인을 기반으로 수정 방안을 제시하거나, 자동으로 안전한 코드로 수정.<br> **AI 모델:** LLM (코드 생성 및 수정). |
| **취약점 우선순위 지정** | AI가 탐지된 취약점의 심각도, 코드의 중요도, 과거 패치 이력 등을 분석하여 개발자가 수정해야 할 취약점의 우선순위를 지정. |

#### PoC: 간단한 SAST (정규표현식 기반)

```python
import re

def simple_sast_scanner(code_content):
    """
    간단한 정규표현식 기반 SAST 스캐너 (개념).
    하드코딩된 비밀번호, SQL Injection 패턴 등을 탐지합니다.
    """
    findings = []

    # 하드코딩된 비밀번호 패턴
    hardcoded_password_pattern = r"(password|secret|api_key)\s*=\s*['"](.+?)['"]"
    for match in re.finditer(hardcoded_password_pattern, code_content, re.IGNORECASE):
        findings.append({
            "type": "HARDCODED_CREDENTIALS",
            "severity": "Critical",
            "line": code_content.count('\n', 0, match.start()) + 1,
            "code": match.group(0),
            "description": f"하드코딩된 민감 정보 '{match.group(1)}' 탐지: '{match.group(2)}'",
            "recommendation": "환경 변수 또는 비밀 관리 시스템을 사용하세요."
        })

    # SQL Injection 패턴 (f-string 사용 시)
    sql_injection_pattern = r"f['"].*SELECT.*FROM.*\sWHERE\s.*=\s*\{.*?\}.*['"]"
    for match in re.finditer(sql_injection_pattern, code_content, re.IGNORECASE):
        findings.append({
            "type": "SQL_INJECTION",
            "severity": "High",
            "line": code_content.count('\n', 0, match.start()) + 1,
            "code": match.group(0),
            "description": "f-string을 사용한 SQL 쿼리에서 SQL Injection 취약점 가능성",
            "recommendation": "Prepared Statement 또는 ORM을 사용하세요."
        })
    
    return findings

# ---
# 테스트 코드
# ---

test_code = """
DB_HOST = "localhost"
DB_USER = "admin"
DB_PASSWORD = 'mysecretpassword' # 💥 취약점
API_KEY = "sk-xxxxxxxxxxxxxxxxxxxx" # 💥 취약점

def get_user_data(user_id):
    query = f"SELECT * FROM users WHERE id = {user_id};" # 💥 취약점
    # cursor.execute(query)
    return "user_data"

def process_data():
    data = "some data"
    return data
"""

sast_findings = simple_sast_scanner(test_code)
if sast_findings:
    print("\n--- 🔬 SAST 스캔 결과 ---")
    for finding in sast_findings:
        print(f"  - 유형: {finding['type']}")
        print(f"  - 심각도: {finding['severity']}")
        print(f"  - 라인: {finding['line']}")
        print(f"  - 코드: {finding['code']}")
        print(f"  - 설명: {finding['description']}")
        print(f"  - 권고: {finding['recommendation']}")
else:
    print("\n--- 🔬 SAST 스캔 결과: 취약점 없음 ---")
```

## 2. 🔬 동적 애플리케이션 보안 테스팅 (DAST, Dynamic Application Security Testing)

DAST는 애플리케이션을 **실행한 상태에서** 실제 공격을 시뮬레이션하고, 그 응답을 분석하여 보안 취약점을 탐지하는 방법입니다. 블랙박스 테스팅 방식으로, 소스 코드에 대한 접근 없이도 수행할 수 있습니다.

### 2.1. ⚙️ DAST의 작동 원리 및 주요 도구

*   **크롤링:** 웹 애플리케이션을 자동으로 탐색하여 모든 페이지와 입력 필드, 링크 등을 식별합니다.
*   **페이로드 주입:** 식별된 입력 필드에 SQL Injection, XSS, Command Injection 등 다양한 공격 페이로드(Payload)를 주입합니다.
*   **응답 분석:** 웹 서버의 응답(HTTP 상태 코드, 에러 메시지, HTML 소스 코드 변경 등)을 분석하여 취약점 존재 여부를 판단합니다.
*   **런타임 환경 분석:** 실제 애플리케이션이 동작하는 환경에서 취약점을 탐지하므로, SAST가 놓칠 수 있는 런타임 환경 관련 취약점(예: 설정 오류, 외부 라이브러리 상호작용)을 발견할 수 있습니다.
*   **주요 DAST 도구:**
    *   **상용:** Acunetix, Netsparker, Burp Suite (Professional)
    *   **오픈소스:** OWASP ZAP, Nikto, w3af

### 2.2. 🛡️ DAST의 장단점

| 장점 | 단점 |
| :--- | :--- |
| **실제 공격 시뮬레이션:** 실제 공격과 유사한 방식으로 취약점 탐지. 공격자가 악용할 수 있는 경로를 직접 확인합니다. | **코드 커버리지 한계:** 크롤링을 통해 접근 가능한 경로만 분석 가능. 모든 코드 라인 분석 불가. 특히 인증이 필요한 페이지나 복잡한 비즈니스 로직은 탐색하기 어렵습니다. |
| **오탐률 낮음:** 실제 공격을 통해 취약점을 확인하므로 오탐률이 상대적으로 낮습니다. | **개발 후반 단계 적용:** 애플리케이션이 실행 가능한 상태여야 분석 가능. 개발 초기 단계에서는 적용하기 어렵습니다. |
| **런타임 환경 취약점 탐지:** 설정 오류, 외부 라이브러리 상호작용 등 런타임 환경 관련 취약점 탐지 가능. | **시간 소요:** 스캔 시간이 길고, 복잡한 애플리케이션의 경우 모든 경로 탐색이 어려울 수 있습니다. |

#### **AI 연동 전략**

| 구분 | AI 연동 방안 |
| :--- | :--- |
| **공격 시나리오 최적화** | AI가 애플리케이션의 구조와 기능을 분석하여 가장 효과적인 공격 시나리오(페이로드, 공격 경로)를 자동으로 생성하고 최적화.<br> **AI 모델:** 강화 학습 (최적의 공격 전략 탐색), LLM (페이로드 생성). |
| **응답 분석 고도화** | AI가 DAST 도구의 응답(HTTP 상태 코드, 에러 메시지, HTML 변경 등)을 분석하여 오탐을 줄이고, 실제 취약점 여부를 판단.<br> (예: AI가 웹 서버 로그를 분석하여 DAST 스캔 중 발생하는 비정상적인 데이터베이스 쿼리나 시스템 명령 실행 탐지) |
| **새로운 취약점 패턴 학습** | AI가 DAST 스캔 결과를 학습하여 알려지지 않은 새로운 형태의 취약점 패턴을 식별하고, 이를 다음 스캔에 반영.<br> **AI 모델:** 비지도 학습 (응답 패턴 이상 탐지). |
| **취약점 우선순위 지정** | AI가 탐지된 취약점의 심각도, 공격 성공 가능성, 비즈니스 영향도 등을 분석하여 개발자가 수정해야 할 취약점의 우선순위를 지정. |

#### PoC: 간단한 DAST (SQL Injection 테스트)

```python
import requests

def simple_dast_sql_injection(target_url, param_name, payloads):
    """
    SQL Injection 취약점을 테스트하는 간단한 DAST 함수 (개념).
    실제 DAST 도구는 더 많은 페이로드와 응답 분석 로직을 가집니다.
    """
    findings = []
    
    print(f"\n--- 🔬 DAST SQL Injection 테스트 시작: {target_url} ({param_name}) ---")
    for payload in payloads:
        test_url = f"{target_url}?{param_name}={payload}"
        try:
            response = requests.get(test_url, timeout=5)
            
            # SQL 에러 메시지 탐지
            if "SQL syntax" in response.text or "mysql_fetch_array" in response.text or "ORA-" in response.text:
                findings.append({
                    "type": "SQL_INJECTION",
                    "severity": "High",
                    "url": test_url,
                    "payload": payload,
                    "description": "응답에서 SQL 에러 메시지 탐지",
                    "recommendation": "Prepared Statement 또는 ORM 사용, 입력값 검증."
                })
                print(f"  - 💥 SQL Injection 취약점 탐지! (Payload: {payload})")
                return findings # 취약점 발견 시 중단
            
            # 참/거짓 기반 블라인드 SQL Injection (간단화)
            # 실제로는 응답 시간, 내용 변화 등을 분석
            if "admin" in payload and "Welcome" in response.text: # 'admin'으로 로그인 성공 가정
                findings.append({
                    "type": "BLIND_SQL_INJECTION",
                    "severity": "High",
                    "url": test_url,
                    "payload": payload,
                    "description": "블라인드 SQL Injection 가능성 (응답 내용 변화)",
                    "recommendation": "Prepared Statement 또는 ORM 사용, 입력값 검증."
                })
                print(f"  - 💥 블라인드 SQL Injection 취약점 탐지! (Payload: {payload})")
                return findings

        except requests.exceptions.RequestException as e:
            print(f"  - 요청 실패 ({test_url}): {e}")
            
    print("  - SQL Injection 취약점 탐지 실패.")
    return findings

target_app_url = "http://example.com/products" # 가상 취약한 웹 애플리케이션 URL
```

## 3. 🛡️ SAST와 DAST의 보완적 활용 및 AI 기반 통합 분석

SAST와 DAST는 각각의 장단점이 명확하므로, 두 가지 방법을 **보완적으로 활용**하는 것이 가장 효과적입니다. SAST로 개발 초기에 코드 레벨의 취약점을 빠르게 발견하고, DAST로 배포 전 런타임 환경에서의 취약점을 최종 검증하는 방식입니다.

### 3.1. 💡 IAST (Interactive Application Security Testing)

IAST는 SAST와 DAST의 장점을 결합한 형태로, 애플리케이션이 실행되는 동안 내부에서 동작하며 코드 레벨의 취약점과 런타임 환경의 취약점을 동시에 탐지합니다. 애플리케이션에 계측(Instrumentation)을 삽입하여 코드 실행 경로와 데이터 흐름을 추적합니다.

### 3.2. 🤖 AI 기반 통합 분석 플랫폼

AI는 SAST, DAST, IAST 도구에서 수집된 방대한 데이터를 통합 분석하여, 각 도구의 한계를 보완하고 전체적인 취약점 진단의 정확도와 효율성을 극대화할 수 있습니다.

*   **데이터 통합:** SAST의 코드 분석 결과, DAST의 공격 시뮬레이션 결과, WAF/IPS 로그, 개발자 코드 리뷰 코멘트 등 모든 보안 관련 데이터를 통합.
*   **상관관계 분석:** AI가 통합된 데이터를 기반으로 서로 다른 도구에서 탐지된 취약점 간의 상관관계를 분석하여, 실제 위협으로 이어질 가능성이 높은 취약점에 우선순위를 부여.
*   **자동화된 리포팅:** AI가 분석 결과를 바탕으로 취약점 보고서를 자동으로 생성하고, 개발자에게 수정 가이드라인을 제시.
*   **오탐 감소 및 미탐 보완:** AI가 SAST의 오탐을 줄이고, DAST가 놓칠 수 있는 코드 레벨의 취약점을 보완하며, IAST의 결과를 검증하여 전체적인 탐지 정확도를 높입니다.

## 4. 🚀 SDLC (Software Development Life Cycle)에 보안 분석 통합

보안 분석은 개발 생명 주기 전반에 걸쳐 통합되어야 합니다. 이를 DevSecOps라고 부르기도 합니다.

*   **설계 단계:** 위협 모델링(Threat Modeling)을 통해 잠재적 위협 식별.
*   **개발 단계:** SAST를 통해 코드 작성 중 취약점 조기 발견. 시큐어 코딩 가이드라인 적용.
*   **테스트 단계:** DAST, 침투 테스트(Penetration Testing)를 통해 실제 공격 시뮬레이션.
*   **배포 단계:** CI/CD 파이프라인에 보안 게이트를 통합하여 취약한 코드 배포 방지.
*   **운영 단계:** WAF, IDS/IPS, SIEM 등을 통해 실시간 모니터링 및 위협 대응.

**[상황]**

**[워크플로우]**

2.  **CI/CD 트리거:** CI/CD 파이프라인이 자동으로 빌드 및 테스트를 시작합니다.
3.  **AI 기반 SAST 수행:**
    *   AI 기반 SAST 도구가 새로 커밋된 코드에 대해 정적 분석을 수행합니다.
    *   AI는 코드에서 SQL Injection, XSS, 하드코딩된 자격 증명 등 시큐어 코딩 위반 사항을 탐지합니다.
4.  **AI 기반 DAST 수행:**
    *   빌드된 애플리케이션을 테스트 환경에 배포하고, AI 기반 DAST 도구가 동적 분석을 수행합니다.
    *   AI는 실제 공격 페이로드를 주입하여 SQL Injection, XSS, SSRF 등 런타임 취약점을 탐지합니다.
5.  **AI 기반 통합 분석 및 의사 결정:**
    *   AI는 SAST와 DAST에서 탐지된 모든 취약점 정보를 통합합니다.
    *   AI는 각 취약점의 심각도, 오탐 가능성, 비즈니스 영향도 등을 종합적으로 평가하여 배포 허용 여부를 결정합니다. (예: Critical/High 등급 취약점 발견 시 배포 중단)
6.  **결과 보고 및 알림:**
    *   AI가 탐지된 취약점 목록, 심각도, 수정 가이드라인을 포함한 상세 보고서를 생성합니다.
    *   배포 중단 또는 허용 여부를 Slack으로 개발팀과 보안팀에 알립니다.

```python
def get_sast_results_mock(code_changes):
    """가상 SAST 결과 반환"""
    results = []
    if "f'SELECT * FROM users WHERE id = {user_id}'" in code_changes:
        results.append({'type': 'SQL_INJECTION', 'severity': 'High', 'location': 'line 10', 'description': 'SQL Injection 취약점 가능성'})

class AISecurityGate:
    def __init__(self):
        # AI 모델 (가상): SAST/DAST 결과를 기반으로 배포 여부 결정
        # 실제로는 취약점 유형, 심각도, 비즈니스 영향도 등을 특징으로 학습
        self.ai_decision_model = RandomForestClassifier(random_state=42)
        # 가상 훈련 데이터: (SQLi_High, XSS_Medium, Hardcoded_Critical) -> Deploy (0) / Block (1)
        X_train_features = [
            [0, 0, 0], # No vulns -> Deploy
            [1, 0, 0], # SQLi High -> Block
            [0, 1, 0], # XSS Medium -> Deploy (minor)
            [0, 0, 1], # Hardcoded Critical -> Block
            [1, 1, 0]  # SQLi High, XSS Medium -> Block
        ]
        y_train_decision = [0, 1, 0, 1, 1]
        self.ai_decision_model.fit(X_train_features, y_train_decision)

    def evaluate_deployment(self, sast_results, dast_results):
        print("\n--- 🤖 AI 기반 통합 분석 및 배포 의사 결정 ---")
        all_findings = sast_results + dast_results
        
        # 취약점 심각도 매핑 (숫자로 변환)
        severity_map = {'Critical': 3, 'High': 2, 'Medium': 1, 'Low': 0}

        sql_high = 0
        xss_medium = 0
        hardcoded_critical = 0

        for finding in all_findings:
            if finding['type'] == 'SQL_INJECTION' and severity_map.get(finding['severity'], 0) >= severity_map['High']:

            if finding['type'] == 'XSS' and severity_map.get(finding['severity'], 0) >= severity_map['Medium']:
                xss_medium = 1
            if finding['type'] == 'HARDCODED_CREDENTIALS' and severity_map.get(finding['severity'], 0) >= severity_map['Critical']:
                hardcoded_critical = 1
        
        input_features = [sql_high, xss_medium, hardcoded_critical]
        
        decision = self.ai_decision_model.predict([input_features])[0]
        
        if decision == 1:
            print("  - 🔴 AI 결정: 배포 중단! (심각한 취약점 발견)")
            return False
        else:
            print("  - ✅ AI 결정: 배포 허용 (경미하거나 취약점 없음)")
            return True
```

## 📌 요약 정리 (Executive Summary)

1.  **보안 분석의 필요성**: 시큐어 코딩으로 모든 취약점을 제거하기 어렵기 때문에, 개발된 애플리케이션에 대해 SAST와 DAST를 수행하여 잠재적인 취약점을 찾아내고 개선하는 과정이 필수적이다.
2.  **SAST (Static Application Security Testing)**:
    *   **특징**: 애플리케이션을 **실행하지 않고** 소스 코드/바이트 코드/바이너리 코드를 분석하여 취약점 탐지. 개발 초기 단계 적용 가능, 취약점 위치 정확.
    *   **단점**: 오탐/미탐 발생 가능성, 복잡한 비즈니스 로직 취약점 탐지 어려움.
    *   **AI 연동**: AI가 코드 문맥을 이해하여 오탐/미탐을 줄이고, 새로운 취약점 패턴을 학습하며, 자동 수정 제안.
    *   **주요 도구**: Checkmarx, Fortify, SonarQube, Bandit 등.
3.  **DAST (Dynamic Application Security Testing)**:
    *   **특징**: 애플리케이션을 **실행한 상태에서** 실제 공격을 시뮬레이션하여 취약점 탐지. 오탐률 낮음, 런타임 환경 취약점 탐지 가능.
    *   **단점**: 코드 커버리지 한계, 개발 후반 단계 적용, 시간 소요.
    *   **AI 연동**: AI가 공격 시나리오를 최적화하고, 응답 분석을 고도화하며, 새로운 취약점 패턴을 학습.
    *   **주요 도구**: Acunetix, Netsparker, OWASP ZAP, Nikto 등.
4.  **SAST와 DAST의 보완적 활용**: 두 가지 방법을 함께 사용하여 개발 초기부터 운영 단계까지 소프트웨어 개발 생명 주기(SDLC) 전반에 걸쳐 보안을 강화하는 것이 가장 효과적이다.
5.  **IAST (Interactive Application Security Testing)**: SAST와 DAST의 장점을 결합한 형태로, 애플리케이션 실행 중 코드 레벨 및 런타임 취약점을 동시에 탐지.
6.  **AI 기반 통합 분석 플랫폼**: SAST, DAST, IAST 도구에서 수집된 데이터를 통합 분석하여 각 도구의 한계를 보완하고, 취약점 진단의 정확도와 효율성을 극대화한다. AI는 취약점 우선순위 지정, 오탐 감소, 미탐 보완, 자동화된 리포팅 등을 지원한다.
7.  **SDLC에 보안 분석 통합**: 설계, 개발, 테스트, 배포, 운영 등 SDLC 전반에 걸쳐 보안 분석을 통합하는 'Shift Left' 보안 전략이 중요하다.
8.  **현직자 통합 시나리오**: AI 기반 CI/CD 파이프라인 보안 게이트를 통해 개발자가 코드를 커밋할 때마다 SAST와 DAST를 자동으로 수행하고, AI가 분석 결과를 통합하여 배포 허용 여부를 결정함으로써 SDLC 전반에 걸쳐 보안을 내재화하는 방법을 보여준다.
9.  **보안 관점**: SAST와 DAST는 애플리케이션 보안의 필수적인 도구이며, AI는 이들 도구의 한계를 보완하고 분석의 정확도와 효율성을 높여, 개발 프로세스에 보안을 통합하고 자동화된 취약점 관리를 가능하게 하는 핵심 기술이다.
