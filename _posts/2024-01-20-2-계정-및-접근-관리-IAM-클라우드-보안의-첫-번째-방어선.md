# 🔑 2. 계정 및 접근 관리 (IAM): 클라우드 보안의 첫 번째 방어선

## 🔄 이전 단원 복습: 클라우드 컴퓨팅과 책임 공유 모델

이전 단원 `1. 클라우드 컴퓨팅과 보안.md`에서 우리는 클라우드 컴퓨팅의 기본 개념, 서비스 모델(IaaS, PaaS, SaaS), 배포 모델, 그리고 클라우드 보안의 핵심인 책임 공유 모델(Shared Responsibility Model)을 학습했습니다. 또한 클라우드 환경의 주요 보안 위협인 설정 오류(Misconfiguration)와 계정 탈취(Account Compromise)에 대해서도 살펴보았습니다.

> **핵심 복습:**
> 
> *   **책임 공유 모델:** CSP는 '클라우드 자체'의 보안을, 사용자는 '클라우드 내'의 보안을 책임지며, 대부분의 클라우드 보안 사고는 사용자 책임 영역에서 발생합니다.
> *   **설정 오류:** S3 버킷 공개, 보안 그룹 과도한 개방 등은 AI 기반 탐지 및 자동 수정으로 대응할 수 있습니다.
> *   **계정 탈취:** 비정상 로그인, 권한 상승 시도 등은 AI 기반 분석으로 탐지하고 대응해야 합니다.
> 
> 클라우드 환경에서 사용자의 책임 영역 중 가장 중요하고 기본적인 것은 바로 **'누가', '무엇에', '어떻게' 접근할 수 있는지를 관리하는 것**입니다. 오늘 배울 '계정 및 접근 관리(IAM)' 단원에서는 클라우드 보안의 첫 번째 방어선이자 가장 중요한 통제 지점인 IAM(Identity and Access Management)의 개념과 작동 원리, 그리고 AI 기반으로 IAM 정책을 분석하고 비정상적인 접근을 탐지하는 방법을 학습합니다.

## 🤔 계정 및 접근 관리 (IAM)를 왜 배워야 할까요? (The "Why")

클라우드 환경에서는 모든 자원(가상 서버, 스토리지, 데이터베이스 등)이 API를 통해 접근 가능합니다. 따라서 '누가' 어떤 자원에 접근할 수 있는지를 통제하는 것은 클라우드 보안의 핵심 중의 핵심입니다. IAM은 이러한 접근을 중앙에서 관리하고 통제하는 서비스입니다.

**[IAM 지식의 중요성]**
*   **최소 권한 원칙 (Principle of Least Privilege):** 사용자나 서비스에게 필요한 최소한의 권한만 부여하여, 권한 오남용이나 계정 탈취 시 피해 범위를 최소화합니다. IAM은 이를 구현하는 핵심 도구입니다.
*   **계정 탈취 방지:** 강력한 인증(MFA)과 세분화된 권한 관리를 통해 계정 탈취 시 공격자가 얻을 수 있는 이득을 제한하고, 비정상적인 접근을 조기에 탐지할 수 있습니다.
*   **컴플라이언스 준수:** GDPR, HIPAA 등 규제 준수를 위해 사용자 접근 기록을 감사하고, 적절한 권한 정책이 적용되었음을 증명해야 합니다.
*   **AI 기반 IAM 보안:** AI는 방대한 IAM 활동 로그를 분석하여 비정상적인 권한 변경, 역할 위임, 접근 패턴 등을 탐지하고, IAM 정책의 취약점을 자동으로 분석하여 개선을 제안할 수 있습니다.

> **보안 전문가에게 IAM이란?**
> 
> 클라우드 환경의 **'신분증 발급 및 출입 통제 시스템'**입니다. 누가 어떤 문을 열 수 있고, 어떤 자원을 사용할 수 있는지에 대한 모든 규칙을 정의하고 강제하는 핵심 보안 메커니즘입니다. IAM을 제대로 이해하고 활용하는 것은 클라우드 환경에서 **'보안의 시작이자 끝'**이라고 할 수 있습니다.

## 1. 🔑 AWS IAM의 핵심 구성 요소 심층 분석

AWS IAM(Identity and Access Management)은 AWS 리소스에 대한 접근을 안전하게 관리할 수 있도록 돕는 웹 서비스입니다.

### 1.1. 👤 사용자 (Users)

AWS 리소스와 상호 작용하는 사람 또는 애플리케이션을 나타냅니다.

*   **IAM 사용자:** AWS 계정 내에서 생성되는 개별 사용자. 고유한 자격 증명(액세스 키, 비밀번호)을 가집니다.
*   **루트 사용자:** AWS 계정을 생성할 때 만들어지는 최상위 사용자. 모든 권한을 가지므로 일상적인 작업에는 사용하지 않는 것이 강력히 권고됩니다. 대신 IAM 사용자를 생성하여 사용해야 합니다.

### 1.2. 👥 그룹 (Groups)

IAM 사용자들의 집합입니다. 그룹에 권한을 부여하면, 해당 그룹에 속한 모든 사용자에게 동일한 권한이 적용됩니다. 이를 통해 여러 사용자에게 동일한 권한을 효율적으로 관리할 수 있습니다.

### 1.3. 👮 역할 (Roles)

특정 엔티티(IAM 사용자, AWS 서비스, 외부 계정 등)가 임시적으로 권한을 위임받아 AWS 리소스에 접근할 수 있도록 하는 보안 자격 증명입니다. 비밀번호나 액세스 키를 가지지 않으며, 임시 자격 증명(Temporary Credentials)을 사용합니다.

> **보안 관점: 역할(Role)의 중요성**
> 
> *   **최소 권한 원칙 구현:** 특정 작업 수행 시에만 필요한 권한을 가진 역할을 부여하여, 평소에는 최소한의 권한만 가지도록 합니다.
> *   **자격 증명 관리 간소화:** EC2 인스턴스나 Lambda 함수와 같은 AWS 서비스가 다른 AWS 서비스에 접근할 때, 액세스 키를 하드코딩하는 대신 역할을 부여하여 자격 증명 유출 위험을 줄입니다.
> *   **교차 계정 접근:** 다른 AWS 계정의 리소스에 안전하게 접근할 수 있도록 합니다.

### 1.4. 📜 정책 (Policies) 심층 분석

AWS 리소스에 대한 권한을 정의하는 문서입니다. JSON 형식으로 작성되며, 어떤 작업(Action)을 어떤 리소스(Resource)에 대해 허용(Allow)하거나 거부(Deny)할지 명시합니다.

```json
{
    "Version": "2012-10-17",
    "Statement": [
        {
            "Effect": "Allow",
            "Action": [
                "s3:GetObject",
                "s3:ListBucket"
            ],
            "Resource": [
                "arn:aws:s3:::my-secure-bucket",
                "arn:aws:s3:::my-secure-bucket/*"
            ],
            "Condition": {
                "IpAddress": {"aws:SourceIp": "203.0.113.0/24"}
            }
        },
        {
            "Effect": "Deny",
            "Action": "s3:DeleteObject",
            "Resource": "arn:aws:s3:::my-secure-bucket/*"
        }
    ]
}
```

*   **`Version`:** 정책 언어의 버전. 항상 최신 버전을 사용하는 것이 좋습니다.
*   **`Statement`:** 정책의 핵심 요소. 여러 개의 Statement를 가질 수 있습니다.
    *   **`Effect`:** `Allow` (허용) 또는 `Deny` (거부). `Deny`는 `Allow`보다 우선합니다.
    *   **`Action`:** 허용하거나 거부할 AWS 서비스 작업. (예: `s3:GetObject`, `ec2:RunInstances`). `*`는 모든 작업을 의미합니다.
    *   **`Resource`:** 작업이 적용될 AWS 리소스. ARN(Amazon Resource Name) 형식으로 지정합니다. `*`는 모든 리소스를 의미합니다.
    *   **`Condition`:** 정책이 적용되는 조건을 정의합니다. (예: 특정 IP 주소에서만 접근 허용, 특정 시간대에만 접근 허용).

> **용어 해설: ARN (Amazon Resource Name)**
> 
> AWS 리소스를 고유하게 식별하는 표준화된 이름 형식입니다. `arn:aws:s3:::my-secure-bucket`과 같이 사용됩니다.

#### 💡 IAM 정책의 종류

1.  **자격 증명 기반 정책 (Identity-based Policies):** IAM 사용자, 그룹, 역할에 직접 연결되는 정책입니다.
2.  **리소스 기반 정책 (Resource-based Policies):** 특정 리소스(예: S3 버킷, SQS 큐)에 연결되는 정책입니다. 누가 해당 리소스에 접근할 수 있는지 정의합니다.
3.  **권한 경계 (Permissions Boundaries):** IAM 엔티티(사용자 또는 역할)가 가질 수 있는 최대 권한을 설정하는 고급 기능입니다.
4.  **서비스 제어 정책 (Service Control Policies, SCPs):** AWS Organizations에서 여러 계정에 걸쳐 권한을 중앙에서 관리하는 데 사용됩니다. 특정 서비스나 작업을 모든 계정에서 허용하거나 거부할 수 있습니다.

## 2. 🛡️ IAM 보안 모범 사례와 AI 기반 분석

IAM은 강력하지만, 잘못 설정하면 심각한 보안 취약점이 됩니다. 다음은 IAM 보안 모범 사례와 이를 AI로 어떻게 강화할 수 있는지 보여줍니다.

### 2.1. 🔑 강력한 인증 (Multi-Factor Authentication, MFA)

비밀번호 외에 추가적인 인증 수단(OTP, 지문 등)을 사용하여 계정 탈취 위험을 크게 줄입니다. 루트 계정 및 관리자 계정에는 반드시 MFA를 활성화해야 합니다.

#### **AI 연동 전략**

| 구분 | AI 연동 방안 |
| :--- | :--- |
| **MFA 미사용 탐지** | AI가 IAM 사용자 목록을 스캔하여 MFA가 활성화되지 않은 루트 사용자나 관리자 계정을 탐지하고 경고.<br> **AI 모델:** 지도 학습 (MFA 활성화 여부 분류), NLP (IAM 설정 문서 분석). |
| **MFA 우회 탐지** | AI가 로그인 로그(CloudTrail)를 분석하여 MFA가 활성화된 계정에서 비정상적으로 MFA가 우회된 시도를 탐지.<br> **AI 모델:** 비지도 학습 (정상 로그인 패턴 학습 후 이상치 탐지). |

### 2.2. 👮 최소 권한 원칙 (Principle of Least Privilege)

사용자나 서비스에게 필요한 최소한의 권한만 부여합니다. `AdministratorAccess`와 같은 광범위한 권한은 피해야 합니다.

#### **AI 연동 전략**

| 구분 | AI 연동 방안 |
| :--- | :--- |
| **권한 과부여 탐지** | AI가 IAM 정책을 분석하여 사용되지 않는 권한(Unused Permissions)이나 과도하게 부여된 권한(Over-privileged)을 식별하고 개선 제안.<br> **AI 모델:** NLP (정책 문서 파싱 및 분석), 그래프 신경망 (GNN, 권한 관계 분석). |
| **비정상 권한 변경 탐지** | AI가 CloudTrail 로그를 분석하여 IAM 정책 변경, 역할 위임 등 비정상적인 권한 변경 시도를 탐지.<br> **AI 모델:** 비지도 학습 (정상 IAM 활동 패턴 학습 후 이상치 탐지). |
| **액세스 분석 (Access Analyzer):** | AI 기반 서비스인 AWS IAM Access Analyzer는 외부 엔티티가 조직의 리소스에 접근할 수 있는 경로를 식별하여 잠재적인 보안 위험을 경고합니다. |

### 2.3. 🔄 임시 자격 증명 사용 (Temporary Credentials)

장기적인 액세스 키 대신 IAM 역할을 통해 임시 자격 증명을 사용하여 자격 증명 유출 위험을 줄입니다. 임시 자격 증명은 만료 시간이 있어 유출되더라도 피해를 최소화할 수 있습니다.

#### **AI 연동 전략**

| 구분 | AI 연동 방안 |
| :--- | :--- |
| **장기 액세스 키 탐지** | AI가 IAM 사용자 목록을 스캔하여 장기 액세스 키가 활성화된 사용자나 루트 사용자의 액세스 키를 탐지하고 경고.<br> **AI 모델:** 지도 학습 (액세스 키 사용 패턴 분류). |
| **액세스 키 유출 탐지** | AI가 GitHub 등 공개 저장소를 모니터링하여 유출된 AWS 액세스 키를 탐지.<br> **AI 모델:** NLP (코드/텍스트에서 액세스 키 패턴 탐지). |
| **액세스 키 사용 패턴 분석:** | AI가 액세스 키 사용 로그를 분석하여 비정상적인 사용 패턴(예: 평소와 다른 지역/시간대 사용)을 탐지. |

### 2.4. 📝 IAM Access Advisor 및 Credential Report 활용

*   **IAM Access Advisor:** 사용자가 마지막으로 특정 서비스에 접근한 시간을 보여주어, 사용되지 않는 권한을 식별하고 최소 권한 원칙을 적용하는 데 도움을 줍니다.
*   **Credential Report:** 계정의 모든 사용자 자격 증명 상태를 CSV 파일로 제공하여, MFA 활성화 여부, 비밀번호 마지막 변경일, 액세스 키 사용 여부 등을 감사할 수 있습니다.

## 3. 💥 일반적인 IAM 취약점 및 공격 시나리오

IAM은 클라우드 보안의 핵심이지만, 잘못 관리될 경우 심각한 공격 벡터가 될 수 있습니다.

### 3.1. 💥 과도한 권한 부여 (Over-privileged)

*   **설명:** 사용자나 역할에 필요한 것보다 훨씬 많은 권한을 부여하는 경우. (예: S3 Read 권한만 필요한 사용자에게 S3 Full Access 부여)
*   **공격 시나리오:** 공격자가 해당 계정을 탈취하면, 부여된 모든 권한을 사용하여 시스템을 장악하거나 데이터를 유출할 수 있습니다.
*   **AI 탐지:** AI가 IAM 정책을 분석하여 `*` (모든 리소스/액션) 사용, `Allow` 규칙의 과도한 범위 등을 탐지.

### 3.2. 💥 사용되지 않는 권한 (Unused Permissions)

*   **설명:** 더 이상 필요하지 않거나 사용되지 않는 권한이 정책에 남아있는 경우.
*   **공격 시나리오:** 공격자가 사용되지 않는 권한을 발견하고 이를 악용하여 시스템에 접근하거나 권한을 상승시킬 수 있습니다.
*   **AI 탐지:** AI가 CloudTrail 로그와 IAM Access Advisor 데이터를 분석하여 특정 기간 동안 사용되지 않은 권한을 식별.

### 3.3. 💥 루트 계정 사용 (Root Account Usage)

*   **설명:** AWS 계정 생성 시 만들어지는 루트 계정은 모든 권한을 가지므로, 일상적인 작업에 사용해서는 안 됩니다.
*   **공격 시나리오:** 루트 계정이 탈취되면 클라우드 환경 전체가 장악될 수 있습니다.
*   **AI 탐지:** AI가 CloudTrail 로그에서 루트 계정의 활동을 모니터링하여 비정상적인 사용 패턴을 탐지.

### 3.4. 💥 액세스 키 유출 (Access Key Leakage)

*   **설명:** IAM 사용자의 액세스 키(Access Key ID, Secret Access Key)가 코드에 하드코딩되거나, 공개 저장소(GitHub)에 노출되는 경우.
*   **공격 시나리오:** 공격자가 유출된 액세스 키를 사용하여 해당 IAM 사용자의 권한으로 클라우드 자원에 접근합니다.
*   **AI 탐지:** AI가 GitHub 등 공개 저장소를 지속적으로 스캔하여 AWS 액세스 키 패턴을 탐지.

## 📌 요약 정리 (Executive Summary)

1.  **IAM의 중요성**: 클라우드 환경에서 '누가', '무엇에', '어떻게' 접근할 수 있는지를 중앙에서 관리하고 통제하는 핵심 보안 서비스이다. 최소 권한 원칙 구현, 계정 탈취 방지, 컴플라이언스 준수에 필수적이다.
2.  **AWS IAM 핵심 구성 요소**:
    *   **사용자 (Users)**: AWS 리소스와 상호 작용하는 개별 엔티티.
    *   **그룹 (Groups)**: 사용자들의 집합으로, 그룹에 정책을 부여하여 일괄 관리.
    *   **역할 (Roles)**: 임시 자격 증명을 통해 권한을 위임하는 메커니즘. 최소 권한 원칙 구현 및 자격 증명 유출 위험 감소에 중요.
    *   **정책 (Policies)**: JSON 형식으로 작성되며, 리소스에 대한 권한(Action, Resource, Effect, Condition)을 정의. 자격 증명 기반, 리소스 기반, 권한 경계, SCPs 등 다양한 종류가 있다.
3.  **IAM 보안 모범 사례**:
    *   **MFA (Multi-Factor Authentication)**: 강력한 인증으로 계정 탈취 방지.
    *   **최소 권한 원칙 (Principle of Least Privilege)**: 필요한 최소한의 권한만 부여. IAM Access Advisor, Access Analyzer 활용.
    *   **임시 자격 증명 사용**: 장기 액세스 키 대신 역할 사용.
    *   **루트 계정 사용 금지**: 일상적인 작업에는 IAM 사용자 사용.
    *   **액세스 키 주기적 교체**: 액세스 키 유출 위험 최소화.
4.  **AI 기반 IAM 보안**:
    *   **정책 분석**: AI가 IAM 정책을 분석하여 권한 과부여, 와일드카드 사용, 위험한 액션 포함 여부 등 취약점을 탐지하고 개선 제안.
    *   **비정상 접근 탐지**: AI가 CloudTrail 로그를 분석하여 비정상 로그인(지역/IP 변경, 실패 횟수), 비정상 권한 변경/위임 시도 등을 탐지.
    *   **자동 대응**: 탐지된 위협에 대해 계정 잠금, MFA 강제, 알림 전송 등의 자동화된 조치 수행.
5.  **보안 관점**: IAM은 클라우드 보안의 첫 번째 방어선이자 가장 중요한 통제 지점이다. AI를 활용하여 IAM 정책의 취약점을 지속적으로 분석하고, 비정상적인 접근을 실시간으로 탐지하며, 자동화된 대응을 통해 클라우드 자원에 대한 무단 접근을 효과적으로 방지해야 한다. AWS CLI와 같은 도구를 통해 IAM 리소스를 관리하고 감사하는 능력은 모의 해킹 및 보안 분석에 필수적이다.
