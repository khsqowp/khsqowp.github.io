```
❌ 나쁜 건축:

│ - 문 잠금장치 안 달음           │

✅ 좋은 건축:
┌────────────────────────────────┐
│ 🏢 튼튼한 재료로 설계부터      │
│ - 철근 콘크리트로 벽 만듦       │
│ - 이중 잠금장치 설치            │

│ → 수리비 거의 안 듦! 💰        │
└────────────────────────────────┘
```

**시큐어 코딩 = 튼튼한 재료로 집 짓기**

### 💻 코드에서는?

```python
# ❌ 비시큐어 코딩 (위험한 코드)
username = input("아이디: ")
password = input("비밀번호: ")
query = f"SELECT * FROM users WHERE id='{username}' AND pw='{password}'"
# → SQL Injection에 취약! 해커가 ' OR 1=1 -- 을 입력하면 뚫림!

# ✅ 시큐어 코딩 (안전한 코드)
username = input("아이디: ")
password = input("비밀번호: ")
query = "SELECT * FROM users WHERE id=? AND pw=?"
cursor.execute(query, (username, password))
# → 안전! 사용자 입력이 SQL 구조를 바꿀 수 없음!
```

## 📊 왜 처음부터 안전하게 짜야 하나요?

### 💰 비용 비교

```
개발 단계별 보안 문제 수정 비용:

설계 단계:        💰 1원
개발 단계:        💰💰💰 10원
테스트 단계:      💰💰💰💰💰 100원
운영 단계:        💰💰💰💰💰💰💰💰💰💰 1,000원
해킹당한 후:      💸💸💸💸💸💸💸💸💸💸💸💸 10,000원+

→ 나중에 고칠수록 10배씩 비용 증가!
```

### 🏆 실제 사례: 비용 차이

| 시점 | 문제 발견 | 수정 비용 | 비고 |
|------|----------|----------|------|
| 설계 단계 | SQL Injection 가능성 | 1시간 | 코드 수정만 |
| 테스트 단계 | SQL Injection 발견 | 1주일 | 코드 + 재테스트 |
| 운영 단계 | SQL Injection 발생 | 1개월 | 코드 + 테스트 + 재배포 |
| 해킹당한 후 | 데이터 유출 | 수억 원 | 벌금 + 배상 + 신뢰 상실 |

```
│ 🏦 은행                        │

│ - 쉬운 비밀번호                 │

│ 🏦 은행                        │
```

### 💻 코드에서는?

**나쁜 예: 약한 보안**

```python
# ❌ 위험한 코드
users = {
    "admin": "1234",  # 쉬운 비밀번호
    "user1": "password"  # 평문 저장!
}

def login(username, password):
    if username in users and users[username] == password:
        return "로그인 성공"
    return "로그인 실패"
```

**좋은 예: 강한 보안**

```python
# ✅ 안전한 코드
import bcrypt

# 비밀번호를 해시로 저장 (평문 저장 안 함)
users = {
    "admin": bcrypt.hashpw("P@ssw0rd!2024".encode(), bcrypt.gensalt()),
    "user1": bcrypt.hashpw("MyS3cur3P@ss".encode(), bcrypt.gensalt())
}

login_attempts = {}  # 로그인 시도 횟수 기록

def login(username, password):
    # 로그인 시도 횟수 확인
    if username in login_attempts and login_attempts[username] >= 3:
        return "계정이 잠겼습니다. 관리자에게 문의하세요."

    # 비밀번호 확인
    if username in users:
        if bcrypt.checkpw(password.encode(), users[username]):
            login_attempts[username] = 0  # 성공 시 초기화
            return "로그인 성공"

    # 실패 시 시도 횟수 증가
    login_attempts[username] = login_attempts.get(username, 0) + 1
    return "로그인 실패"
```

### 🔑 보안 기능 체크리스트

```
인증 (Authentication):
☐ 비밀번호는 8자 이상, 복잡하게
☐ 비밀번호는 bcrypt/scrypt로 해시 저장
☐ 2단계 인증 (MFA) 지원
☐ 로그인 실패 시 계정 잠금

접근 제어 (Authorization):
☐ 사용자마다 권한 다르게
☐ 기본은 "차단", 허용은 명시적으로
☐ 관리자 페이지는 일반 사용자 차단

암호화 (Encryption):
☐ HTTPS 사용 (http → https)
☐ 민감 데이터는 AES-256으로 암호화
☐ 암호화 키는 안전하게 보관
```

### 🏃 실생활 비유: 화장실 사용

```
❌ 동시성 문제:
┌────────────────────────────────┐
│ 🚽 화장실                      │
│ - 잠금장치 없음!                │
│                                │
│ 상황:                          │
│ 사람 A: 문을 봄 → 비어있네!     │
│ 사람 B: 문을 봄 → 비어있네!     │
│ 사람 A: 들어감                  │
│ 사람 B: 들어감                  │
│ → 충돌! 😱                     │
└────────────────────────────────┘

✅ 해결책:
┌────────────────────────────────┐
│ 🚽 화장실                      │
│ - 잠금장치 있음!                │
│                                │
│ 상황:                          │
│ 사람 A: 문을 봄 → 비어있네!     │

│ 사람 B: 문을 봄 → 잠겼네, 대기  │
│ 사람 A: 나감, 잠금 해제          │
│ 사람 B: 들어감                  │
│ → 안전! ✅                     │
└────────────────────────────────┘
```

### 💻 코드에서는?

**나쁜 예: Race Condition**

```python
# ❌ 위험한 코드 (여러 사람이 동시에 접근하면 문제!)
balance = 10000  # 계좌 잔액

def withdraw(amount):
    global balance
    if balance >= amount:
        # 여기서 다른 사람이 끼어들 수 있음!
        balance = balance - amount
        return f"{amount}원 출금 성공"
    return "잔액 부족"

# 2명이 동시에 10000원 출금 시도하면?
# → 둘 다 성공해서 잔액이 -10000원이 될 수 있음!
```

**좋은 예: 잠금장치 사용**

```python
# ✅ 안전한 코드
import threading

balance = 10000
balance_lock = threading.Lock()  # 잠금장치!

def withdraw(amount):
    global balance
    with balance_lock:  # 잠금!
        if balance >= amount:
            balance = balance - amount
            return f"{amount}원 출금 성공"
        return "잔액 부족"
    # 자동으로 잠금 해제됨

# 2명이 동시에 시도해도 한 명씩 처리됨!
```

**좋은 예: 캡슐화**

```python
# ✅ 안전한 코드
class BankAccount:
    def __init__(self):
        self.__balance = 0  # __ = private (외부 접근 불가)
        self.__password = "1234"  # private

    def deposit(self, amount):
        """입금 (안전하게 잔액 변경)"""
        if amount > 0:
            self.__balance += amount

    def withdraw(self, amount, password):
        """출금 (비밀번호 확인 후)"""
        if password == self.__password and self.__balance >= amount:
            self.__balance -= amount
            return True
        return False

    def get_balance(self, password):
        """잔액 조회 (비밀번호 확인 후)"""
        if password == self.__password:
            return self.__balance
        return "비밀번호가 틀렸습니다."

# 사용
account = BankAccount()
# account.__balance = 1000000  # 에러! 접근 불가!
account.deposit(10000)  # 안전한 방법으로만 입금 가능!
```

```python
# ❌ 위험한 코드
session_counter = 0

def create_session(username):
    global session_counter
    session_counter += 1
```

**좋은 예: 안전한 세션**

```python
# ✅ 안전한 코드
import secrets
from datetime import datetime, timedelta

sessions = {}

def create_session(username):
    # 암호학적으로 안전한 랜덤 문자열 (32바이트)
    session_id = secrets.token_urlsafe(32)

    # 세션 만료 시간 설정 (30분 후)
    expiry_time = datetime.now() + timedelta(minutes=30)

    sessions[session_id] = {
        "username": username,
        "expiry": expiry_time
    }

    return session_id

def is_valid_session(session_id):
    if session_id not in sessions:
        return False

    # 만료 시간 확인
    if datetime.now() > sessions[session_id]["expiry"]:
        del sessions[session_id]  # 만료된 세션 삭제
        return False

    return True
```

## 🔍 실습: 코드 보안 문제 찾기

### 퀴즈 1: 이 코드의 문제는?

```python
def delete_user(user_id):
    query = f"DELETE FROM users WHERE id = '{user_id}'"
    database.execute(query)
```

<details>
<summary>정답 및 해설</summary>

**문제:** SQL Injection 취약점!

**왜?**
- 사용자 입력(`user_id`)을 검증 없이 SQL 쿼리에 직접 삽입

**공격 시나리오:**
```python
# 해커가 이렇게 입력하면?
user_id = "1' OR '1'='1"
# 실행되는 쿼리: DELETE FROM users WHERE id = '1' OR '1'='1'
# → 모든 사용자 삭제! 😱
```

**해결책:**
```python
def delete_user(user_id):
    query = "DELETE FROM users WHERE id = ?"
    database.execute(query, (user_id,))
```
</details>

### 퀴즈 2: 이 코드의 문제는?

```python
API_KEY = "sk-1234567890abcdef"

def call_api():

    return response.json()
```

<details>
<summary>정답 및 해설</summary>

**문제:** 하드코딩된 자격 증명!

**왜?**

- 코드가 GitHub에 올라가면 전 세계 공개!

```python
def call_api():
    response = requests.get(f"https://api.example.com?key={API_KEY}")
    return response.json()
```

**.env 파일에 저장:**
```
API_KEY=sk-1234567890abcdef
```
</details>

### 퀴즈 3: 이 코드의 문제는?

```python
def login(username, password):
    if username == "admin" and password == "admin123":
        return "관리자 로그인 성공!"
    return "로그인 실패!"
```

<details>
<summary>정답 및 해설</summary>

**문제:** 여러 가지!

1. **하드코딩된 자격 증명**
   - 관리자 비밀번호가 코드에 그대로 적혀있음

2. **약한 비밀번호**
   - "admin123"는 쉽게 추측 가능

3. **비밀번호 평문 비교**
   - 비밀번호를 해시로 저장하지 않음

4. **자세한 에러 메시지**
   - "관리자"라는 정보 노출

**해결책:**
```python
import bcrypt

# 미리 해시된 비밀번호를 저장
ADMIN_PASSWORD_HASH = bcrypt.hashpw("P@ssw0rd!2024".encode(), bcrypt.gensalt())

def login(username, password):
    if username == "admin":
        if bcrypt.checkpw(password.encode(), ADMIN_PASSWORD_HASH):
            return "로그인 성공"
    return "ID 또는 비밀번호가 틀렸습니다"
```
</details>

## ❓ 자주 묻는 질문 (FAQ)

### Q1: 시큐어 코딩을 하면 개발 속도가 느려지지 않나요?
**A:**
- **단기적으로는** 조금 느릴 수 있습니다 (10-20% 정도)
- **장기적으로는** 훨씬 빠릅니다!
  - 나중에 보안 문제 고치는 시간 없음
  - 해킹당해서 복구하는 시간 없음
  - 법적 문제 대응하는 시간 없음

### Q2: 모든 코드에 다 적용해야 하나요?
**A:**
- **필수:** 사용자 입력 받는 부분, 데이터베이스 접근, 인증/권한
- **권장:** 모든 코드 (습관 들이기)
- **우선순위:** 외부와 연결되는 부분 먼저

### Q3: AI 도구를 믿어도 되나요?
**A:**
- AI는 **보조 도구**입니다
- 100% 정확하지 않음 (오탐/미탐 있음)
- 최종 확인은 사람이 해야 함
- 하지만 사람이 놓치는 부분을 잘 찾아줌!

### Q4: 레거시 코드는 어떻게 하나요?
**A:**
```
우선순위 접근:
1. 위험도 높은 부분부터 수정
   - 로그인, 결제, 개인정보 처리
2. 새로운 기능은 무조건 시큐어 코딩
3. 리팩토링 시 보안도 함께 개선
4. 한 번에 다 고치려 하지 말기!
```
